## 1. 背景与问题

暑假喂食任务不是一次性任务。一个固定喂食点可能有多个执行日期，例如：

```text
白敬亭喂食任务
  - 2026-07-15
  - 2026-07-25
  - 2026-08-05
```

如果成员在 2026-07-15 完成投喂，系统不能把整个父任务置为已完成。正确含义是：

```text
2026-07-15 这一次子任务已完成
父任务继续进行
2026-07-25 到来后，进入下一次独立子任务
```

当前库表和后端已经具备父任务与执行日期的基础结构：

- `tasks` 表表示父任务。
- `task_execution_dates` 表表示每一次执行日期，也就是子任务实例。
- `task_checkins` 表表示成员完成某个子任务后的打卡归档。
- `task_activity_logs` 可通过 `task_execution_date_id` 关联到具体子任务。

当前主要缺口在展示和派生状态：

1. 任务列表只展示一个父任务和一个 `current_execution`，用户看不清各子任务完成情况。
2. 父任务详情没有区分父任务聚合视角和子任务单日视角。
3. 地图 marker 当前更多依赖父任务状态或当天状态，不足以表达当前激活子任务。
4. 个人中心统计仍是占位，应统计子任务完成记录，而不是父任务数量。
5. 任务筛选应筛选子任务卡片，而不是简单过滤父任务状态。

## 2. 核心概念

### 2.1 父任务

父任务对应 `tasks`。

一条父任务表示一个固定喂食点在一段时间内的投喂安排。父任务保存稳定信息：

- 标题
- 描述
- 地图点位
- 参考图片
- 路线说明
- 物资要求
- 整体可见性
- 整体状态

父任务状态只表示任务整体是否还有效，不表示某一天是否完成。

### 2.2 子任务

子任务对应 `task_execution_dates`。

一条子任务表示某个父任务在某一天的一次执行实例。子任务保存：

- 执行日期
- 子任务状态
- 完成人
- 完成时间
- 对应打卡记录
- 当日备注

用户完成、地图 marker 当前态、任务列表日期卡片、个人完成统计，都应以子任务为主。

### 2.3 完成记录

完成记录对应 `task_checkins`。

一条完成记录表示某个成员完成了某个子任务。它是个人历史和统计的主要来源。完成照片通过 `task_checkin_photos` 关联到具体打卡记录。

## 3. 状态模型

### 3.1 父任务状态

父任务状态继续使用 `tasks.status`：

| 状态 | 含义 | 展示口径 |
|---|---|---|
| `in_progress` | 整个周期任务仍在执行中 | 进行中 |
| `completed` | 整个周期任务已结束 | 已结束 |
| `cancelled` | 整个父任务取消 | 已取消 |
| `archived` | 历史归档 | 已归档 |

父任务状态不因单个子任务完成而变为 `completed`。

### 3.2 子任务存储状态

子任务状态使用 `task_execution_dates.status`：

| 状态 | 含义 |
|---|---|
| `pending` | 待执行，尚未完成 |
| `completed` | 已完成 |
| `cancelled` | 已取消 |
| `skipped` | 已跳过，不计异常 |

不新增“逾期/未完成”状态。

### 3.3 子任务展示状态

前端展示状态从子任务存储状态和当前日期派生：

| 条件 | 展示状态 | 颜色 |
|---|---|---|
| `status = completed` | 已完成 | 绿色 |
| `status in (cancelled, skipped)` | 已取消 | 灰色 |
| `status = pending` 且当前日期早于执行日期 | 未开始 | 红色 |
| `status = pending` 且当前日期大于等于执行日期 | 进行中 | 黄色 |

说明：

```text
执行日已过但还没有完成，仍展示为“进行中”。
```

### 3.4 自动取消规则

当下一个子任务开启时，如果上一个子任务仍未完成，则上一个子任务自动判定为取消。

示例：

```text
父任务日期：2026-07-15、2026-07-25、2026-08-05

2026-07-15：进入 7.15 子任务，状态进行中
2026-07-24：7.15 仍未完成，仍展示进行中
2026-07-25：7.25 子任务开启，7.15 自动改为 cancelled，7.25 进入进行中
```

该自动取消不影响下一次子任务开启。

建议后端以“懒处理”为 MVP 落地方式：

1. 在任务列表、任务详情、地图点位查询、完成打卡前调用统一状态归一化函数。
2. 对同一父任务的执行日期按日期升序扫描。
3. 如果某个 `pending` 子任务存在下一个执行日期，并且当前日期大于等于下一个执行日期，则将该子任务改为 `cancelled`。
4. 写入系统动态，可选标题为“子任务自动取消”。

后期如果需要更强一致性，可以增加每日定时任务，但 V1 不引入额外基础设施。

## 4. 当前激活子任务

地图 marker、地图任务列表和任务列表默认卡片，都需要一个“当前激活子任务”。

定义：

1. 如果今天正好有子任务，激活今天的子任务。
2. 如果今天没有子任务，且上一个子任务已完成，则继续展示上一个已完成子任务，直到下一次子任务开启。
3. 如果今天没有子任务，且上一个子任务未完成，则继续展示上一个进行中子任务，直到下一次子任务开启。
4. 当下一次子任务开启时，上一未完成子任务自动取消，激活下一子任务。
5. 如果所有子任务都已结束，激活最后一个子任务用于历史展示。

示例：

```text
7.15 完成后，7.16 - 7.24 地图 marker 保持 7.15 已完成图标。
7.25 开始，marker 切换到 7.25 子任务状态。
```

## 5. 后端接口设计调整

### 5.1 任务列表接口

现有：

```http
GET /api/v1/tasks
```

建议在列表项中增加 `display_executions`：

```json
{
  "task_id": "uuid",
  "title": "白敬亭喂食任务",
  "status": "in_progress",
  "status_label": "进行中",
  "current_execution": {},
  "next_execution": {},
  "display_executions": [
    {
      "execution_date_id": "uuid",
      "execute_date": "2026-07-15",
      "status": "completed",
      "display_status": "completed",
      "display_status_label": "已完成"
    },
    {
      "execution_date_id": "uuid",
      "execute_date": "2026-07-25",
      "status": "pending",
      "display_status": "not_started",
      "display_status_label": "未开始"
    }
  ]
}
```

筛选规则：

- `status` 继续筛选父任务整体状态。
- `execution_status` 筛选子任务展示卡片。
- `execute_date` / `execute_date_start` / `execute_date_end` 筛选子任务日期。
- 如果某父任务筛选后没有任何 `display_executions`，该父任务不返回。
- 如果某父任务筛选后有匹配子任务，则父任务卡片保留，只显示匹配的子任务卡片。

### 5.2 任务详情接口

现有：

```http
GET /api/v1/tasks/{task_id}
```

建议支持两种视角：

```http
GET /api/v1/tasks/{task_id}
GET /api/v1/tasks/{task_id}?execution_date_id={execution_date_id}
```

不新增独立子任务资源路径，避免与现有任务详情页路由大改。前端可以通过 query 参数决定进入父任务详情或子任务详情。

响应中增加：

```json
{
  "detail_scope": "parent",
  "active_execution_id": "uuid",
  "execution_groups": [
    {
      "execution": {},
      "activities": [],
      "checkin_photos": []
    }
  ]
}
```

当传入 `execution_date_id` 时：

```json
{
  "detail_scope": "execution",
  "execution": {},
  "activities": [],
  "checkin_photos": []
}
```

父任务详情仍可保留现有 `execution_dates`、`activities`、`checkin_photos` 字段以兼容旧前端，但新 UI 优先使用 `execution_groups`。

### 5.3 完成打卡接口

现有：

```http
POST /api/v1/tasks/{task_id}/checkins
```

建议继续复用，但前端从子任务详情页提交时必须携带：

```json
{
  "execute_date": "2026-07-25"
}
```

后端规则：

1. 只能完成当前日期已开启的子任务。
2. 不能完成未来子任务。
3. 已完成、已取消、已跳过的子任务不能完成。
4. 提交前先执行自动取消归一化，避免用户补打上一轮已被自动取消的子任务。

## 6. 前端页面设计

### 6.1 任务列表页

父任务卡片仍只出现一次。

卡片调整：

1. 缩略图不变。
2. 父任务状态角标保留。
3. 去掉喂食点名称。
4. 任务描述上移到标题下方，仅保留一行，超出省略。
5. 去掉物资行。
6. 原日期位置改为子任务状态卡片横向滑动区域。

子任务状态卡片：

- 文案为子任务日期，例如 `7月15日`。
- 背景色代表子任务展示状态。
- 横向自由滑动。
- 点击子任务卡片进入子任务详情页。
- 点击父任务卡片空白区域进入父任务详情页。

事件处理上需要避免冒泡：

```text
点击子任务卡片 -> 子任务详情
点击父任务卡片 -> 父任务详情
```

### 6.2 父任务详情页

父任务详情页展示全周期信息。

调整：

1. “本次任务日期”改为“任务日期”。
2. 日期区域展示所有子任务日期，并作为进入子任务详情的入口。
3. 任务动态按子任务分组展示。
4. 完成照片按子任务分组展示。

分组卡片示例：

```text
2026年7月15日    已完成

任务动态
- Nietzsche 完成投喂

完成照片
[照片] [照片]
```

这样每个子任务一个卡片，不需要在每条动态和每张照片上重复标记日期。

### 6.3 子任务详情页

子任务详情页与父任务详情页共用页面组件，但通过 `execution_date_id` 进入单日视角。

差异：

1. 日期区域只展示本次子任务日期，不作为入口。
2. 任务动态只展示本子任务相关动态。
3. 完成照片只展示本子任务相关照片。
4. 完成按钮只作用于本子任务。

## 7. 地图联动设计

### 7.1 Marker 状态

地图 marker 状态绑定当前激活子任务，而不是父任务状态。

后端 marker extra 建议增加：

```json
{
  "task_status": "in_progress",
  "task_status_label": "进行中",
  "active_execution": {
    "execution_date_id": "uuid",
    "execute_date": "2026-07-25",
    "status": "pending",
    "display_status": "in_progress",
    "display_status_label": "进行中"
  }
}
```

前端 marker 图标根据 `active_execution.display_status` 决定：

| display_status | 图标/颜色 |
|---|---|
| `not_started` | 未开始图标 |
| `in_progress` | 进行中图标 |
| `completed` | 已完成图标 |
| `cancelled` | 已取消图标 |

### 7.2 地图页任务列表

地图页底部任务列表仍展示父任务，但标题后增加当前激活子任务日期标识。

示例：

```text
白敬亭喂食任务  [7月25日]
进行中
```

日期标识背景色与子任务状态一致。

### 7.3 地图详情入口

地图页点击“查看详情”进入子任务详情页：

```text
/pages/tasks/detail?task_id={task_id}&execution_date_id={active_execution_id}
```

如果没有 `active_execution_id`，再退回父任务详情页。

## 8. 个人中心统计

个人任务完成统计启用后，统计子任务完成记录，而不是父任务。

数据来源：

```text
task_checkins
JOIN task_execution_dates
JOIN tasks
```

统计口径：

- 累计任务：当前用户完成的有效子任务数量。
- 本月完成：当前用户本月完成的有效子任务数量。
- 个人完成任务列表：展示子任务记录，而不是父任务。

点击“累计任务”或“本月完成”进入的列表，应展示：

```text
白敬亭喂食任务
2026年7月15日
已完成
完成时间
完成照片缩略图
```

## 9. 筛选设计

### 9.1 任务类型

保持现有逻辑。当前只有 `feeding`。

### 9.2 父任务状态筛选

父任务整体状态筛选保留：

```text
全部 / 进行中 / 已结束 / 已取消 / 已归档
```

### 9.3 子任务状态筛选

子任务状态筛选作用于任务卡片内的子任务卡片。

示例：

```text
白敬亭喂食任务
  7.15 已完成
  7.20 进行中
```

选择“已完成”后：

```text
白敬亭喂食任务
  7.15 已完成
```

如果父任务没有任何匹配子任务，则整张父任务卡片隐藏。

### 9.4 日期筛选

日期筛选同样作用于子任务卡片。

如果选择的日期不是任何子任务日期，则父任务卡片不展示。

## 10. 自动取消的实现边界

MVP 不引入 WebSocket、Redis 或复杂调度系统。

推荐实现方式：

1. 在任务服务中新增统一函数，例如 `normalize_execution_date_states(task, today)`。
2. 查询任务列表、任务详情、地图 marker、地图 summary、完成打卡前调用。
3. 函数只处理当前任务相关的执行日期。
4. 发现需要自动取消时，更新 `task_execution_dates.status = cancelled`。
5. 自动取消原因写入 `remark` 或 `cancel_reason`。
6. 可选写入 `task_activity_logs`，避免动态刷屏，建议一个子任务最多写一次。

自动取消不会修改父任务状态。

## 11. 风险与注意点

1. `completed` 父任务不能再被误认为“某天完成”。前端父状态和子状态要分开命名。
2. 任务列表需要新增 `display_executions`，否则无法支持筛选后只显示部分子任务卡片。
3. 地图 marker 需要新增 `active_execution`，否则无法稳定表达“完成保持到下一次开启”。
4. 子任务详情不要复制父任务数据到新表，只通过 `execution_date_id` 切换视角。
5. 自动取消要在完成打卡前执行，防止用户完成已经被下一轮任务覆盖的旧子任务。
6. 父任务编辑删除执行日期时，仍需保留已完成子任务，不允许直接删除历史归档。

## 12. 推荐落地顺序

1. 后端补齐子任务展示状态和自动取消归一化。
2. 列表接口增加 `display_executions` 并更新筛选逻辑。
3. 详情接口增加父任务分组结构和子任务详情视角。
4. 地图 marker 和 summary 增加 `active_execution`。
5. 前端任务列表改为父卡 + 横向子任务卡。
6. 前端详情页支持父/子两种视角。
7. 地图详情入口改为进入当前激活子任务详情。
8. 个人中心统计和完成记录列表改为基于 `task_checkins`。
9. 补齐测试和开发进度记录。

## 13. 结论

本设计沿用当前库表方向，不复制父任务，不新增重复业务表。

最终口径：

```text
父任务负责“这个喂食点长期安排是否有效”。
子任务负责“某一天这一次投喂是否完成”。
完成记录负责“谁完成了哪一天的投喂”。
```

这样可以同时满足：

- 列表不重复展示父任务。
- 用户能看到每个子任务完成情况。
- 父详情能聚合全周期历史。
- 子详情能聚焦单次执行。
- 地图 marker 与当前子任务状态一致。
- 个人统计真实反映用户完成次数。
