# 同机开发环境隔离设计

## 目标

在不改动当前生产服务、生产路径、生产数据库和生产发布脚本的前提下，在现有服务器上建立可独立发布、独立访问、独立写入开发数据库的开发环境。开发小程序请求 `https://dev-api.trmx.fun/api/v1`；生产小程序继续请求当前生产域名。

## 已确认的现状

- 生产服务运行在独立的 systemd 单元 `catmap-backend`，仅监听本机 8000 端口。
- 生产数据库为 `catmap`，开发数据库为 `catmap_dev`；开发库比生产库多出由 `20260712_0015` 创建的 `wechat_guests` 表，其他已比较的逻辑结构一致。
- 开发数据库已有专用角色；开发服务必须只使用该角色，不能使用生产应用角色。
- `dev-api.trmx.fun` 的 A 记录已指向现有服务器，微信小程序的 request/upload/download 合法域名已提交。
- 对象存储暂共用一个桶，但开发写入必须使用 `dev/` 前缀，生产路径保持不变。

## 推荐架构

```text
生产小程序 → 生产域名 → Nginx → catmap-backend (127.0.0.1:8000) → catmap
开发小程序 → dev-api.trmx.fun → Nginx → catmap-backend-dev (127.0.0.1:8001) → catmap_dev
```

新增资源均使用 `-dev` 或 `catmap-dev` 命名：

- 部署目录：`/opt/catmap-dev/backend`
- systemd 单元：`catmap-backend-dev`
- Nginx 虚拟主机：`catmap-dev.conf`
- 部署脚本：`scripts/deploy-backend-dev.ps1`
- 开发前端环境：`frontend/.env.development`；通用 `npm run build`、`npm run build:mp-weixin` 与 `npm run build:mp-weixin:dev` 均固定为开发模式，生产验证必须显式使用 `npm run build:mp-weixin:prod`

开发端口 8001 只监听 `127.0.0.1`，不对公网开放；外部请求仅通过带 HTTPS 证书的开发子域名进入 Nginx。`catmap-backend-dev` 以专用非特权系统用户 `catmap-dev` 运行，仅可写入 `/opt/catmap-dev`，不能直接读取或改写生产服务文件。

## 发布与安全边界

1. 保留现有生产部署脚本、生产 systemd 文件和生产 Nginx 文件原样不动。
2. 新开发部署脚本固定校验开发路径、开发服务名、开发域名、开发数据库名和开发数据库角色，发现生产标识即拒绝执行；远端部署临时文件放在 root-only 的随机目录，并无论成功或失败均清理。
3. 开发环境使用独立的服务环境文件；部署前校验其数据库指向 `catmap_dev`、用户为开发专用角色，并校验对象存储环境前缀为 `dev`。不在仓库、日志或文档中记录凭据。
4. 开发域名的证书先使用 HTTP challenge 进行签发，再启用 HTTPS 反向代理配置；证书、Nginx 配置和服务均须通过语法/健康检查后才切换。
5. 未通过开发环境验证的代码不得替换生产服务。上线前使用生产部署流程做最后一次验证，且不改变生产数据库数据结构以外的既有数据。

## 数据与功能边界

现阶段不做自动生产数据同步。已有开发库只作为开发环境的唯一写库；今后如需用脱敏生产快照刷新开发库，应在单独任务中增加备份、脱敏、恢复、校验和回滚流程。

开发库已处于 `20260712_0015`，因此首次开发部署必须使用包含该迁移的、已提交的源码版本。当前 `public-pages` 工作区的相关改动仍未提交，不能被部署脚本当作可追溯发布源；本次先完成隔离脚本、模板和验证，实际服务器部署以前必须先获得该功能的明确提交基线。共享 COS Bucket 的前缀只是命名隔离；开发凭据最终仍应由 CAM 限制到 `dev/*`，防止开发代码使用全桶权限影响生产对象。

## 验收条件

- `https://dev-api.trmx.fun/api/v1/health` 返回成功，且生产健康接口持续正常。
- 开发服务的数据库身份和迁移版本符合开发库预期，生产库未发生写入。
- 开发上传对象位于 `dev/` 前缀，生产对象路径未受影响。
- 开发前端显式使用开发 HTTPS API 地址，生产前端仍使用生产地址。
- 生产 systemd 服务、生产 Nginx 文件、生产部署脚本均无改动。
