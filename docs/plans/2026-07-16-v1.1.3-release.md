# v1.1.3 热修发布清单

## 发布基线

- 源分支：`hotfix/1.1.3-task-media-member`
- 基线：`main` / `v1.1.2@0097c06`
- 目标版本：`v1.1.3`
- 小程序版本：`versionName=1.1.3`、`versionCode=113`
- 远端状态：`origin/main` 当前停在 v1.1.1 发布记录，本地 `main` 已包含尚未推送的 v1.1.2 正式历史；发布 v1.1.3 时会一并推送这段已审查的主分支历史。
- Release 更新说明：`docs/releases/v1.1.2.md`、`docs/releases/v1.1.3.md`；远端缺失的 v1.1.2 tag 和 GitHub Release 需要在本次发布中补齐。
- 生产后端已提前运行本热修的兼容后端代码，正式发布时仍从最终发布提交再部署一次，使代码、标签和生产记录一致。

## 已验收范围

- 任务动态时间按中国标准时间（UTC+8）展示。
- 微信原生大图预览可正常切图、缩放和平移，放大后不会因浏览图片而误切图。
- 父任务入口修改父任务状态，子任务入口只修改对应子任务状态。
- 任务点、物资点、药品和地标图片可拖拽排序，第一张作为列表封面。
- 手动指定高位喵喵号不会推进自动序列，清理测试账号后自动编号从 `trmx0010` 恢复。

## 发布前门禁

在 hotfix 工作树执行：

```powershell
cd backend
$env:CATMAP_WECHAT_CONTENT_SECURITY_MODE = "off"
py -3.11 -m pytest -q
py -3.11 -m ruff check .
py -3.11 -m alembic heads

cd ..\frontend
npm run test -- --run
npm run type-check
npm run build:mp-weixin

cd ..
powershell -ExecutionPolicy Bypass -File scripts\test-deployment-contracts.ps1
git diff --check
```

同时确认：

- `backend/.env` 与 `frontend/.env.production` 存在且保持 ignored，不输出其内容。
- 微信小程序构建产物使用生产 API Base，并包含 `versionName=1.1.3` / `versionCode=113` 对应配置。
- 提交范围不包含 `.env`、证书、私钥、Token、数据库备份或临时调试文件。
- `https://trmx.fun/api/v1/health` 返回 HTTP 200 和统一成功响应。

## 正式发布步骤

1. 在干净的 `main` 工作树将 hotfix 最终差异以 squash 方式合入，避免把已被撤销的中间大图试验提交带入主分支历史：

   ```powershell
   git merge --squash hotfix/1.1.3-task-media-member
   git commit -m "release: v1.1.3"
   ```

2. 在合并后的 `main` 再执行一次发布前门禁。
3. 创建 annotated tag：

   ```powershell
   git tag -a v1.1.3 -m "release: v1.1.3"
   ```

4. 从该提交部署生产后端：

   ```powershell
   powershell -ExecutionPolicy Bypass -File scripts\deploy-backend.ps1 -EnvFile backend\.env
   ```

5. 复核生产健康接口、systemd 服务状态和 Alembic head `20260716_0015`。
6. 将 `frontend/dist/build/mp-weixin` 导入微信开发者工具，清缓存编译后上传 1.1.3，并提交微信审核。
7. 只推送预期的 `main` 和 `v1.1.3`；推送前再次检查 `origin/main..main` 完整范围中的敏感信息，并确认其中包含尚未推送但已正式发布的 v1.1.2 历史。
8. 推送本地已有的 `v1.1.2` annotated tag，并用 `docs/releases/v1.1.2.md` 补建 GitHub Release；用 `docs/releases/v1.1.3.md` 创建 v1.1.3 GitHub Release。
9. 将 `main` 上的单个 `release: v1.1.3` 提交同步到 `dev`，解决冲突并重新运行受影响验证；不要直接覆盖当前已领先的开发代码。

## 回滚

- 小程序回滚：重新发布上一稳定小程序版本；本次后端保持兼容，不要求同步回滚数据库。
- 后端回滚：将生产代码恢复到 `v1.1.2` 后重启服务；除非明确确认不再使用自动编号，不执行迁移 downgrade，不删除 `meow_no_auto_sequence`。
- 数据恢复：成员清理前的受限备份继续保留；未经再次授权不恢复或修改生产成员数据。
- 回滚后必须复核健康接口、服务状态和核心登录/任务查询流程，并在 `docs/开发进度.md` 记录实际操作。
