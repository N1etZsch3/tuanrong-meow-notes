# 校园猫协地图任务系统｜图片上传模块库表优化设计文档（多缩略图版）

> 版本：V1.1  
> 适用范围：腾讯云 COS 图片上传模块、用户头像、猫咪照片、猫咪头像、地图点位照片、任务打卡照片、观察记录照片  
> 数据库：PostgreSQL  
> 后端建议：FastAPI + SQLAlchemy + Alembic  
> 存储服务：腾讯云 COS  
> 核心目标：建立统一图片资产表与图片变体表，让项目中不同页面可以按场景加载不同尺寸图片，避免列表、地图、头像等轻量场景误加载大图。

---

# 1. 本版调整说明

上一版图片上传库表设计中，主要采用：

```text
file_assets.file_url
file_assets.thumbnail_url
```

也就是一张图片只对应一个标准展示图和一个缩略图。

这个设计可以满足 MVP 的基础展示，但无法很好地覆盖以下场景：

| 场景 | 需要的图片尺寸 | 问题 |
|---|---:|---|
| 地图 Marker 小卡片 | 120px - 180px | 加载 320px 或 1280px 都浪费 |
| 猫咪列表封面 | 240px - 360px | 需要比 Marker 更清晰 |
| 猫咪详情照片墙 | 720px 左右 | 用小缩略图会模糊，用原展示图会偏大 |
| 任务打卡详情 | 1280px 左右 | 需要能看清现场情况 |
| 用户头像列表 | 64px - 128px | 不应该加载 512px 头像 |
| 个人中心头像 | 256px - 512px | 需要比列表头像更清晰 |
| 路线图 / 入口图 | 1280px - 1600px | 需要保留更多环境细节 |

因此本版做如下调整：

1. 新增 `file_asset_variants` 表，用于保存同一张图片的不同尺寸版本。
2. `file_assets` 只表示“一次上传产生的一张逻辑图片资产”。
3. 不再将 `thumbnail_url` 设计成唯一缩略图来源，而是由 `file_asset_variants.variant_key` 表示不同缩略图规格。
4. 业务表仍可保留 `file_url`、`thumbnail_url`、`avatar_url` 等冗余字段，用于兼容现有接口和提高查询效率。
5. 本版去掉 `file_upload_logs`，V1 不单独建图片上传日志表。
6. 本版不建 `file_cleanup_tasks`，COS 物理删除和孤儿文件清理作为后期任务处理。

# 2. 设计边界

## 2.1 图片上传模块负责

图片上传模块负责以下能力：

1. 接收用户上传的图片文件。
2. 校验图片格式、大小、像素数量和真实 MIME。
3. 修正 EXIF 方向。
4. 去除 EXIF 元数据。
5. 按不同用途生成标准展示图和多个缩略图。
6. 将处理后的图片上传到腾讯云 COS。
7. 写入 `file_assets` 和 `file_asset_variants`。
8. 返回统一图片 DTO 给业务模块。
9. 为头像、猫咪照片、点位照片、打卡照片提供统一文件能力。
10. 调用微信图片内容安全接口，记录异步审核状态，并阻止待审/拒绝图片发布。

## 2.2 图片上传模块不负责

图片上传模块不负责业务含义本身：

| 业务动作 | 所属模块 |
|---|---|
| 设置某张猫咪照片为头像 | 猫咪库模块 |
| 设置某张点位照片为封面 | 地图模块 |
| 判断任务打卡是否有效 | 任务模块 |
| 判断成员是否有权限上传某只猫照片 | 猫咪库 / 管理员模块 |
| 判断头像是否可展示 | 用户与个人中心模块 |

图片模块只负责：

```text
图片是否可上传、如何压缩、如何生成变体、如何存储、如何返回 URL。
```

业务模块负责：

```text
这张图片属于谁、用于什么页面、是否展示、是否作为封面或头像。
```

# 3. 核心设计原则

## 3.1 一张上传图对应一个 file_asset

`file_assets` 表表示一张逻辑图片资产。

例如用户上传一张猫咪照片，系统会生成：

```text
file_assets：1 条主记录
file_asset_variants：多条图片变体记录
```

示例：

```text
file_asset: 小橘照片 asset_001
  ├── thumb_sm       地图 Marker 小卡片
  ├── thumb_md       猫咪列表 / 任务列表
  ├── preview_lg     照片墙预览
  └── display        详情页展示
```

## 3.2 业务表只关心图片资产，不直接管理多尺寸图片

业务表不需要增加很多字段：

```text
thumb_sm_url
thumb_md_url
preview_url
display_url
webp_url
...
```

这会让每张业务表越来越臃肿。

正确方式是：

```text
业务表保存 file_asset_id
文件模块保存不同 variant
接口层根据场景返回合适 variant_url
```

## 3.3 可以保留少量高频 URL 冗余

为了避免高频列表接口每次 JOIN `file_asset_variants`，业务表可以保留一个常用 URL 快照：

```text
file_url / display_url       标准展示图 URL
thumbnail_url / thumb_url    默认缩略图 URL
avatar_url                   当前头像展示 URL
```

这些字段不是图片元数据的唯一来源，只是查询优化字段。

真正的图片元数据来源是：

```text
file_assets
file_asset_variants
```

## 3.4 V1 不保留用户原始大图

V1 中，用户手机原图只在上传处理期间临时存在。

数据库只记录源文件摘要信息：

```text
source_filename
source_mime_type
source_size_bytes
source_width
source_height
```

V1 不保存：

```text
original_url
original_object_key
```

后期如果确实需要保留原始图，可以给 `file_assets` 增加：

```text
original_object_key
original_url
original_size_bytes
```

## 3.5 V1 不做图片上传日志表

本版不建立：

```text
file_upload_logs
```

原因：

1. MVP 阶段图片上传流程应保持简单。
2. 后端应用日志和统一 `trace_id` 已可覆盖基础排错。
3. 数据库不需要额外承担大量操作日志写入。
4. 如果后期 COS 上传失败排查需求变强，再单独引入日志或审计表。

# 5. file_assets 文件资产主表

## 5.1 表名

```text
file_assets
```

## 5.2 作用

`file_assets` 表保存每一张上传图片的主记录。

它表示图片的逻辑身份，而不是某一个具体尺寸文件。

例如：

```text
用户上传“小橘在体育馆后草坪”的照片
```

系统中应该只有一条 `file_assets` 记录，但可能有多条 `file_asset_variants`：

```text
thumb_sm
thumb_md
preview_lg
display
```

---

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 文件资产 ID |

| `owner_id` | uuid | nullable | 业务归属 ID |

| `uploaded_by` | uuid | FK -> users.id, nullable | 上传人 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

## 5.4 建表示例

```sql
CREATE TABLE file_assets (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),

    storage_provider varchar(32) NOT NULL DEFAULT 'tencent_cos',
    bucket varchar(128) NOT NULL,
    region varchar(64) NOT NULL,
    env varchar(32) NOT NULL DEFAULT 'dev',

    usage_type varchar(64) NOT NULL,
    owner_type varchar(64) NULL,
    owner_id uuid NULL,

    source_filename varchar(255) NULL,
    source_mime_type varchar(64) NULL,
    source_size_bytes bigint NULL,
    source_width int NULL,
    source_height int NULL,
    source_checksum_sha256 varchar(128) NULL,

    default_variant_key varchar(64) NOT NULL DEFAULT 'display',
    default_url varchar(1024) NULL,
    default_thumb_variant_key varchar(64) NOT NULL DEFAULT 'thumb_md',
    default_thumb_url varchar(1024) NULL,

    process_preset varchar(64) NOT NULL,
    process_status varchar(32) NOT NULL DEFAULT 'completed',
    visibility varchar(32) NOT NULL DEFAULT 'public',

    uploaded_by uuid NULL REFERENCES users(id),
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz NULL
);
```

## 5.5 字段语义说明

### default_url

`default_url` 是默认展示图 URL。

一般对应 `file_asset_variants.variant_key = 'display'`。

用于业务表快速冗余：

```text
cat_photos.file_url
map_point_photos.file_url
task_checkin_photos.file_url
```

### default_thumb_url

`default_thumb_url` 是默认缩略图 URL。

普通照片一般对应：

```text
thumb_md
```

头像一般对应：

```text
avatar_md
```

地图 Marker 场景不要直接依赖 `default_thumb_url`，而应根据接口场景取：

```text
thumb_sm
```

### owner_type / owner_id

`owner_type` 和 `owner_id` 用于表达文件归属。

在部分上传流程中，图片可能先上传后绑定业务对象，此时可以先写：

```text
owner_type = temporary
owner_id = null
```

业务对象创建成功后，再回填真实归属。

## 5.8 process_status 枚举

| 值 | 说明 |
|---|---|
| `processing` | 处理中，异步处理时使用 |
| `completed` | 已完成 |
| `failed` | 处理失败 |
| `deleted` | 已删除 |

V1 采用同步处理，正常情况下创建成功即为：

```text
completed
```

`process_status` 只描述转码、变体和 COS 写入，不代表内容安全通过；发布必须同时检查 `security_status`。

## 5.9 security_status 内容安全状态

| 值 | 说明 |
|---|---|
| `legacy` | 1.1.2 以前已存在的历史图片，保持原有关联，不允许重新绑定成新内容 |
| `pending` | 微信已受理，尚未回调；不可读取、绑定或发布 |
| `passed` | 微信审核通过，可发布 |
| `rejected` | 微信返回 risky/review，不可发布 |
| `failed` | 微信调用或回调失败，不可发布 |

## 5.10 visibility 枚举

| 值 | 说明 |
|---|---|
| `public` | 可公开访问，适合 MVP 公有读桶或 CDN 访问 |
| `private` | 需要签名 URL 或后端代理 |
| `admin_only` | 仅管理员可访问 |

# 6. file_asset_variants 图片变体表

## 6.1 表名

```text
file_asset_variants
```

## 6.2 作用

`file_asset_variants` 保存同一张图片的不同尺寸和不同用途版本。

例如一张猫咪照片可以生成：

| variant_key | 用途 |
|---|---|
| `thumb_sm` | 地图 Marker 小卡片 |
| `thumb_md` | 猫咪列表、任务列表 |
| `preview_lg` | 照片墙预览 |
| `display` | 详情页展示 |

头像图片可以生成：

| variant_key | 用途 |
|---|---|
| `avatar_sm` | 成员头像小图 |
| `avatar_md` | 个人中心头像预览 |
| `avatar_lg` | 头像展示图 |

---

## 6.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 变体 ID |
| `file_asset_id` | uuid | FK -> file_assets.id, not null | 所属图片资产 |
| `variant_key` | varchar(64) | not null | 变体标识 |
| `object_key` | varchar(512) | not null | COS 对象 Key |
| `url` | varchar(1024) | not null | 访问 URL |
| `mime_type` | varchar(64) | not null | 输出 MIME，例如 `image/jpeg` |
| `file_ext` | varchar(16) | not null | 文件扩展名，例如 `jpg` |
| `width` | int | not null | 图片宽度 |
| `height` | int | not null | 图片高度 |
| `size_bytes` | bigint | not null | 文件大小 |
| `quality` | int | nullable | 输出质量参数 |
| `resize_mode` | varchar(32) | not null | 缩放模式 |
| `checksum_sha256` | varchar(128) | nullable | 当前变体文件 sha256 |
| `sort_order` | int | not null, default 0 | 排序 |
| `created_at` | timestamptz | not null | 创建时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

---

## 6.4 建表示例

```sql
CREATE TABLE file_asset_variants (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    file_asset_id uuid NOT NULL REFERENCES file_assets(id),

    variant_key varchar(64) NOT NULL,
    object_key varchar(512) NOT NULL,
    url varchar(1024) NOT NULL,

    mime_type varchar(64) NOT NULL DEFAULT 'image/jpeg',
    file_ext varchar(16) NOT NULL DEFAULT 'jpg',

    width int NOT NULL,
    height int NOT NULL,
    size_bytes bigint NOT NULL,

    quality int NULL,
    resize_mode varchar(32) NOT NULL DEFAULT 'fit',
    checksum_sha256 varchar(128) NULL,
    sort_order int NOT NULL DEFAULT 0,

    created_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz NULL,

    CONSTRAINT uq_file_asset_variants_asset_key
        UNIQUE (file_asset_id, variant_key)
);
```

## 6.6 variant_key 枚举

### 普通照片变体

| variant_key | 用途 | 生成规则 |
|---|---|---|
| `thumb_sm` | 地图 Marker 小卡片、小型头像旁图 | 长边 160px |
| `thumb_md` | 猫咪列表、任务列表、点位卡片 | 长边 320px |
| `preview_lg` | 图片墙预览、详情轻量预览 | 长边 720px |
| `display` | 普通详情页展示图 | 长边 1280px |

### 头像变体

| variant_key | 用途 | 生成规则 |
|---|---|---|
| `avatar_sm` | 列表头像、任务参与人头像 | 128×128 |
| `avatar_md` | 个人中心头像、成员卡片 | 256×256 |
| `avatar_lg` | 头像详情展示、裁剪后大图 | 512×512 |

## 6.7 resize_mode 枚举

| 值 | 说明 | 适用场景 |

## 8.1 user_profiles 用户资料表

### 当前字段

```text

```

### 建议新增字段

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `avatar_asset_id` | uuid | FK -> file_assets.id, nullable | 用户头像文件资产 ID |
| `avatar_thumb_url` | varchar(1024) | nullable | 头像默认缩略图 URL，通常来自 `avatar_md` 或 `avatar_sm` |
| `avatar_review_asset_id` | uuid | FK -> file_assets.id, nullable | 最近一次头像审核资产；用于防止乱序回调覆盖新头像 |
| `avatar_review_status` | varchar(32) | not null, default `idle` | `idle/pending/passed/rejected/failed` |
| `avatar_review_updated_at` | timestamptz | nullable | 最近一次头像审核状态更新时间 |

### 字段语义

| 字段 | 语义 |
|---|---|
| `avatar_url` | 头像展示图 URL，通常来自 `avatar_lg` 或 `avatar_md` |
| `avatar_thumb_url` | 头像缩略图 URL，列表、参与人头像优先使用 |
| `avatar_asset_id` | 头像文件资产源头 |

### 迁移 SQL

```sql
ALTER TABLE user_profiles
ADD COLUMN avatar_asset_id uuid NULL REFERENCES file_assets(id),
ADD COLUMN avatar_thumb_url varchar(1024) NULL;

CREATE INDEX idx_user_profiles_avatar_asset
ON user_profiles (avatar_asset_id)
WHERE avatar_asset_id IS NOT NULL;
```

### 业务规则

1. 用户上传头像后，写入 `file_assets` 和 `file_asset_variants`。
2. 提交审核时只更新 `avatar_review_asset_id/status`，旧头像继续展示。
3. 回调通过且回调 asset 仍是最新 review asset 时，更新 `avatar_asset_id`。
4. `user_profiles.avatar_url` 冗余保存通过审核的 `avatar_lg` 或 `avatar_md`。
5. `user_profiles.avatar_thumb_url` 冗余保存通过审核的 `avatar_sm` 或 `avatar_md`。
6. 成员列表、任务参与人头像等轻量场景优先使用 `avatar_thumb_url`。

# 12. 上传写入流程

## 12.1 普通图片上传流程

```text
1. 前端选择图片
2. 可选：小程序端轻量压缩，减少上传体积
3. POST /api/v1/files/images
4. 后端校验图片格式、大小、像素
5. 后端读取图片，修正方向，去除 EXIF
6. 根据 usage_type 选择 process_preset
7. 生成多个 variant
8. 上传所有 variant 到 COS
9. 写入 file_assets
10. 写入 file_asset_variants
11. 回填 file_assets.default_url / default_thumb_url
12. 调用微信 mediaCheckAsync，保存 trace_id 并返回 pending ImageAssetDTO
13. 微信回调 passed 后才向业务模块返回可发布 URL
14. 业务模块保存 asset_id 和常用 URL 冗余
```

---


# 14. 数据一致性规则

## 14.1 上传成功定义

只有同时满足以下条件，才算上传成功：

```text
1. 所有必要 variant 已生成
2. 所有必要 variant 已上传 COS
3. file_assets 已写入数据库
4. file_asset_variants 已写入数据库
5. file_assets.default_url / default_thumb_url 已回填
6. 业务表已写入或返回 asset_id 给前端
7. 微信审核请求已受理；业务发布还必须等待 security_status = passed
```

---

## 14.2 失败处理

| 失败点 | 处理方式 |
|---|---|
| 图片校验失败 | 不上传 COS，直接返回 650xx |
| 生成某个必要 variant 失败 | 不上传 COS，返回失败 |
| 部分 variant 上传 COS 成功，后续失败 | 尝试删除已上传 COS 对象；删除失败则写应用日志 |
| `file_assets` 写入失败 | 尝试删除已上传 COS 对象 |
| `file_asset_variants` 写入失败 | 回滚数据库事务，并尝试删除 COS 对象 |
| 业务表写入失败 | `file_assets.owner_type` 可保留为 `temporary`，后续清理 |

---

## 14.3 temporary 文件

如果前端先上传图片，再创建业务对象，可以先写：

```text
owner_type = temporary
owner_id = null
```

业务对象创建成功后再绑定：

```text
owner_type = cat / map_point / task_checkin
owner_id = 对应业务 ID
```

后期可以定时清理超过 24 小时未绑定的 temporary 图片。

V1 可以先不做定时物理删除，只在应用日志中记录。

# 15. 删除策略

## 15.1 业务删除

删除业务图片时：

```text
业务图片表 deleted_at = now()
file_assets.deleted_at = now()
file_asset_variants.deleted_at = now()
```

V1 不立即物理删除 COS 对象。

原因：

1. 避免误删后无法恢复。
2. 避免数据库事务成功但 COS 删除失败导致状态复杂。
3. MVP 阶段数据量较小，延迟物理删除成本低。

---

## 15.2 头像替换

用户或猫咪更换头像时：

```text
1. 上传新头像
2. 更新 avatar_asset_id / avatar_url / avatar_thumb_url
3. 旧头像 file_assets 可标记为 deleted，也可以保留为历史头像
```

V1 建议：

```text
用户头像旧图标记 deleted_at
猫咪头像旧图保留在 cat_photos 中，除非管理员主动删除
```

# 18. 最终推荐落地方案

V1.1 推荐采用下面的设计：

```text
新增：
- file_assets
- file_asset_variants

删除 / 不做：
- file_upload_logs
- file_cleanup_tasks

修改：
- user_profiles 增加 avatar_asset_id、avatar_thumb_url
- cats 增加 avatar_asset_id、avatar_thumb_url
- cat_photos 增加 asset_id、display_url、thumb_url、width、height、size_bytes、deleted_at
- map_point_photos 增加 asset_id、display_url、thumb_url、width、height、size_bytes、deleted_at
- map_points 确认 cover_photo_id
- task_checkin_photos 增加 asset_id、display_url、thumb_url、uploaded_by、width、height、size_bytes、deleted_at

保留兼容字段：
- avatar_url
- file_url
- thumbnail_url

核心能力：
- 一张图片一个 file_asset
- 多个缩略图 / 展示图进入 file_asset_variants
- 接口层根据页面场景返回合适 variant
- 前端不直接拼 COS URL
- 业务表不膨胀为一堆缩略图字段
```

这套设计可以覆盖当前项目中的所有图片使用场景，也能避免后续猫咪库、地图模块、任务模块各自扩展缩略图字段导致库表混乱。
