# 校园猫协地图任务系统｜暑假喂食任务模块库表设计文档

> 版本：Summer Feeding V1  
> 数据库：PostgreSQL + PostGIS  
> 后端建议：FastAPI + SQLAlchemy + Alembic  
> 设计目标：在现有总库表设计基础上，支持暑假喂食任务的发布、地图点位、日历执行日期、任务图片、完成记录和任务动态，并为后期完整任务模块预留字段。

---

## 1. 设计结论

当前总库表中的 `tasks`、`map_points`、`task_checkins`、`task_checkin_photos`、`task_activity_logs` 可以支撑任务基础闭环，但对暑假喂食任务有一个核心缺口：

```text
现有 tasks.start_at / deadline_at 更适合一次性任务或有开始截止时间的任务，不能直接表达“暑假期间管理员从日历中选择若干个具体执行日期”。
```

因此暑假版建议新增一张执行日期表：

```text
task_execution_dates
```

同时建议新增任务图片表：

```text
task_photos
```

原因：任务详情页顶部轮播图属于任务展示内容，不完全等同于地图点位照片，也不同于成员完成打卡照片。

## 2. 本版本表结构清单

### 2.1 直接复用现有表

| 表名 | 用途 | 本版本处理 |
|---|---|---|
| `users` | 成员与管理员 | 复用 |
| `user_profiles` | 展示昵称、头像 | 复用 |
| `campuses` | 校园配置 | 复用 |
| `campus_areas` | 校园区域 | 复用 |
| `map_points` | 任务地图点 | 复用，发布任务时写入或复用 |
| `tasks` | 任务主表 | 复用，并建议补充少量扩展字段 |
| `task_checkins` | 完成投喂记录 | 复用，并建议关联执行日期 |
| `task_checkin_photos` | 完成照片 | 复用，可选 |
| `task_activity_logs` | 任务动态时间线 | 复用，并建议关联执行日期 |
| `admin_operation_logs` | 管理员操作日志 | 复用，可选但建议写入 |
| `notifications` | 通知 | 可后置 |

### 2.2 新增表

| 表名 | 用途 | 是否必做 |
|---|---|---:|
| `task_execution_dates` | 保存一个长期喂食任务的具体执行日期和每日完成状态 | 必做 |
| `task_photos` | 保存任务详情页轮播图、封面图 | 建议必做 |

### 2.3 本版本暂不启用或不依赖

| 表名 | 处理方式 |
|---|---|
| `task_participants` | 本版本不做接取、放弃、满员，暂不依赖 |
| `task_assignments` | 本版本不做指派流程，保留后期扩展 |
| `duty_schedules` | 本版本不做排班筛选，保留后期扩展 |
| `task_review_records` | 本版本不做审核，保留后期扩展 |
| `task_required_supplies` | 当前物资用 `tasks.required_items = '猫粮、水'`，结构化物资后置 |

## 3. 数据关系

```text
map_points 1 ─── n tasks
                   │
                   ├── n task_execution_dates
                   │        │
                   │        └── 0/1 task_checkins
                   │                 └── n task_checkin_photos
                   │
                   ├── n task_photos
                   │
                   └── n task_activity_logs
```

说明：

1. `map_points` 保存任务点在哪里。
2. `tasks` 保存暑假喂食任务本身。
3. `task_execution_dates` 保存管理员日历选择的每一天。
4. `task_checkins` 保存成员完成某一天投喂的记录。
5. `task_photos` 保存任务详情页轮播图。
6. `task_activity_logs` 保存任务动态时间线。

## 4. map_points 地图点位表使用规则

### 4.1 表用途

`map_points` 只保存空间位置和地图展示信息。

暑假喂食任务发布时：

```text
point_type = 'task'
point_scope = 'long_term'
icon_key = 'task_feeding' 或 'task_summer_feeding'
```

如果某个喂食点未来长期复用，也可以考虑：

```text
point_type = 'feeding'
point_scope = 'long_term'
```

但为了与现有地图任务点卡片逻辑保持一致，当前建议优先使用：

```text
point_type = 'task'
```

### 4.2 本版本关键字段

| 字段 | 使用方式 |
|---|---|
| `id` | 任务表 `tasks.map_point_id` 引用 |
| `campus_id` | 所属校园 |
| `area_id` | 所属区域，可为空 |
| `point_type` | 固定为 `task` |
| `point_scope` | 建议为 `long_term` |
| `name` | 与任务名称一致或保存点位名称 |
| `description` | 点位说明，可与任务说明不同 |
| `location_name` | 如“学生宿舍区北侧” |
| `location_detail` | 详细位置描述 |
| `lng` / `lat` / `geom` | 经纬度与空间点 |
| `route_instruction` | 校园内部路线说明 |
| `landmark_hint` | 附近地标提示 |
| `entrance_hint` | 入口提示 |
| `icon_key` | Marker 图标 |
| `visibility` | 默认 `public` |
| `status` | 默认 `active` |

## 5. tasks 任务主表

### 5.1 表用途

保存暑假喂食任务本身。

一条 `tasks` 记录代表一个固定喂食点在暑假期间的一组投喂安排。

### 5.2 现有字段使用规则

| 字段 | 本版本使用方式 |
|---|---|
| `id` | 任务 ID |
| `task_no` | 系统生成任务编号 |
| `title` | 任务名称，如“学生宿舍区北侧喂食点” |
| `task_type` | 固定为 `feeding` |
| `priority` | 默认 `normal` |
| `status` | 发布后为 `in_progress` |
| `map_point_id` | 关联地图点 |
| `area_id` | 冗余保存区域 ID，方便筛选 |
| `related_cat_id` | 本版本可为空 |
| `publisher_id` | 管理员 ID |
| `max_participants` | 本版本不使用，可保留默认值 1 |
| `participant_count` | 本版本不使用，保持 0 |
| `description` | 任务说明，默认“暑假投喂” |
| `route_instruction` | 可从 `map_points.route_instruction` 冗余 |
| `required_items` | 默认“猫粮、水” |
| `start_at` | 可存最早执行日期 00:00 |
| `deadline_at` | 可存最晚执行日期 23:59 |
| `published_at` | 发布时间 |
| `completed_at` | 整个暑假任务结束时间，不代表单日完成 |
| `cancelled_at` | 整个任务取消时间 |
| `is_public` | 默认 true |

### 5.3 建议新增字段

为了让任务主表能区分一次性任务和长期任务，建议新增以下字段，允许为空或默认值兼容旧数据。

| 字段名 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `task_mode` | varchar(32) | `single` | 任务模式：`single` 一次性、`recurring` 多日期/周期任务 |
| `schedule_type` | varchar(32) | `once` | 时间安排类型：`once`、`selected_dates`、`weekly`，暑假版用 `selected_dates` |
| `completion_policy` | varchar(32) | `task` | 完成策略：`task` 整体完成、`per_execution_date` 按执行日期完成 |
| `material_summary` | varchar(255) | null | 物资摘要，暑假版可冗余“猫粮、水”；也可直接使用 `required_items` |

如果希望改动更少，可以只新增 `task_mode` 和 `schedule_type`，物资继续使用 `required_items`。

### 5.4 枚举建议

#### task_type

本版本只使用：

| 值 | 说明 |
|---|---|
| `feeding` | 喂食任务 |

后期保留：

```text
daily / emergency / patrol / supply / rescue / medical / capture / observation
```

#### status

| 值 | 说明 | 本版本是否使用 |
|---|---|---:|
| `pending` | 未发布 / 草稿 | 可选 |
| `in_progress` | 进行中 | 是 |
| `completed` | 整个任务周期结束 | 是 |
| `cancelled` | 已取消 | 是 |
| `pending_review` | 待审核 | 本版本不用 |
| `archived` | 已归档 | 可选扩展 |

### 5.5 建议索引

```sql
CREATE INDEX idx_tasks_summer_feeding
ON tasks (task_type, status, published_at DESC)
WHERE deleted_at IS NULL;

CREATE INDEX idx_tasks_map_point
ON tasks (map_point_id)
WHERE deleted_at IS NULL;

CREATE INDEX idx_tasks_schedule
ON tasks (task_mode, schedule_type, status)
WHERE deleted_at IS NULL;
```

## 6. task_execution_dates 任务执行日期表

### 6.1 表名

```text
task_execution_dates
```

### 6.2 作用

保存管理员在日历里选择的具体执行日期。

一条记录代表：

```text
某个暑假喂食任务在某一天需要执行一次。
```

这是暑假版最关键的新增表。

### 6.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 执行日期 ID |
| `task_id` | uuid | FK -> tasks.id, not null | 所属任务 |
| `execute_date` | date | not null | 执行日期 |
| `status` | varchar(32) | not null, default `pending` | 当日执行状态 |
| `completed_by` | uuid | FK -> users.id, nullable | 完成人 |
| `completed_at` | timestamptz | nullable | 完成时间 |
| `checkin_id` | uuid | FK -> task_checkins.id, nullable | 对应完成记录，可选冗余 |
| `cancelled_by` | uuid | FK -> users.id, nullable | 取消人 |
| `cancelled_at` | timestamptz | nullable | 取消时间 |
| `cancel_reason` | text | nullable | 当日取消原因 |
| `remark` | text | nullable | 当日备注 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

### 6.4 status 枚举

| 值 | 说明 |
|---|---|
| `pending` | 待完成 |
| `completed` | 已完成 |
| `cancelled` | 当日取消 |
| `skipped` | 当日跳过，不计为异常 |
| `missed` | 过期未完成，可作为存储状态或查询态 |

### 6.5 约束与索引

```sql
CREATE UNIQUE INDEX uq_task_execution_dates_task_date
ON task_execution_dates (task_id, execute_date)
WHERE deleted_at IS NULL;

CREATE INDEX idx_task_execution_dates_task
ON task_execution_dates (task_id, execute_date);

CREATE INDEX idx_task_execution_dates_date_status
ON task_execution_dates (execute_date, status);

CREATE INDEX idx_task_execution_dates_completed_by
ON task_execution_dates (completed_by, completed_at DESC)
WHERE completed_by IS NOT NULL;
```

### 6.6 建表示例

```sql
CREATE TABLE task_execution_dates (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id uuid NOT NULL REFERENCES tasks(id),
    execute_date date NOT NULL,
    status varchar(32) NOT NULL DEFAULT 'pending',
    completed_by uuid NULL REFERENCES users(id),
    completed_at timestamptz NULL,
    checkin_id uuid NULL,
    cancelled_by uuid NULL REFERENCES users(id),
    cancelled_at timestamptz NULL,
    cancel_reason text NULL,
    remark text NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz NULL
);
```

> 如果迁移时 `task_checkins` 已存在，可以在两个表创建后再添加 `checkin_id` 外键，避免循环依赖问题。

## 7. task_photos 任务图片表

### 7.1 表名

```text
task_photos
```

### 7.2 作用

保存任务详情页顶部轮播图和任务封面图。

与其他图片表的区别：

| 表 | 用途 |
|---|---|
| `task_photos` | 任务发布时上传，任务详情页展示 |
| `map_point_photos` | 点位现场图、路线图、入口图 |
| `task_checkin_photos` | 成员完成投喂时上传的打卡照片 |

### 7.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 图片记录 ID |
| `task_id` | uuid | FK -> tasks.id, not null | 所属任务 |
| `file_id` | uuid | nullable | 统一上传模块返回的文件 ID；如果上传模块有文件表则关联 |
| `file_url` | varchar(512) | not null | 原图 URL |
| `thumbnail_url` | varchar(512) | nullable | 缩略图 URL |
| `cos_object_key` | varchar(512) | nullable | COS 对象 key，可选 |
| `photo_type` | varchar(32) | not null, default `scene` | 图片类型 |
| `caption` | varchar(255) | nullable | 图片说明 |
| `sort_order` | int | not null, default 0 | 排序 |
| `is_cover` | boolean | not null, default false | 是否封面 |
| `uploaded_by` | uuid | FK -> users.id, nullable | 上传人 |
| `created_at` | timestamptz | not null | 创建时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

### 7.4 photo_type 枚举

| 值 | 说明 |
|---|---|
| `cover` | 封面图 |
| `scene` | 现场图 |
| `route` | 路线参考图 |
| `reference` | 参考图 |

### 7.5 约束与索引

```sql
CREATE INDEX idx_task_photos_task
ON task_photos (task_id, sort_order)
WHERE deleted_at IS NULL;

CREATE UNIQUE INDEX uq_task_photos_cover
ON task_photos (task_id)
WHERE is_cover = true AND deleted_at IS NULL;
```

### 7.6 建表示例

```sql
CREATE TABLE task_photos (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id uuid NOT NULL REFERENCES tasks(id),
    file_id uuid NULL,
    file_url varchar(512) NOT NULL,
    thumbnail_url varchar(512) NULL,
    cos_object_key varchar(512) NULL,
    photo_type varchar(32) NOT NULL DEFAULT 'scene',
    caption varchar(255) NULL,
    sort_order int NOT NULL DEFAULT 0,
    is_cover boolean NOT NULL DEFAULT false,
    uploaded_by uuid NULL REFERENCES users(id),
    created_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz NULL
);
```

## 8. task_checkins 完成记录表调整

### 8.1 表用途

保存成员完成投喂的记录。

暑假版中，一条完成记录应关联到某一个执行日期。

### 8.2 建议新增字段

| 字段名 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `task_execution_date_id` | uuid | null | 关联 `task_execution_dates.id` |
| `execute_date` | date | null | 冗余执行日期，方便查询 |
| `checkin_type` | varchar(32) | `feeding` | 打卡类型，暑假版固定 feeding |
| `checkin_lng` | numeric(10,7) | null | 完成时定位经度，可选 |
| `checkin_lat` | numeric(10,7) | null | 完成时定位纬度，可选 |

如果现有 `task_checkins` 已有类似字段，只需要保持命名一致或在接口层映射。

### 8.3 本版本关键字段

| 字段 | 使用方式 |
|---|---|
| `task_id` | 所属任务 |
| `user_id` / `submitter_id` | 完成人 |
| `task_execution_date_id` | 对应执行日期 |
| `execute_date` | 执行日期冗余 |
| `is_completed` | 默认 true |
| `process_result` | 如“已补充猫粮和水” |
| `remark` | 备注 |
| `submitted_at` | 提交时间 |
| `review_status` | 本版本固定 `no_review` |

### 8.4 索引建议

```sql
CREATE INDEX idx_task_checkins_execution_date
ON task_checkins (task_execution_date_id);

CREATE INDEX idx_task_checkins_task_execute_date
ON task_checkins (task_id, execute_date);

CREATE INDEX idx_task_checkins_submitter_time
ON task_checkins (submitter_id, submitted_at DESC);
```

字段名如果当前表使用 `user_id` 而不是 `submitter_id`，索引按实际字段调整。

## 10. task_activity_logs 任务动态表调整

### 10.1 表用途

保存任务详情页时间线。

暑假版中，动态应围绕“谁在什么时候完成了哪一天投喂”来展示。

### 10.2 建议新增字段

| 字段名 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `task_execution_date_id` | uuid | null | 关联执行日期，可为空，方便按日期筛选 |

如果不想改表，也可以放在 `metadata`：

```json
{
  "task_execution_date_id": "uuid",
  "execute_date": "2026-07-02"
}
```

为了查询清晰，建议新增 nullable 字段。

### 10.3 activity_type 枚举扩展

现有动态类型可以继续使用，同时补充以下类型：

| 值 | 说明 |
|---|---|
| `created` | 发布任务 |
| `updated` | 修改任务 |
| `checkin_submitted` | 提交完成记录 |
| `execution_completed` | 某个执行日期已完成 |
| `execution_cancelled` | 某个执行日期取消 |
| `execution_reopened` | 撤销某天完成，重新打开 |
| `cancelled` | 整个任务取消 |
| `archived` | 任务归档 |

### 10.4 动态内容示例

```json
{
  "activity_type": "execution_completed",
  "title": "第一次任务完成",
  "content": "Nietzsche 于 2026-07-02 18:24 完成投喂",
  "metadata": {
    "execute_date": "2026-07-02",
    "completed_at": "2026-07-02T18:24:00+08:00"
  }
}
```

### 10.5 索引建议

```sql
CREATE INDEX idx_task_activity_logs_execution_date
ON task_activity_logs (task_execution_date_id, created_at DESC)
WHERE task_execution_date_id IS NOT NULL;
```

## 11. admin_operation_logs 管理员操作日志

### 11.1 使用场景

建议记录以下操作：

| 操作 | operation_type | target_type |
|---|---|---|
| 发布暑假喂食任务 | `publish` | `task` |
| 修改任务 | `update` | `task` |
| 取消任务 | `cancel` | `task` |
| 修改地图点 | `update` | `map_point` |
| 上传任务图片 | `create` | `task` 或 `task_photo` |
| 修改执行日期 | `update` | `task_execution_date` |

如果当前 `target_type` 枚举没有 `task_photo`、`task_execution_date`，可以先用：

```text
target_type = task
```

并把具体对象 ID 放入 `after_data` 或 `metadata`。

## 12. 发布任务写入流程

### 12.1 请求入参核心结构

```json
{
  "title": "学生宿舍区北侧喂食点",
  "description": "暑假投喂",
  "required_items": "猫粮、水",
  "map_point": {
    "lng": 114.1234567,
    "lat": 30.1234567,
    "location_name": "学生宿舍区北侧",
    "location_detail": "靠近教学楼B",
    "route_instruction": "到达教学楼B后沿右侧小路进入"
  },
  "execute_dates": [
    "2026-07-02",
    "2026-07-09",
    "2026-07-16"
  ],
  "photos": [
    {
      "file_id": "uuid",
      "file_url": "https://cos.example.com/task/1.jpg",
      "thumbnail_url": "https://cos.example.com/task/1_thumb.jpg",
      "is_cover": true,
      "sort_order": 0
    }
  ]
}
```

### 12.2 后端事务流程

```text
1. 校验管理员权限
2. 校验 execute_dates 非空且日期合法
3. 创建或复用 map_points
4. 创建 tasks
5. 批量写入 task_execution_dates
6. 写入 task_photos
7. 写入 task_activity_logs：created
8. 写入 admin_operation_logs：publish
9. 提交事务
```

### 12.3 父任务时间字段生成规则

```text
tasks.start_at = min(execute_dates) 00:00:00
tasks.deadline_at = max(execute_dates) 23:59:59
tasks.status = in_progress
```

## 13. 完成投喂写入流程

### 13.1 请求入参核心结构

```json
{
  "execute_date": "2026-07-02",
  "is_completed": true,
  "process_result": "已补充猫粮和水",
  "remark": "现场正常",
  "photo_ids": ["uuid"],
  "checkin_lng": 114.1234567,
  "checkin_lat": 30.1234567
}
```

### 13.2 后端事务流程

```text
1. 校验用户已登录
2. 查询 tasks，确认任务存在且 status = in_progress
3. 查询 task_execution_dates，确认 execute_date 属于该任务
4. 如果该执行日期已 completed，返回重复完成错误
5. 创建 task_checkins
6. 写入 task_checkin_photos，若有完成照片
7. 更新 task_execution_dates.status = completed
8. 写入 completed_by、completed_at、checkin_id
9. 写入 task_activity_logs：execution_completed / checkin_submitted
10. 刷新任务详情返回给前端
```

### 13.3 是否修改 tasks.status

完成某一天投喂时：

```text
不修改 tasks.status 为 completed。
```

只有满足以下条件之一，才修改父任务状态：

1. 所有执行日期都已完成，且管理员点击归档或结束任务。
2. 所有执行日期都已结束，系统后期通过定时任务自动结束。
3. 管理员取消整个任务。

## 17. 迁移建议

### 17.1 最小迁移清单

```text
1. tasks 增加 task_mode、schedule_type、completion_policy，可选
2. 新建 task_execution_dates
3. 新建 task_photos
4. task_checkins 增加 task_execution_date_id、execute_date，可选但强烈建议
5. task_activity_logs 增加 task_execution_date_id，可选
```

### 17.2 最小可用方案

如果希望最快落地，可以只做：

```text
1. 新建 task_execution_dates
2. 新建 task_photos
3. task_checkins 增加 task_execution_date_id
```

`tasks.task_mode` 等字段可以后置，用 `task_type = feeding` + 是否存在执行日期判断是否为暑假喂食任务。

### 17.3 推荐落地方案

推荐一次性加入：

```text
tasks.task_mode
tasks.schedule_type
tasks.completion_policy
task_execution_dates
task_photos
task_checkins.task_execution_date_id
task_checkins.execute_date
task_activity_logs.task_execution_date_id
```

这些字段不复杂，能明显降低后续扩展成本。

| 统计缓存 | 启用 `user_stats` |
