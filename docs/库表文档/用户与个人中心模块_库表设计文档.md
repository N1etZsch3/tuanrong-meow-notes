# 校园猫协地图任务系统｜用户与个人中心模块库表设计文档

> 版本：V1.1.1 独立模块设计稿
> 适用范围：用户认证、用户资料、个人中心、我的任务、我的打卡、我的观察记录、我的排班、我的指派、通知入口、个人统计、管理员入口判断
> 数据库：PostgreSQL + PostGIS
> 后端建议：FastAPI + SQLAlchemy + Alembic
> 说明：本文档从总库表设计与项目说明中抽离「用户与个人中心」相关内容，并补充首登改密、资料初始化、个人中心聚合查询的实现边界。

---

## 1. 模块定位

用户与个人中心模块负责解决三个层面的事情：

1. **身份层**：喵喵号登录、密码校验、验证码、微信 OpenID 绑定、JWT 鉴权、角色判断、账号禁用、首登改密。
2. **资料层**：头像、昵称、真实姓名、部门/专业、年级、加入猫协时间、个人简介、擅长任务类型、个人设置。
3. **聚合层**：个人中心首页、我的任务、我的打卡、我的观察记录、我的排班、指派给我的任务、通知入口、个人统计、管理员入口展示。

移动端底部 Tab 中「我的」页面属于本模块；管理员入口不单独放到底部 Tab，而是在当前用户角色为 `admin` 时出现在个人中心。

---

## 2. 设计原则

### 2.1 喵喵号是邀请制入口，微信 OpenID 是绑定后的自动登录标识

V1.1.1 支持：

```text
喵喵号 + 密码 + 字母验证码
已绑定微信 OpenID 后自动登录
```

不支持成员自主注册、手机号登录、邮箱登录、QQ/其他第三方登录、找回密码。成员账号由管理员创建或批量导入，成员首次绑定微信前仍必须先拥有喵喵号。

微信 OpenID 只用于本小程序内的自动登录和账号保护。前端不能直接提交 OpenID，必须由后端使用微信小程序临时 code 换取。管理员清除绑定后，成员下一次登录回到喵喵号密码兜底流程，并重新绑定当前微信。

---

### 2.2 账号数据与展示资料分离

账号安全相关字段放在：

```text
users
```

个人展示资料放在：

```text
user_profiles
```

`user_profiles` 不再重复保存学号，学号只保存在 `users.student_no`。

---

### 2.3 首次登录必须改密

管理员创建账号后，默认：

```text
users.must_change_password = true
```

成员首次登录后，只允许访问：

```text
GET /api/v1/auth/me
PATCH /api/v1/auth/password
POST /api/v1/auth/logout
```

改密成功后：

```text
users.must_change_password = false
users.token_version = token_version + 1
```

旧 JWT 因 `token_version` 不一致而失效。

---

### 2.4 用户资料初始化独立处理

为了适配首次登录后的「初始化身份」页面，建议在 `user_profiles` 中加入：

```text
profile_completed
profile_completed_at
```

初始化页面只要求用户填写第一版必要资料：

```text
昵称、头像、部门、联系方式（手机号）
```

手机号保存到 `user_profiles.contact_info`，仅管理员可见。真实姓名、年级、加入猫协时间可由管理员维护或后续补充。

---

### 2.5 个人中心不重复建“我的任务”表

「我的任务」「我的打卡」「我的观察记录」「我的统计」都通过已有业务表反查或聚合得到，不额外创建 `my_tasks`、`my_checkins` 之类的重复表。

主要来源：

| 个人中心内容 | 主要来源表 |
|---|---|
| 我的进行中任务 | `task_participants` + `tasks` |
| 我的已完成任务 | `task_participants` + `tasks` |
| 我的已放弃任务 | `task_participants` + `tasks` |
| 我的打卡记录 | `task_checkins` + `task_checkin_photos` |
| 我的观察记录 | `cat_observation_records` + `cats` |
| 我的排班 | `duty_schedules` |
| 指派给我的任务 | `task_assignments` + `tasks` |
| 通知中心 | `notifications` |
| 个人统计 | 实时 SQL 聚合；后期可启用 `user_stats` 缓存 |

# 4. 表结构设计

## 4.1 users 用户账号表

### 表名

```text
users
```

### 作用

保存成员登录账号、密码哈希、角色、账号状态。V1 中 `student_no` 是唯一登录账号，唯一约束覆盖软删除记录；删除后继续占用原喵喵号，成员回归时恢复原 `user.id`。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | uuid | PK | 用户 ID |
| student_no | varchar(64) | unique, not null | 学号，V1 登录账号 |
| password_hash | varchar(255) | not null | 密码哈希，不保存明文密码 |
| role | varchar(32) | not null, default 'member' | 用户角色 |
| status | varchar(32) | not null, default 'active' | 用户状态 |
| must_change_password | boolean | default true | 是否必须修改初始密码 |
| password_updated_at | timestamptz | nullable | 密码最近更新时间 |
| last_login_at | timestamptz | nullable | 最近登录时间 |
| login_failed_count | int | default 0 | 连续登录失败次数 |
| locked_until | timestamptz | nullable | 登录锁定截止时间 |
| token_version | int | default 1 | Token 版本号，改密、重置密码、删除或恢复账号后自增 |
| created_at | timestamptz | not null | 创建时间 |
| updated_at | timestamptz | not null | 更新时间 |
| deleted_at | timestamptz | nullable | 软删除时间 |

### role 枚举

| 值 | 说明 |
|---|---|
| member | 普通成员 |
| admin | 管理员 |
| super_admin | 超级管理员，后期扩展 |

V1 实际使用：

```text
member / admin
```

### status 枚举

| 值 | 说明 |
|---|---|
| active | 正常 |
| inactive | 未激活 |
| blocked | 禁用 |
| left | 已退出猫协 |
| deleted | 已删除 |

### 索引

```sql
CREATE UNIQUE INDEX uq_users_student_no
ON users (student_no);

CREATE INDEX idx_users_role
ON users (role);

CREATE INDEX idx_users_status
ON users (status);

CREATE INDEX idx_users_locked_until
ON users (locked_until)
WHERE locked_until IS NOT NULL;
```

### 业务规则

1. `student_no` 在包含软删除记录的全部账号中不能重复；禁止为不同 `user.id` 回收同一喵喵号。
2. `password_hash` 必须使用安全哈希算法生成，不能保存明文密码。
3. 管理员创建用户时默认 `must_change_password = true`。
4. 用户修改密码、管理员重置密码后，`token_version` 自增。
5. 当前业务将 `users.student_no` 作为喵喵号兼容存储字段。自动创建账号时由 PostgreSQL 序列 `meow_no_auto_sequence` 分配数字部分并跳过已占用值；手动指定喵喵号不调用、不推进该序列。序列由迁移 `20260716_0015` 管理。
5. 鉴权时必须比较 JWT 中的 `token_version` 与数据库当前值。
6. `status != active` 的账号不能登录。
7. `locked_until > now()` 时账号临时锁定，不能登录。
8. 恢复软删除账号时保留资料与历史关联，清除 `deleted_at`、旧微信绑定和登录锁定，重置临时密码并强制首次改密。

## 4.2 auth_captchas 登录验证码表

### 表名

```text
auth_captchas
```

### 作用

保存登录页字母验证码的短期校验记录。V1 可以先落表保存，后续可迁移到 Redis。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | uuid | PK | 验证码 ID |
| code_hash | varchar(255) | not null | 验证码哈希，不保存明文验证码 |
| scene | varchar(32) | not null, default 'login' | 使用场景 |
| expires_at | timestamptz | not null | 过期时间 |
| used_at | timestamptz | nullable | 使用时间 |
| client_ip | varchar(64) | nullable | 请求 IP |
| user_agent | text | nullable | 请求设备信息 |
| created_at | timestamptz | not null | 创建时间 |

### scene 枚举

| 值 | 说明 |
|---|---|
| login | 登录验证码 |

### 索引

```sql
CREATE INDEX idx_auth_captchas_expires_at
ON auth_captchas (expires_at);

CREATE INDEX idx_auth_captchas_scene_created
ON auth_captchas (scene, created_at DESC);

CREATE INDEX idx_auth_captchas_unused
ON auth_captchas (scene, expires_at)
WHERE used_at IS NULL;
```

### 业务规则

1. 验证码明文只返回给前端图片或 SVG，不落库。
2. 登录校验时对用户输入验证码计算哈希后比对 `code_hash`。
3. 验证码只能使用一次，使用后写入 `used_at`。
4. 已过期或已使用的验证码不能再次使用。
5. 后续接入 Redis 时，本表可保留为审计表，也可以废弃。

## 4.3 user_profiles 用户资料表

### 表名

```text
user_profiles
```

### 作用

保存个人中心展示资料，并支持首次登录后的资料初始化流程。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | uuid | PK | 主键 |
| user_id | uuid | FK -> users.id, unique, not null | 用户 ID |
| nickname | varchar(64) | not null | 昵称 |
| avatar_url | varchar(512) | nullable | 头像地址 |
| real_name | varchar(64) | nullable | 真实姓名，管理员可见 |
| department | varchar(128) | nullable | 猫协部门 |
| grade | varchar(32) | nullable | 年级 |
| joined_at | date | nullable | 加入猫协时间 |
| bio | text | nullable | 个人简介 |
| contact_info | varchar(128) | nullable | 联系方式，管理员可见 |
| profile_completed | boolean | default false | 资料是否已完成初始化 |
| profile_completed_at | timestamptz | nullable | 资料初始化完成时间 |
| created_at | timestamptz | not null | 创建时间 |
| updated_at | timestamptz | not null | 更新时间 |

### 索引

```sql
CREATE UNIQUE INDEX uq_user_profiles_user
ON user_profiles (user_id);

CREATE INDEX idx_user_profiles_department
ON user_profiles (department);

CREATE INDEX idx_user_profiles_completed
ON user_profiles (profile_completed);
```

### 业务规则

1. 每个用户只对应一条资料记录。
2. 管理员创建成员账号时，可以同步创建默认资料记录。
3. 默认昵称可以使用「猫协成员 + 学号后四位」或由管理员批量导入。
4. 用户初始化资料时至少填写：`nickname`、`department`、`contact_info`；头像未上传时前端使用默认萌猫头像。
5. `department` 限定为：生存保障部、活动部、宣传部、秘书部、养护部。
6. `contact_info` 使用手机号正则校验，并仅管理员可见。
7. 初始化完成后设置 `profile_completed = true`。
8. 普通成员只能编辑自己的 `nickname`、`avatar_url`、`department`、`bio` 和联系方式初始化项。
9. `real_name`、`joined_at` 可以限制为管理员维护。

## 4.4 user_task_preferences 用户任务偏好表

### 表名

```text
user_task_preferences
```

### 作用

保存成员擅长或愿意参与的任务类型。管理员指派任务时可以结合排班表筛选成员。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | uuid | PK | 主键 |
| user_id | uuid | FK -> users.id, not null | 用户 ID |
| task_type | varchar(32) | not null | 任务类型 |
| preference_level | int | default 3 | 偏好程度，1-5 |
| is_enabled | boolean | default true | 是否启用 |
| remark | text | nullable | 备注 |
| created_at | timestamptz | not null | 创建时间 |
| updated_at | timestamptz | not null | 更新时间 |

### task_type 枚举

| 值 | 说明 |
|---|---|
| daily | 日常任务 |
| emergency | 紧急任务 |
| feeding | 喂食任务 |
| patrol | 巡查任务 |
| supply | 物资任务 |
| rescue | 救助任务 |
| medical | 送医任务 |
| capture | 抓捕任务 |
| observation | 观察任务 |

### 索引

```sql
CREATE INDEX idx_user_task_preferences_user
ON user_task_preferences (user_id);

CREATE INDEX idx_user_task_preferences_type
ON user_task_preferences (task_type);

CREATE UNIQUE INDEX uq_user_task_preferences_user_type
ON user_task_preferences (user_id, task_type);
```

### 业务规则

1. 同一个用户对同一种任务类型只能有一条偏好记录。
2. `preference_level` 推荐限制在 1-5。
3. `is_enabled = false` 表示该成员暂不希望参与该类型任务。
4. 管理员指派任务时，可优先筛选：排班可用 + 偏好启用 + 任务类型匹配。

### 表名

```text
user_stats
```

### 作用

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | uuid | PK | 主键 |

| total_joined_tasks | int | default 0 | 累计参与任务数 |
| total_completed_tasks | int | default 0 | 累计完成任务数 |

| total_checkins | int | default 0 | 累计打卡数 |
| total_observation_records | int | default 0 | 累计观察记录数 |
| total_related_cats | int | default 0 | 参与过记录的猫咪数量 |
| emergency_task_count | int | default 0 | 紧急任务参与次数 |
| feeding_task_count | int | default 0 | 喂食任务参与次数 |
| patrol_task_count | int | default 0 | 巡查任务参与次数 |
| monthly_completed_tasks | int | default 0 | 本月完成任务数 |
| stat_month | varchar(16) | nullable | 统计月份，例如 2026-06 |
| updated_at | timestamptz | not null | 更新时间 |

### 索引

```sql
CREATE UNIQUE INDEX uq_user_stats_user
ON user_stats (user_id);
```

### 业务规则

1. V1 推荐先不写入缓存，直接聚合 `task_participants`、`task_checkins`、`cat_observation_records`。

3. 如果需要月度统计，应明确 `stat_month` 与 `user_id` 的唯一约束：

```sql
CREATE UNIQUE INDEX uq_user_stats_user_month
ON user_stats (user_id, stat_month)
WHERE stat_month IS NOT NULL;
```

# 5. 个人中心依赖表摘要

## 5.1 task_participants 任务参与人表

个人中心通过该表查询「我的任务」。

关键字段：

| 字段名 | 说明 |
|---|---|
| task_id | 任务 ID |
| user_id | 参与成员 |
| status | joined / abandoned / completed / removed |
| joined_at | 接取时间 |
| abandoned_at | 放弃时间 |
| completed_at | 成员完成时间 |

常用查询：

```sql
SELECT *
FROM task_participants tp
JOIN tasks t ON t.id = tp.task_id
WHERE tp.user_id = :current_user_id
ORDER BY tp.joined_at DESC;
```

## 5.3 cat_observation_records 猫咪观察记录表

个人中心通过该表查询「我的观察记录」。

关键字段：

| 字段名 | 说明 |
|---|---|
| observer_id / created_by | 观察人，具体字段以猫咪库表设计为准 |
| cat_id | 关联猫咪 |
| map_point_id | 观察点位 |
| observed_at | 观察时间 |

# 8. 自检

## 8.1 是否重复建表？

没有重复建表。

个人中心不单独创建 `my_tasks`、`my_checkins`、`my_observations`，而是从业务表聚合查询。

---

支持。

依赖字段：

```text
users.must_change_password
users.token_version
users.password_updated_at
```

---

支持。

建议依赖字段：

```text
user_profiles.profile_completed
user_profiles.profile_completed_at
```

---

## 8.4 是否支持管理员入口判断？

支持。

依赖字段：

```text

```

当 `role IN ('admin', 'super_admin')` 时，个人中心显示管理员入口。

## 8.5 是否支持排班与指派？

支持。

依赖表：

```text
duty_schedules
task_assignments

```

## 8.6 是否适合后续扩展？

适合。

# 9. 头衔字段当前实现（2026-07-18）

迁移：`20260718_0019_add_profile_title`，依赖 `20260718_0018_create_notifications`。

`user_profiles` 新增：

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `title` | varchar(64) | nullable | 当前头衔 key；NULL 表示无头衔 |

索引：

```sql
CREATE UNIQUE INDEX uq_user_profiles_title
ON user_profiles (title)
WHERE title IS NOT NULL;
```

该部分唯一索引允许任意数量的 NULL，但保证 12 个非空头衔各自最多只有一个持有者。合法 key 由应用层注册表校验；并发授予同时由服务层占用检查和数据库唯一索引保护。

软删除用户前将 `user_profiles.title` 置 NULL。恢复用户不恢复旧头衔。会长转让锁定双方资料并在单事务内完成旧值释放、新值设置、接任者角色提升与审计记录。

当前不建头衔历史表：本期没有任期追踪、一人多头衔或多人共享头衔需求；若后续需要任期审计，应新增独立历史表，不应从管理员操作日志反推完整任期事实。
