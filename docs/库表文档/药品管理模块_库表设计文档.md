# 校园猫协地图任务系统｜药品管理模块库表设计文档

> 版本：V1.0
> 模块名称：药品管理模块
> 适用范围：药品主档、成员持有库存、库存流水、用药申请、申请审核、药品动态、任务关联用药预留
> 数据库：PostgreSQL
> 后端建议：FastAPI + SQLAlchemy + Alembic
> 设计原则：药品不进入地图点位，不保存经纬度；库存变化必须通过流水记录；药品采用“药品主档 + 成员持有库存”的父子结构。

---

# 1. 模块定位

药品管理模块用于解决猫协内部药品的库存协作问题：

```text
有什么药
谁手里有
还剩多少
为什么变化
谁申请使用
谁审核通过
用在了哪次任务或什么场景中
```

药品默认由成员个人持有，不公开具体存放位置，也不设置地图点和经纬度。

本模块数据库设计围绕四个核心对象展开：

1. `medicine_catalogs`：药品主档，表示“这是什么药”。
2. `medicine_holdings`：成员持有库存，表示“谁手里有多少”。
3. `medicine_stock_logs`：药品库存流水，表示“库存为什么变化”。
4. `medicine_use_applications`：用药申请，表示“非持有人向具体持有人申请用药”。

---

# 2. 设计原则

## 2.1 药品不进入地图点位

药品不写入 `map_points`，不保存：

```text
lng
lat
geom
location_name
location_detail
route_instruction
```

原因：

1. 药品由成员个人持有，不适合公开位置。
2. 药品管理关注库存和流转，不关注空间导航。
3. 避免将持有人隐私暴露到地图模块。

---

## 2.2 主档与库存分离

药品主档只描述药品本身：

```text
名称
分类
规格
单位
说明
注意事项
图片
状态
```

成员持有库存描述某个成员手里的数量：

```text
药品 ID
持有人 ID
初始数量
累计入库
当前库存
最近变化时间
```

同一种药品可以有多个持有人。

---

## 2.3 库存变化必须留流水

系统不允许绕过流水直接修改库存。

以下操作都必须写入 `medicine_stock_logs`：

1. 初始入库。
2. 购入。
3. 使用。
4. 报废。
5. 分配转出。
6. 分配转入。
7. 转交转出。
8. 转交转入。
9. 申请使用。
10. 库存校正。
11. 归档记录。
12. 删除持有库存记录。

---

## 2.4 一名成员对同一种药品只保留一条有效持有库存

V1 建议对同一药品和同一持有人只保留一条有效库存记录。

也就是说：

```text
medicine_id + holder_id + deleted_at IS NULL
```

应保持唯一。

如果别人再次分配同种药品给该成员：

```text
不新增第二条持有库存
而是累加到该成员已有持有库存中
并写入 distribute_in 流水
```

如果转交给已经持有同种药品的成员：

```text
目标成员库存增加
原持有人库存清零或原库存记录标记为 transferred
并写入 transfer_out / transfer_in 流水
```

这样可以避免前端持有人标签重复。

---

## 2.5 历史记录长期保留

药品主档、成员持有库存和库存流水都应优先软删除或归档。

不建议物理删除，因为历史任务、申请和库存变化都需要追溯。

# 4. 通用约定

## 4.1 主键

所有业务表统一使用：

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid()
```

需要启用：

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

## 4.2 时间字段

核心业务表统一包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |
| `deleted_at` | timestamptz | 软删除时间，可为空 |

流水类表只追加不修改，可以不设置 `updated_at`。

## 4.3 数量字段

药品库存数量使用：

```sql
numeric(12, 2)
```

原因：

1. 可以表示整数单位，例如片、粒、支。
2. 也可以兼容后期 ml、g、kg 等小数单位。

V1 前端可以根据单位限制是否允许小数。

# 5. medicine_categories 药品分类表

## 5.1 表名

```text
medicine_categories
```

## 5.2 作用

保存药品分类，用于列表筛选和主档归类。

示例：

```text
抗生素
```

## 5.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 分类 ID |
| `name` | varchar(64) | not null, unique | 分类名称 |
| `code` | varchar(64) | unique, nullable | 分类编码 |
| `description` | text | nullable | 分类说明 |
| `sort_order` | int | default 0 | 排序值 |
| `is_enabled` | boolean | default true | 是否启用 |
| `created_by` | uuid | FK -> users.id, nullable | 创建人 |
| `updated_by` | uuid | FK -> users.id, nullable | 更新人 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

## 5.4 主要索引

```sql
CREATE UNIQUE INDEX uq_medicine_categories_name
ON medicine_categories (name)
WHERE deleted_at IS NULL;

CREATE INDEX idx_medicine_categories_enabled
ON medicine_categories (is_enabled, sort_order);
```

## 5.5 业务规则

1. 分类由管理员维护。
2. 普通成员创建药品时只能选择已有分类。
3. 如果第一版不做分类管理页面，可以先写入默认分类种子数据。
4. 已被药品主档使用的分类不建议物理删除，只能禁用或软删除。

# 6. medicine_catalogs 药品主档表

## 6.1 表名

```text
medicine_catalogs
```

## 6.2 作用

保存药品主档，表示“这是什么药”。

药品主档不保存某个成员手里有多少库存。

## 6.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 药品 ID |
| `name` | varchar(128) | not null | 药品名称 |
| `category_id` | uuid | FK -> medicine_categories.id, nullable | 药品分类 |
| `specification` | varchar(128) | nullable | 药品规格，如 `250mg/片` |
| `unit` | varchar(32) | not null | 库存单位，如 `片`、`粒`、`支`、`瓶` |
| `description` | text | nullable | 药品说明 |
| `usage_notes` | text | nullable | 使用注意事项 |
| `cover_image_url` | varchar(512) | nullable | 药品封面图 |
| `status` | varchar(32) | not null, default 'active' | 主档状态 |
| `created_by` | uuid | FK -> users.id, nullable | 创建人，可能是成员或管理员 |
| `updated_by` | uuid | FK -> users.id, nullable | 更新人 |
| `archived_by` | uuid | FK -> users.id, nullable | 归档操作人 |
| `archived_at` | timestamptz | nullable | 归档时间 |
| `archive_reason` | text | nullable | 归档原因 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

## 6.4 status 枚举

| 值 | 说明 |
|---|---|
| `active` | 正常使用 |
| `archived` | 已归档，不再默认展示 |
| `deleted` | 已软删除 |

## 6.5 unit 建议枚举

V1 可以先使用固定选项：

```text
片
粒
支
瓶
袋
盒
包
ml
g
kg
个
```

## 6.6 主要索引

```sql
CREATE INDEX idx_medicine_catalogs_category
ON medicine_catalogs (category_id);

CREATE INDEX idx_medicine_catalogs_status
ON medicine_catalogs (status);

CREATE INDEX idx_medicine_catalogs_created_by
ON medicine_catalogs (created_by);

CREATE UNIQUE INDEX uq_medicine_catalogs_name_spec_unit
ON medicine_catalogs (lower(name), COALESCE(specification, ''), unit)
WHERE deleted_at IS NULL;
```

如需中文模糊搜索，建议启用 `pg_trgm`：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX idx_medicine_catalogs_name_trgm
ON medicine_catalogs USING gin (name gin_trgm_ops);
```

## 6.7 业务规则

1. 第一次创建某种药品时，由创建人写入药品主档。
2. 药品主档创建后，普通成员不能编辑主档。
3. 管理员可以编辑药品主档。
4. 只有当该药品所有持有库存均为 0，且没有待审核申请时，管理员才允许归档。
5. 删除优先采用软删除；如果有历史流水或任务关联，建议只归档，不删除。

# 9. medicine_holdings 成员持有库存表

## 9.1 表名

```text
medicine_holdings
```

## 9.2 作用

保存某个成员持有某种药品的库存状态。

这是药品模块的“子状态”。

## 9.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 持有库存 ID |
| `medicine_id` | uuid | FK -> medicine_catalogs.id, not null | 药品 ID |
| `holder_id` | uuid | FK -> users.id, not null | 当前持有人 |
| `source_type` | varchar(32) | not null | 来源类型 |
| `source_holding_id` | uuid | FK -> medicine_holdings.id, nullable | 来源库存，分配/转交时使用 |
| `created_by` | uuid | FK -> users.id, nullable | 创建人 |
| `admin_creator_id` | uuid | FK -> users.id, nullable | 管理员代创建人，可为空 |
| `initial_quantity` | numeric(12,2) | not null, default 0 | 初始数量 |
| `total_in_quantity` | numeric(12,2) | not null, default 0 | 累计入库数量 |
| `current_quantity` | numeric(12,2) | not null, default 0 | 当前库存 |
| `unit_snapshot` | varchar(32) | not null | 单位快照，来自药品主档 |
| `status` | varchar(32) | not null, default 'active' | 持有库存状态 |
| `last_operation_at` | timestamptz | nullable | 最近库存变化时间 |
| `remark` | text | nullable | 备注 |
| `deleted_by` | uuid | FK -> users.id, nullable | 删除操作人 |
| `delete_reason` | text | nullable | 删除原因 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

## 9.4 source_type 枚举

| 值 | 说明 |
|---|---|
| `self_created` | 成员自己创建 |
| `admin_assigned` | 管理员为成员创建 |
| `distributed` | 由其他持有人分配产生 |
| `transferred` | 由其他持有人转交产生 |
| `imported` | 后期批量导入 |

## 9.5 status 枚举

| 值 | 说明 |
|---|---|
| `active` | 正常持有，有库存或可继续记录 |
| `empty` | 当前库存为 0 |
| `transferred` | 已转交或合并到其他持有人库存 |
| `deleted` | 管理员软删除 |

## 9.6 约束

```sql
ALTER TABLE medicine_holdings
ADD CONSTRAINT ck_medicine_holdings_quantity_non_negative
CHECK (
    initial_quantity >= 0
    AND total_in_quantity >= 0
    AND current_quantity >= 0
);

ALTER TABLE medicine_holdings
ADD CONSTRAINT ck_medicine_holdings_total_in_gte_current
CHECK (total_in_quantity >= current_quantity);
```

同一成员对同一药品只保留一条有效库存：

```sql
CREATE UNIQUE INDEX uq_medicine_holdings_medicine_holder_active
ON medicine_holdings (medicine_id, holder_id)
WHERE deleted_at IS NULL AND status IN ('active', 'empty');
```

## 9.7 索引

```sql
CREATE INDEX idx_medicine_holdings_medicine
ON medicine_holdings (medicine_id);

CREATE INDEX idx_medicine_holdings_holder
ON medicine_holdings (holder_id);

CREATE INDEX idx_medicine_holdings_status
ON medicine_holdings (status);

CREATE INDEX idx_medicine_holdings_last_operation
ON medicine_holdings (last_operation_at DESC);
```

## 9.8 业务规则

1. 成员创建药品时，`holder_id = 当前登录用户`。
2. 管理员为成员创建时，`holder_id = 被指定成员`，`admin_creator_id = 当前管理员`。
3. 创建库存时自动写入 `initial_in` 流水。
4. `initial_quantity` 只表示创建时数量，产生流水后不建议直接修改。
5. `current_quantity` 只能通过服务层根据流水操作更新。
6. `total_in_quantity` 随初始入库、购入、分配转入、转交转入、校正入库增加。
7. 当前库存为 0 时，状态可自动变为 `empty`。
8. 成员退会时，不建议直接删除非空库存，应先转交或分配给其他成员。

# 10. medicine_stock_logs 药品库存流水表

## 10.1 表名

```text
medicine_stock_logs
```

## 10.2 作用

记录所有库存变化，同时作为药品详情和持有人库存详情中的“药品动态”数据来源。

V1 不单独建立药品动态表，直接使用库存流水生成动态展示。

## 10.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 流水 ID |
| `medicine_id` | uuid | FK -> medicine_catalogs.id, not null | 药品 ID |
| `holding_id` | uuid | FK -> medicine_holdings.id, nullable | 当前流水所属库存 |
| `operator_id` | uuid | FK -> users.id, nullable | 操作人 |
| `operation_type` | varchar(32) | not null | 操作类型 |
| `quantity_delta` | numeric(12,2) | not null, default 0 | 数量变化，增加为正，减少为负 |
| `quantity_before` | numeric(12,2) | not null | 操作前库存 |
| `quantity_after` | numeric(12,2) | not null | 操作后库存 |
| `related_application_id` | uuid | FK -> medicine_use_applications.id, nullable | 关联用药申请 |
| `related_task_id` | uuid | FK -> tasks.id, nullable | 关联任务，V1 预留 |
| `related_task_snapshot` | jsonb | nullable | 任务快照，任务模块未完善时可为空 |
| `target_user_id` | uuid | FK -> users.id, nullable | 分配/转交目标成员 |
| `source_holding_id` | uuid | FK -> medicine_holdings.id, nullable | 来源库存 |
| `target_holding_id` | uuid | FK -> medicine_holdings.id, nullable | 目标库存 |
| `reason_type` | varchar(64) | nullable | 原因类型 |
| `reason_text` | text | nullable | 原因文本 |
| `description` | text | nullable | 详细说明 |
| `remark` | text | nullable | 备注 |
| `evidence_file_ids` | jsonb | nullable | 凭证文件 ID 列表，兼容统一上传模块 |
| `display_title` | varchar(128) | nullable | 动态标题冗余 |
| `display_content` | text | nullable | 动态内容冗余 |
| `operated_at` | timestamptz | not null | 业务发生时间 |
| `created_at` | timestamptz | not null | 创建时间 |

## 10.4 operation_type 枚举

| 值 | 说明 | quantity_delta |
|---|---|---:|
| `initial_in` | 初始入库 | 正数 |
| `purchase` | 购入 | 正数 |
| `use_self` | 持有人自己使用 | 负数 |
| `scrap` | 报废 | 负数 |
| `distribute_out` | 分配转出 | 负数 |
| `distribute_in` | 分配转入 | 正数 |
| `transfer_out` | 转交转出 | 0 或负数，按实现策略 |
| `transfer_in` | 转交转入 | 0 或正数，按实现策略 |
| `application_use` | 申请通过后使用 | 负数 |
| `adjustment` | 库存校正 | 可正可负 |
| `archive` | 药品归档记录 | 0 |
| `delete_holding` | 管理员删除持有库存记录 | 0 |

说明：

1. 分配是数量从一个持有人流向另一个持有人，因此一出一入。
2. 转交如果采用“直接变更 holder_id”，可以记录两条数量变化为 0 的流水。
3. 如果转交给已有同药持有人并合并库存，可以记录 `transfer_out` 和 `transfer_in` 对应数量变化。

## 10.5 reason_type 建议枚举

| 值 | 说明 |
|---|---|
| `task_related` | 关联任务 |
| `free_text` | 自由填写说明 |
| `expired` | 过期报废 |
| `polluted` | 污染报废 |
| `damaged` | 包装破损 |
| `unknown_source` | 无法确认来源 |
| `storage_issue` | 保存不当 |
| `inventory_diff` | 盘点差异 |
| `input_error` | 录入错误 |
| `other` | 其他 |

## 10.6 索引


## 10.7 业务规则

1. 流水只追加，不编辑，不删除。
2. 库存数量变化必须同时更新 `medicine_holdings.current_quantity`。
3. 操作失败时，库存更新和流水写入必须在同一个数据库事务中回滚。
4. 药品详情页动态直接查询该药品下所有流水。
5. 持有人库存详情页动态只查询对应 `holding_id` 的流水。
6. 关联任务 V1 只预留字段，任务搜索与任务动态同步可后置。

## 11.7 业务规则

1. 申请人不能申请自己的持有库存，自己使用走 `use_self` 流水。
2. 申请数量必须大于 0。
3. 提交申请时可以校验 `quantity <= current_quantity`，但通过审核时必须再次校验库存。
4. 申请默认 24 小时失效。
5. 只有持有人可以审核发给自己的申请。
6. 申请通过后，必须扣减对应 `holding_id` 的库存，并写入 `application_use` 流水。
7. 申请拒绝不改变库存。
8. 申请人只能在 `pending` 状态下取消申请。

# 13. 典型写入事务

## 13.1 创建全新药品

```text
1. 创建 medicine_catalogs
2. 创建 medicine_holdings
3. 写 medicine_stock_logs.initial_in
4. 更新 medicine_holdings.last_operation_at
5. 提交事务
```

## 13.2 选择已有药品新增自己库存

```text
1. 查询 medicine_catalogs
2. 检查当前用户是否已有有效 holding
3. 如无 holding，创建 medicine_holdings
4. 写 medicine_stock_logs.initial_in
5. 提交事务
```

## 13.3 购入

```text
1. 校验当前用户是 holder
2. 锁定 medicine_holdings 行
3. current_quantity += quantity
4. total_in_quantity += quantity
5. 写 medicine_stock_logs.purchase
6. 更新 last_operation_at
7. 提交事务
```

## 13.4 使用 / 报废

```text
1. 校验当前用户是 holder
2. 校验 quantity <= current_quantity
3. 锁定 medicine_holdings 行
4. current_quantity -= quantity
5. 写 medicine_stock_logs.use_self 或 scrap
6. 如 current_quantity = 0，holding.status = empty
7. 提交事务
```

## 13.5 分配

```text
1. 校验当前用户是原 holder
2. 校验 quantity <= 原 current_quantity
3. 锁定原 holding
4. 原 holding.current_quantity -= quantity
5. 查找或创建目标成员 holding
6. 目标 holding.current_quantity += quantity
7. 目标 holding.total_in_quantity += quantity
8. 写 distribute_out 流水
9. 写 distribute_in 流水
10. 提交事务
```

## 13.6 转交

V1 可以按以下规则实现：

```text
如果目标成员没有同药 holding：
  直接变更当前 holding.holder_id
  写 transfer_out / transfer_in 流水

如果目标成员已有同药 holding：
  合并到目标 holding
  原 holding 标记 transferred
  写 transfer_out / transfer_in 流水
```

## 13.7 申请通过

```text
1. 校验当前用户是 holder
2. 校验 application.status = pending
3. 校验未过期
4. 锁定 holding
5. 校验 quantity <= current_quantity
6. current_quantity -= quantity
7. application.status = approved
8. 写 medicine_stock_logs.application_use
9. application.stock_log_id = log.id
10. 提交事务
```
