# 校园猫协地图任务系统｜接口设计规范文档

> 版本：V1 接口规范稿  
> 适用范围：地图模块、猫咪库模块、任务模块、个人中心模块、管理员模块、鉴权模块  
> 后端建议：FastAPI + Pydantic  
> 前端建议：uni-app + Vue 3 + TypeScript  

---

## 1. 设计目标

本文档定义系统所有接口的统一风格，避免后续每个模块各写各的。

核心目标：

1. 路径命名统一。
2. 请求参数统一。
3. 响应结构统一。
4. 错误码统一。
5. 分页、筛选、排序统一。
6. 登录鉴权方式统一。
7. 文件上传、时间、经纬度等基础格式统一。
8. 前后端联调时可以直接复用同一套规则。

---

## 2. 基础约定

### 2.1 Base URL

开发环境示例：

```text
http://localhost:8000/api/v1
```

生产环境示例：

```text
https://cat-map.example.com/api/v1
```

所有接口统一加版本前缀：

```text
/api/v1
```

后期如果接口发生不兼容变更，再升级到：

```text
/api/v2
```

---

### 2.2 路径命名

统一使用小写英文和复数资源名。

推荐：

```http
GET /api/v1/tasks
GET /api/v1/tasks/{task_id}
POST /api/v1/tasks/{task_id}/join
```

避免：

```http
GET /api/v1/getTaskList
POST /api/v1/task/joinTask
```

---

### 2.3 字段命名

接口字段统一使用 `snake_case`，与数据库字段和 Pydantic 模型保持一致。

示例：

```json
{
  "student_no": "20252160A1010",
  "must_change_password": true,
  "created_at": "2026-06-19T10:00:00+08:00"
}
```

前端 TypeScript 类型也建议保持 `snake_case`，减少转换成本。

---

### 2.4 Content-Type

普通 JSON 接口：

```http
Content-Type: application/json
```

文件上传接口：

```http
Content-Type: multipart/form-data
```

## 3. HTTP 方法规范

| 方法 | 用途 | 示例 |
|---|---|---|
| GET | 查询资源 | `GET /tasks` |
| POST | 创建资源或执行动作 | `POST /tasks/{task_id}/join` |
| PATCH | 局部更新资源 | `PATCH /auth/password` |
| PUT | 整体替换资源，V1 尽量少用 | `PUT /profiles/me` |
| DELETE | 删除或软删除资源，V1 谨慎使用 | `DELETE /map-points/{point_id}` |

动作型接口可以使用资源子路径：

```http
POST /tasks/{task_id}/join
POST /tasks/{task_id}/abandon
POST /tasks/{task_id}/checkins
POST /auth/logout
```

## 4. 统一响应结构

所有接口统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| code | int | 业务错误码，成功为 0 |
| message | string | 给前端展示或调试的信息 |
| data | any | 实际数据，失败时可以为 null |
| trace_id | string | 请求追踪 ID，方便定位日志 |

成功示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": "b5f7a6d8-2a3e-4f91-98e5-4c22f47a1f31"
  },
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

失败示例：

```json
{
  "code": 40003,
  "message": "验证码错误",
  "data": null,
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 5. HTTP 状态码规范

| HTTP 状态码 | 使用场景 |
|---:|---|
| 200 | 查询成功、更新成功、动作执行成功 |
| 201 | 创建成功 |
| 400 | 请求参数错误、业务参数不合法 |
| 401 | 未登录、token 无效、token 过期 |
| 403 | 已登录但无权限，或必须先改密 |
| 404 | 资源不存在 |
| 409 | 资源冲突，如学号已存在、重复接取任务 |
| 422 | 请求体结构不符合校验规则 |
| 423 | 账号临时锁定 |
| 500 | 服务端异常 |

原则：

1. HTTP 状态码表达大类错误。
2. `code` 表达具体业务错误。
3. 前端主要根据 `code` 做业务提示和跳转。

| 范围 | 模块 |

## 7. 鉴权规范

### 7.1 Header

所有需要登录的接口统一使用：

```http
Authorization: Bearer <access_token>
```

### 7.2 JWT Payload

建议字段：

```json
{
  "sub": "0d65df4f-10c0-45e4-a48d-b04e0af7d3d2",
  "student_no": "20252160A1010",
  "role": "member",
  "token_version": 1,
  "iat": 1781750000,
  "exp": 1781757200
}
```

### 7.3 Token 过期

MVP 建议：

```text
access_token 有效期：2 小时
refresh_token：V1 暂不做
```

Token 过期后，前端跳转登录页重新登录。

### 7.4 首次登录改密限制

当用户 `must_change_password = true` 时，只允许访问：

```http
GET /api/v1/auth/me
PATCH /api/v1/auth/password
POST /api/v1/auth/logout
```

访问其他业务接口时返回：

```json
{
  "code": 40301,
  "message": "请先修改初始密码",
  "data": null,
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 8. 分页规范

### 8.1 请求参数

列表接口统一使用：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| page | int | 1 | 当前页，从 1 开始 |
| page_size | int | 20 | 每页数量 |

示例：

```http
GET /api/v1/tasks?page=1&page_size=20
```

### 8.2 响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [],
    "page": 1,
    "page_size": 20,
    "total": 100,
    "has_more": true
  },
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

### 8.3 限制

建议：

```text
page_size 默认 20
page_size 最大 100
```

## 9. 筛选与排序规范

### 9.1 筛选

筛选参数直接放在 query string。

示例：

```http
GET /api/v1/tasks?status=pending&task_type=emergency&area_id=xxx
```

### 9.2 排序

统一使用：

```text
sort_by
sort_order
```

示例：

```http
GET /api/v1/tasks?sort_by=published_at&sort_order=desc
```

`sort_order` 可选值：

```text
asc
desc
```

### 9.3 搜索

统一使用：

```text
keyword
```

示例：

```http
GET /api/v1/cats?keyword=小橘
```

## 10. 时间格式规范

所有时间统一使用 ISO 8601 / RFC3339 字符串。

示例：

```json
{
  "created_at": "2026-06-19T10:00:00+08:00",
  "deadline_at": "2026-06-19T18:00:00+08:00"
}
```

数据库使用：

```text
timestamptz
```

前端展示时再转成本地时间。

业务表主键统一使用 UUID。

路径参数命名：

```text
{user_id}
{task_id}
{cat_id}
{map_point_id}
```

示例：

```http
GET /api/v1/tasks/{task_id}
```

## 12. 经纬度与地图字段规范

经纬度字段统一使用：

| 字段 | 类型 | 说明 |
|---|---|---|
| lng | number | 经度 |
| lat | number | 纬度 |

示例：

```json
{
  "lng": 114.1234567,
  "lat": 30.1234567
}
```

前端传给后端时不要把经纬度合成字符串。

地图点位相关接口中，如果涉及高德地图信息，可以补充：

| 字段 | 说明 |
|---|---|
| amap_poi_id | 高德 POI ID |
| amap_address | 高德逆地理地址 |
| location_name | 前端展示的位置名称 |
| location_detail | 人工补充的详细位置描述 |
| route_instruction | 校园内部路线说明 |

## 13. 文件上传规范

### 13.1 上传方式

统一使用：

```http
POST /api/v1/files/upload
Content-Type: multipart/form-data
```

### 13.2 请求字段

| 字段 | 类型 | 说明 |
|---|---|---|
| file | file | 文件本体 |
| biz_type | string | 业务类型，如 cat_photo、task_checkin、map_point |

### 13.3 响应示例

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "file_url": "/uploads/2026/06/xxx.jpg",
    "thumbnail_url": "/uploads/2026/06/xxx_thumb.jpg",
    "mime_type": "image/jpeg",
    "size": 204800
  },
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 14. 空值与空数组规范

| 场景 | 返回 |
|---|---|
| 对象不存在 | 404 + `data: null` |
| 列表为空 | 200 + `items: []` |
| 可选字符串没有值 | `null` |
| 可选数组没有值 | `[]` |

## 15. 管理员接口规范

管理员接口统一放在：

```text
/api/v1/admin
```

示例：

```http
POST /api/v1/admin/users
POST /api/v1/admin/tasks
PATCH /api/v1/admin/users/{user_id}/status
```

管理员接口必须校验：

```text
当前用户已登录
users.role = admin 或 super_admin
users.status = active
must_change_password = false
```

重要操作建议写入：

```text
admin_operation_logs
```

## 16. OpenAPI 文档规范

FastAPI 会自动生成 OpenAPI 文档，但需要在代码中写清楚：

1. `summary`：接口一句话说明。
2. `description`：必要的业务规则。
3. Request Schema：请求体模型。
4. Response Schema：响应体模型。
5. Error Response：常见错误码。
6. Tags：模块分组。

Tags 建议：

```text
Auth
Map
Cats
Tasks
Profile
Admin Users
Admin Tasks
Files
Notifications
```

## 18. MVP 总结

V1 接口应优先保证：

1. 命名稳定。
2. 响应结构统一。
3. 错误码可预期。
4. 前端容易写类型。
5. 后端容易生成 OpenAPI。
6. 后期模块扩展时不破坏现有规范。

本规范作为后续地图、猫咪库、任务、个人中心、管理员模块接口文档的统一基础。
