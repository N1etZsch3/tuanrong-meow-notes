# 校园猫协地图任务系统｜暑假喂食任务模块接口文档

> 版本：Summer Feeding V1  
> Base URL：`/api/v1`  
> 模块范围：任务模块收敛版，仅支持「暑假喂食任务」  
> 字段命名：统一使用 `snake_case`  
> 响应结构：统一使用 `{ code, message, data, trace_id }`  
> 文件上传：统一走 `/api/v1/files/upload`，任务接口只保存上传后的文件引用  

---

# 1. 文档定位

本文档定义「暑假喂食任务模块」的 HTTP API。

本版本任务模块只做一种任务：

```text
暑假喂食任务 / summer feeding task
```

目标是先支撑暑假真实使用：

1. 管理员发布固定喂食点任务。
2. 管理员在日历中选择具体执行日期。
3. 管理员可从「我的」页管理员入口发布，也可从地图长按位置进入发布。
4. 成员查看任务列表和详情。
5. 成员导航到任务点。
6. 成员完成某一天投喂。
7. 任务详情页展示任务图片轮播、任务信息和任务动态时间线。

本版本不实现接取、放弃、满员、排班指派、审核和库存扣减接口。

# 2. 接口边界

## 2.1 本模块负责

| 能力 | 说明 |
|---|---|
| 暑假喂食任务列表 | 成员端和管理员端任务列表 |
| 暑假喂食任务详情 | 图片、名称、日期、地图点、说明、物资、动态 |
| 执行日期查询 | 查询某个任务的所有执行日期状态 |
| 完成投喂 | 成员完成某个执行日期 |
| 任务动态查询 | 查询详情页时间线 |
| 管理员发布任务 | 创建任务、地图点、执行日期、任务图片、动态 |
| 管理员编辑任务 | 修改基础信息、点位、执行日期、图片 |
| 管理员状态管理 | 取消、结束、归档、恢复任务 |
| 管理员执行日期管理 | 取消、跳过、重开、撤销某日完成 |

## 2.2 本模块不负责

| 能力 | 处理方式 |
|---|---|
| 原始图片上传 | 调用统一上传模块 `/api/v1/files/upload` |
| 地图初始化、地图点搜索、导航信息 | 复用地图模块接口 |
| 成员登录、角色判断 | 复用鉴权模块 |
| 个人中心统计 | 后续从 `task_checkins` 聚合 |
| 任务接取 / 放弃 / 满员 | 暑假版不做 |
| 管理员审核完成 | 暑假版不做，完成后直接生效 |
| 排班与指派 | 暑假版不做页面闭环 |

# 3. 通用规范

## 3.1 鉴权

除特别说明外，本文档所有接口都需要登录：

```http
Authorization: Bearer <access_token>
```

管理员接口要求当前用户：

```text
users.role in ('admin', 'super_admin')
users.status = 'active'
```

当用户 `must_change_password = true` 时，访问任务接口返回：

```json
{
  "code": 40301,
  "message": "请先修改初始密码",
  "data": null,
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

当用户 `profile_completed = false` 时，访问任务接口建议返回：

```json
{
  "code": 63006,
  "message": "请先完成个人资料初始化",
  "data": null,
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 3.2 统一响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 3.3 分页结构

列表接口统一使用：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| `page` | int | 1 | 当前页，从 1 开始 |
| `page_size` | int | 20 | 每页数量，最大 100 |

分页响应统一为：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0,
  "has_more": false
}
```

## 3.4 时间与日期格式

| 类型 | 格式 | 示例 |
|---|---|---|
| 日期 | `YYYY-MM-DD` | `2026-07-02` |
| 时间 | ISO 8601 / RFC3339 | `2026-07-02T18:24:00+08:00` |

## 3.5 经纬度格式

```json
{
  "lng": 114.1234567,
  "lat": 30.1234567
}
```

经纬度不合并成字符串。

## 3.6 文件上传关系

任务图片和完成照片不直接通过任务接口上传原始文件。

标准流程：

```text
前端选择图片
  ↓
调用 POST /api/v1/files/upload
  ↓
上传模块压缩、生成缩略图、上传 COS
  ↓
返回 file_id / file_url / thumbnail_url
  ↓
任务发布、编辑或完成接口保存文件引用
```

上传接口建议：

```http
POST /api/v1/files/upload
Content-Type: multipart/form-data
```

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `file` | file | 是 | 文件本体 |
| `biz_type` | string | 是 | 建议值：`task_photo`、`task_checkin` |

上传响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "file_id": "b5f7a6d8-2a3e-4f91-98e5-4c22f47a1f31",
    "file_url": "https://cos.example.com/tasks/cover.jpg",
    "thumbnail_url": "https://cos.example.com/tasks/cover_thumb.jpg",
    "mime_type": "image/jpeg",
    "size": 204800
  },
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

如果当前上传模块暂未返回 `file_id`，任务接口允许 `file_id = null`，但必须提供 `file_url`。

# 4. 错误码

任务模块错误码范围：`62000 - 62999`。

| code | HTTP | 说明 |
|---:|---:|---|
| `62001` | 400 | 任务参数错误 |
| `62002` | 400 | 执行日期不能为空 |
| `62003` | 400 | 执行日期不合法 |
| `62004` | 400 | 只支持暑假喂食任务 |
| `62005` | 400 | 任务状态不支持 |
| `62006` | 409 | 执行日期重复 |
| `62007` | 400 | 今天不是该任务执行日 |
| `62008` | 409 | 该执行日期已完成 |
| `62009` | 404 | 执行日期不存在 |
| `62010` | 404 | 任务不存在 |
| `62011` | 400 | 任务图片参数不合法 |
| `62012` | 400 | 地图点参数不合法 |
| `62013` | 409 | 当前任务状态不允许修改 |
| `62014` | 409 | 已完成的执行日期不能直接删除 |
| `62015` | 409 | 已取消的任务不能完成投喂 |
| `62016` | 403 | 无权完成该任务 |
| `62017` | 400 | 完成记录参数不合法 |

通用错误码继续使用统一规范：

| code | HTTP | 说明 |
|---:|---:|---|
| `40001` | 400 | 参数错误 |
| `40002` | 400 | 缺少必要参数 |
| `40101` | 401 | 未登录 |
| `40102` | 401 | Token 无效或已过期 |
| `40301` | 403 | 请先修改初始密码 |
| `40302` | 403 | 权限不足 |
| `40303` | 403 | 账号已被禁用 |
| `40401` | 404 | 资源不存在 |
| `40901` | 409 | 资源已存在 |
| `50001` | 500 | 服务端异常 |

# 5. 枚举定义

## 5.1 task_type

暑假版只使用：

| 值 | 说明 |
|---|---|
| `feeding` | 喂食任务 |

后期完整任务模块可继续扩展：

```text
daily / emergency / patrol / supply / rescue / medical / capture / observation
```

## 5.2 task_mode

| 值 | 说明 | 暑假版 |
|---|---|---:|
| `single` | 一次性任务 | 不使用 |
| `recurring` | 多日期 / 长期任务 | 使用 |

## 5.3 schedule_type

| 值 | 说明 | 暑假版 |
|---|---|---:|
| `once` | 一次性日期 | 不使用 |
| `selected_dates` | 日历手动选择多个日期 | 使用 |
| `weekly` | 每周周期生成 | 后期扩展 |

## 5.4 completion_policy

| 值 | 说明 | 暑假版 |
|---|---|---:|
| `task` | 整个任务一次性完成 | 不使用 |
| `per_execution_date` | 按执行日期分别完成 | 使用 |

## 5.5 task.status

| 值 | 说明 | 暑假版 |
|---|---|---:|
| `pending` | 草稿 / 未发布 | 可选 |
| `in_progress` | 进行中 | 使用 |
| `completed` | 整个暑假周期结束 | 使用 |
| `cancelled` | 整个任务取消 | 使用 |
| `pending_review` | 待审核 | 不使用 |
| `archived` | 历史归档 | 可选 |

## 5.6 execution_date.status

| 值 | 说明 |
|---|---|
| `pending` | 待完成 |
| `completed` | 已完成 |
| `cancelled` | 当日取消 |
| `skipped` | 当日跳过，不计异常 |
| `missed` | 过期未完成 |

## 5.7 activity_type

| 值 | 说明 |
|---|---|
| `created` | 发布任务 |
| `updated` | 修改任务 |
| `checkin_submitted` | 提交完成记录 |
| `execution_completed` | 某个执行日期已完成 |
| `execution_cancelled` | 某个执行日期取消 |
| `execution_skipped` | 某个执行日期跳过 |
| `execution_reopened` | 某个执行日期重新打开 |
| `cancelled` | 整个任务取消 |
| `completed` | 整个任务结束 |
| `archived` | 任务归档 |

## 5.8 photo_type

| 值 | 说明 |
|---|---|
| `cover` | 封面图 |
| `scene` | 现场图 |
| `route` | 路线参考图 |
| `reference` | 参考图 |

# 8. 成员端接口设计

---

## 8.1 获取暑假喂食任务列表

### 基本信息

```http
GET /api/v1/tasks
```

| 项目 | 内容 |
|---|---|
| 权限 | 登录用户 |
| 说明 | 成员端任务列表，本版本只返回暑假喂食任务 |
| 涉及表 | `tasks`、`map_points`、`task_execution_dates`、`task_photos` |

### Query 参数

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---:|---|---|
| `task_type` | string | 否 | `feeding` | 暑假版固定为 `feeding`，传其他值返回 `62004` |
| `status` | string | 否 | `in_progress` | 多个用逗号分隔，如 `in_progress,completed` |
| `area_id` | uuid | 否 | null | 校园区域 ID |
| `keyword` | string | 否 | null | 搜索任务名称、位置名称 |
| `execute_date` | date | 否 | null | 查询某个执行日相关任务 |
| `only_today` | boolean | 否 | false | 是否只看今天需要投喂的任务 |
| `execution_status` | string | 否 | null | `pending`、`completed`、`cancelled`、`skipped`、`missed` |
| `user_lng` | number | 否 | null | 用户当前位置经度，用于计算距离 |
| `user_lat` | number | 否 | null | 用户当前位置纬度，用于计算距离 |

```text
1. 校验用户已登录且资料初始化完成
2. 查询 tasks，确认任务存在且 task_type = feeding
3. 校验 tasks.status = in_progress
4. 查询 task_execution_dates，确认执行日期属于该任务
5. 如果执行日期 status = completed，返回 62008
6. 如果执行日期 status in cancelled / skipped，返回 62013
7. 创建 task_checkins
8. 如有照片，写入 task_checkin_photos
9. 更新 task_execution_dates.status = completed

12. 返回完成记录和更新后的执行日期
```

### 业务规则

1. 完成某一天投喂不会把父任务 `tasks.status` 改成 `completed`。
2. 本版本不需要管理员审核，完成后立即生效。
3. 默认只允许完成当前日期对应的执行任务。
4. 如果需要补录，可由后端配置允许完成最近未完成日期；接口结构不变。
5. 同一个执行日期只能完成一次。

| code | HTTP | 说明 |

| `62007` | 400 | 今天不是该任务执行日 |
| `62008` | 409 | 该执行日期已完成 |
| `62009` | 404 | 执行日期不存在 |
| `62010` | 404 | 任务不存在 |
| `62013` | 409 | 当前任务状态不允许修改 |
| `62015` | 409 | 已取消的任务不能完成投喂 |
| `62017` | 400 | 完成记录参数不合法 |

## 9.4 修改任务整体状态

### 基本信息

```http
PATCH /api/v1/admin/tasks/{task_id}/status
```

| 项目 | 内容 |
|---|---|
| 权限 | 管理员 |
| 说明 | 取消、结束、归档、恢复整个任务 |
| 涉及表 | `tasks`、`task_activity_logs`、`admin_operation_logs` |

### 请求体

```json
{
  "status": "cancelled",
  "reason": "暑假安排调整，暂停该点投喂"
}
```

### 请求字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `status` | string | 是 | `in_progress`、`completed`、`cancelled`、`archived` |
| `reason` | string | 否 | 原因说明 |

### 响应示例

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "task_id": "6bf9e8aa-9ce1-4eb2-a1c3-ef3e086b4c59",
    "status": "cancelled",
    "updated_at": "2026-07-01T12:30:00+08:00"
  },
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

### 业务规则

1. `cancelled` 表示整个任务取消，成员不能再完成投喂。
2. `completed` 表示整个暑假周期结束，不代表某一天完成。
3. `archived` 用于历史归档，可以从成员端默认列表中隐藏。
4. 从 `cancelled` 恢复到 `in_progress` 仅管理员可操作。
5. 每次状态变化写入 `task_activity_logs` 和 `admin_operation_logs`。

# 10. 地图模块联动

## 10.1 地图长按发布入口

地图长按是前端交互，不需要单独后端接口。

前端流程：

```text
地图页长按
  ↓
获取长按坐标 lng / lat
  ↓
弹出“发布暑假喂食任务”按钮
  ↓
进入发布页并带入 lng / lat
  ↓
调用 POST /api/v1/admin/tasks/summer-feeding
```

## 10.2 地图点 Marker 展示

发布暑假喂食任务时，后端创建或复用 `map_points`。

建议写入：

```text
map_points.point_type = task
map_points.point_scope = long_term
map_points.icon_key = task_feeding 或 task_summer_feeding
map_points.visibility = public
map_points.status = active
```

地图模块 `GET /api/v1/map/points` 聚合返回 Marker 时：

```json
{
  "point_type": "task",
  "business_type": "feeding",
  "business_id": "task_id",
  "marker_key": "task_feeding",
  "extra": {
    "task_status": "in_progress",
    "next_execute_date": "2026-07-09",
    "today_status": "pending"
  }
}
```

## 10.3 导航

任务详情页「导航前往」按钮调用地图模块：

```http
GET /api/v1/map/points/{map_point_id}/navigation
```

任务详情接口只返回 `map_point` 基础信息，不重复实现导航路线生成。

# 12. 后端落地建议

## 12.1 最小必做接口

为了暑假先可用，第一阶段只需要实现：

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/api/v1/tasks` | 任务列表 |
| `GET` | `/api/v1/tasks/{task_id}` | 任务详情 |
| `POST` | `/api/v1/tasks/{task_id}/checkins` | 完成投喂 |
| `GET` | `/api/v1/admin/tasks` | 管理员任务列表 |
| `POST` | `/api/v1/admin/tasks/summer-feeding` | 发布任务 |
| `PATCH` | `/api/v1/admin/tasks/{task_id}` | 编辑任务 |
| `PATCH` | `/api/v1/admin/tasks/{task_id}/status` | 取消 / 归档任务 |

可以后置：

```text
GET /tasks/{task_id}/execution-dates
GET /tasks/{task_id}/activities
GET /tasks/{task_id}/checkins
PATCH /admin/tasks/{task_id}/execution-dates/{execution_date_id}
任务图片单独增删改接口
```

因为任务详情接口已经可以聚合返回执行日期和动态。

## 12.2 事务边界

以下接口必须使用数据库事务：

1. `POST /admin/tasks/summer-feeding`
2. `PATCH /admin/tasks/{task_id}`
3. `POST /tasks/{task_id}/checkins`
4. `PATCH /admin/tasks/{task_id}/execution-dates/{execution_date_id}`

尤其是完成投喂接口，必须保证：

```text
task_checkins
task_execution_dates
task_activity_logs
```

三者要么全部成功，要么全部回滚。

## 12.3 并发控制

完成投喂接口需要防止两个人同时完成同一天。

建议方式：

```text
在事务中 SELECT task_execution_dates FOR UPDATE
```

然后再次检查：

```text
status != completed
```

如果已完成，返回：

```json
{
  "code": 62008,
  "message": "该执行日期已完成",
  "data": null,
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 12.4 与完整任务模块的兼容

本接口文档保留 `/tasks` 资源路径，而不是新建 `/summer-feeding-tasks`，原因是后期完整任务模块可以自然扩展：

```http
GET /api/v1/tasks?task_type=feeding
GET /api/v1/tasks?task_type=emergency
POST /api/v1/admin/tasks
```

暑假版新增的 `POST /api/v1/admin/tasks/summer-feeding` 是为了让发布流程更明确，避免与后期通用任务发布接口混在一起。
