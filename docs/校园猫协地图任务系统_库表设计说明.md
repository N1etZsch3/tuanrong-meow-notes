# 校园猫协地图任务系统库表设计说明

> 版本：MVP 数据库设计稿（鉴权模块微调版）  
> 项目：校园猫协地图任务系统  
> 数据库：PostgreSQL + PostGIS  
> 后端建议：FastAPI + SQLAlchemy + Alembic  
> 设计目标：支撑地图、猫咪库、任务、个人中心、管理员五大模块的 MVP 功能，并为后续通知、审核、排班、指派、健康追踪、物资管理、自定义地图扩展预留空间。

> 本次微调：围绕登录页与内部成员认证流程，补充 `users.token_version` 字段；将初始账号默认要求改密作为 V1 业务默认；明确不开放注册，成员账号由管理员创建或批量导入；修改密码、管理员重置密码后通过 `token_version` 让旧 JWT 失效。

## 0. 设计总览

### 0.1 五大模块

本项目主要拆分为五个业务模块：

1. 地图模块
2. 猫咪库模块
3. 任务模块
4. 个人中心模块
5. 管理员模块

同时存在几个跨模块能力：

1. 通知系统
2. 图片文件能力
3. 操作日志
4. 任务动态
5. 角色权限
6. 成员认证与验证码
7. 物资点管理
8. 排班与任务指派

---

## 0.2 核心设计原则

### 原则 1：地图模块只负责空间位置

`map_points` 只保存：

- 经纬度
- 点位名称
- 位置说明
- 路线说明
- 点位类型
- 显隐状态
- 地图展示信息

不在 `map_points` 里保存完整任务详情、猫咪健康信息、物资库存信息。

业务数据分别放在：

- `tasks`
- `cats`
- `supply_points`

---

### 原则 2：任务状态和参与状态分开

任务整体状态放在：

```text
tasks.status
```

用户参与状态放在：

```text
task_participants.status
```

不要用一个字段同时表达“任务进行中”和“某个人已放弃”。

---

### 原则 3：“已满员”不作为任务状态

任务状态只表示流程：

```text
pending / in_progress / pending_review / completed / cancelled
```

任务是否满员通过下面条件动态计算：

```text
participant_count >= max_participants
```

---

### 原则 4：个人中心不重复存“我的任务”

个人中心里的：

- 我的任务
- 我的打卡
- 我的观察记录
- 我的统计

都从已有业务表反查或统计得到。

不要额外建一张 `my_tasks` 表。

---

### 原则 5：管理员模块是管理层，不是独立业务孤岛

管理员发布任务时，会同时写入：

```text
map_points
tasks
task_activity_logs
notifications
admin_operation_logs
```

管理员添加猫咪时，会同时写入：

```text
cats
cat_aliases
cat_photos
cat_map_points
map_points
admin_operation_logs
```

管理员添加物资点时，会同时写入：

```text
map_points
supply_points
supply_point_items
admin_operation_logs
```

---

### 原则 6：第一版包含排班与任务指派

本项目 V1 需要支持管理员根据成员排班筛选可指派人员，并主动指派成员完成任务。

因此下面两张表属于 MVP 必做表：

```text
duty_schedules
task_assignments
```

任务可以有两种参与来源：

```text
自由接取：成员主动点击接取任务
管理员指派：管理员根据排班和任务类型指派成员
```

如果成员接受指派，系统应同步写入：

```text
task_participants
```

也就是说：

```text
task_assignments 记录“管理员指派关系”
task_participants 记录“实际参与关系”
```
---

### 原则 7：第一版不做管理员完成审核

V1 中，成员提交完成打卡后，任务可以直接进入已完成状态，不需要管理员审核通过。

因此：

```text
task_review_records
```

属于后期扩展表，不纳入 V1 必做表。

V1 的任务完成流程为：

```text
成员提交完成打卡
        ↓
写入 task_checkins
        ↓
任务直接 completed
        ↓
生成任务动态和通知
```

可以保留 `tasks.status = pending_review` 和 `task_checkins.review_status` 作为后期扩展枚举，但 V1 默认不使用审核流程。


---

### 原则 8：第一版认证只支持学号登录

V1 登录入口只保留：

```text
学号 + 密码 + 字母验证码
```

因此认证相关设计采用以下规则：

1. `users.student_no` 是登录账号主标识。
2. `users.password_hash` 保存密码哈希，不保存明文密码。
3. `user_profiles` 只保存展示资料，不再重复保存学号。
4. 不开放成员自主注册，成员账号由管理员创建或批量导入。
5. 手机号、邮箱、第三方登录暂不进入 V1。
6. 字母验证码由 `auth_captchas` 保存短期校验记录；后期可以迁移到 Redis。


---

### 原则 9：默认密码必须改密，密码变更后旧 Token 失效

第一版成员账号由管理员创建或批量导入，因此通常会存在统一初始密码或管理员设置的默认密码。

为避免初始密码长期有效，V1 采用以下规则：

1. 管理员创建成员账号时，默认 `users.must_change_password = true`。
2. 成员首次登录后，只允许访问当前用户信息、修改密码和退出登录接口。
3. 成员修改密码成功后，`users.must_change_password = false`。
4. 成员修改密码或管理员重置密码后，`users.token_version` 自增。
5. JWT 中携带 `token_version`，后端鉴权时与数据库当前值比对。
6. 如果 JWT 中的 `token_version` 与数据库不一致，则视为旧 Token，拒绝访问。

这样可以在不引入复杂 Token 黑名单表的情况下，让改密和重置密码立即失效旧登录态。

## 0.3 主键、时间字段、软删除约定

### 主键建议

所有业务表推荐使用：

```sql
id UUID PRIMARY KEY
```

也可以使用：

```sql
id BIGSERIAL PRIMARY KEY
```

如果你后端使用 SQLAlchemy，推荐统一使用 UUID，方便分布式扩展和前后端传输。

---

### 通用时间字段

大部分业务表建议包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| created_at | timestamptz | 创建时间 |
| updated_at | timestamptz | 更新时间 |
| deleted_at | timestamptz | 软删除时间，可为空 |

---

### 软删除约定

核心业务数据不建议物理删除。

例如：

- 点位隐藏：`map_points.visibility = 'hidden'`
- 点位归档：`map_points.status = 'archived'`
- 猫咪档案归档：`cats.status = 'archived'`
- 任务取消：`tasks.status = 'cancelled'`

---

## 0.4 推荐扩展类型

PostgreSQL 需要启用：

```sql
CREATE EXTENSION IF NOT EXISTS postgis;
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

如果使用 UUID：

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid()
```

空间点字段建议使用：

```sql
geography(Point, 4326)
```

空间区域字段建议使用：

```sql
geography(Polygon, 4326)
```

## 1.1 users 用户账号表

### 表名

```text
users
```

### 作用

保存成员登录账号、密码哈希、角色、账号状态。

V1 中，`student_no` 是唯一登录账号。系统不开放自主注册，成员账号由管理员创建或批量导入。

### 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | uuid | PK | 用户 ID |
| student_no | varchar(64) | unique, not null | 学号，V1 登录账号 |
| password_hash | varchar(255) | not null | 密码哈希，不保存明文密码 |
| role | varchar(32) | not null, default 'member' | MVP 用户角色 |
| status | varchar(32) | not null, default 'active' | 用户状态 |
| must_change_password | boolean | default true | 是否必须修改初始密码；管理员创建账号时默认要求首次登录改密 |
| password_updated_at | timestamptz | nullable | 密码最近更新时间 |
| last_login_at | timestamptz | nullable | 最近登录时间 |
| login_failed_count | int | default 0 | 连续登录失败次数 |
| locked_until | timestamptz | nullable | 登录锁定截止时间 |
| token_version | int | default 1 | Token 版本号，修改密码或管理员重置密码后自增，用于让旧 JWT 失效 |
| created_at | timestamptz | not null | 创建时间 |
| updated_at | timestamptz | not null | 更新时间 |
| deleted_at | timestamptz | nullable | 软删除时间 |

### 后期扩展字段

以下字段暂不纳入 V1 登录方式，可以后期根据需要再加入：

| 字段名 | 说明 |
|---|---|
| username | 用户名登录 |
| phone | 手机号登录 |
| email | 邮箱登录 |
| openid | 小程序或第三方登录 ID |

### role 枚举

| 值 | 说明 |
|---|---|
| member | 普通成员 |
| admin | 管理员 |
| super_admin | 超级管理员，后期 |

MVP 可以只使用：

```text
member
admin
```

### status 枚举

| 值 | 说明 |
|---|---|
| active | 正常 |
| inactive | 未激活 |
| blocked | 禁用 |
| left | 已退出猫协 |
| deleted | 已删除 |

### token_version 说明

`token_version` 用于无状态 JWT 的失效控制：

```text
用户修改密码 / 管理员重置密码
        ↓
users.token_version + 1
        ↓
旧 JWT 中的 token_version 与数据库不一致
        ↓
后端拒绝访问，要求重新登录
```

MVP 不单独设计 `auth_sessions` 或 `refresh_tokens` 表，先通过 `access_token + token_version` 完成最小可用登录态管理。

### 主要索引

```sql
CREATE UNIQUE INDEX uq_users_student_no
ON users (student_no);

CREATE INDEX idx_users_role
ON users (role);

CREATE INDEX idx_users_status
ON users (status);

CREATE INDEX idx_users_locked_until
ON users (locked_until)
WHERE locked_until IS NOT NULL;
```

| client_ip | varchar(64) | nullable | 请求 IP |
| user_agent | text | nullable | 请求设备信息 |
| created_at | timestamptz | not null | 创建时间 |

### scene 枚举

| 值 | 说明 |
|---|---|
| login | 登录验证码 |

### 主要索引


# 9. 关键业务流程与表写入关系


## 9.0 管理员创建成员账号

涉及表：

```text
users
user_profiles
admin_operation_logs，可选
```

流程：

```text
1. 管理员填写学号、初始密码和成员资料
2. 校验 users.student_no 是否已存在
3. 对初始密码进行哈希处理，写入 users.password_hash
4. 写入 users，默认 role = member，status = active
5. 写入 users.must_change_password = true，强制首次登录后修改默认密码
6. 写入 users.token_version = 1
7. 写入 user_profiles
8. 写入 admin_operation_logs
```

---

## 9.0.1 成员登录

涉及表：

```text
users
auth_captchas
user_profiles
```

流程：

```text
1. 校验协议勾选项
2. 校验 auth_captchas 是否存在、未过期、未使用
3. 校验验证码
4. 根据 users.student_no 查询用户
5. 判断 users.status 是否 active
6. 判断 users.locked_until 是否仍在锁定期
7. 校验 password_hash
8. 登录失败则增加 login_failed_count，必要时写入 locked_until
9. 登录成功则更新 users.last_login_at，并清空 login_failed_count、locked_until
10. 标记验证码 used_at
11. 签发 JWT，并在 payload 中写入 users.token_version
12. 返回用户基础信息、资料、角色和 must_change_password
13. 如果 must_change_password = true，前端跳转修改密码页，后端限制访问业务接口
```

---

## 9.1 管理员发布任务

涉及表：

```text
map_points
tasks
task_activity_logs
notifications
admin_operation_logs
```

流程：

```text
1. 创建 map_points
   point_type = task
   point_scope = temporary

2. 创建 tasks
   map_point_id = map_points.id
   publisher_id = 当前管理员 ID

3. 写入 task_activity_logs
   activity_type = created

4. 如果是紧急任务，写入 notifications 给所有成员

5. 写入 admin_operation_logs
   operation_type = publish
   target_type = task

   注：如果 V1 暂不做管理员操作日志，则此步骤可后置。
```

---


## 9.2 管理员根据排班与任务指派任务

涉及表：

```text
duty_schedules
task_assignments
tasks
task_activity_logs
notifications
admin_operation_logs
```

流程：

```text
1. 管理员选择任务
2. 系统根据任务时间、任务类型、区域筛选 duty_schedules
3. 只展示 status = available 且时间匹配的成员
4. 管理员选择成员并创建 task_assignments
5. task_assignments.status = pending
6. 给被指派成员写入 notifications
7. 写入 task_activity_logs: assigned
8. 写入 admin_operation_logs: assign
```

成员接受指派后：

```text
1. task_assignments.status = accepted
2. responded_at = 当前时间
3. 写入 task_participants
4. tasks.participant_count + 1
5. 如果任务原状态是 pending，改为 in_progress
6. duty_schedules.assigned_task_count + 1
7. 写入 task_activity_logs: assignment_accepted
8. 通知任务发布者和其他参与人
```

成员拒绝指派后：

```text
1. task_assignments.status = rejected
2. 写入 reject_reason
3. responded_at = 当前时间
4. 通知任务发布者
5. 写入 task_activity_logs: assignment_rejected
```

---

## 9.3 成员接取任务

涉及表：

```text
tasks
task_participants
task_activity_logs
notifications
```

流程：

```text
1. 事务内锁定 tasks 行
2. 判断任务状态是否允许接取
3. 判断 participant_count < max_participants
4. 写入 task_participants
5. tasks.participant_count + 1
6. 如果原状态是 pending，改为 in_progress
7. 写入 task_activity_logs: joined
8. 如果达到满员，写入 task_activity_logs: full
9. 触发通知
```

---

## 9.4 成员放弃任务

涉及表：

```text
tasks
task_participants
task_activity_logs
notifications
```

流程：

```text
1. task_participants.status = abandoned
2. 写入 abandoned_at 和 abandon_reason
3. tasks.participant_count - 1
4. 如果参与人数变成 0，可将 tasks.status 改回 pending
5. 写入 task_activity_logs: abandoned
6. 触发通知
```

---

## 9.5 成员完成任务打卡

涉及表：

```text
task_checkins
task_checkin_photos
task_participants
tasks
task_activity_logs
notifications
cat_observation_records
```

流程：

```text
1. 写入 task_checkins
2. 写入 task_checkin_photos
3. task_participants.status = completed
4. task_checkins.review_status = no_review
5. tasks.status = completed
6. 写入 task_activity_logs: checkin_submitted
7. 如果关联猫咪，生成 cat_observation_records

```

# 12. 整体自检

## 12.1 是否存在重复建表？

### 检查结果：无严重重复，但有几个需要明确边界的表。

| 表 | 可能重复点 | 最终处理 |
|---|---|---|
| `map_points` 和 `supply_points` | 都涉及物资点 | `map_points` 存位置，`supply_points` 存物资点业务 |
| `cat_observation_records` 和 `cat_health_records` | 都可能记录健康异常 | 观察记录存“某次看到什么”，健康记录存“健康事件时间线” |
| `task_activity_logs` 和 `notifications` | 都记录任务变化 | 动态属于任务详情，通知属于用户提醒 |
| `task_review_records` 和 `admin_operation_logs` | 都记录管理员审核 | 审核表存业务审核结果，操作日志存管理员操作留痕 |
| `users.role` 和 `roles/user_roles` | 都表达角色 | MVP 用 `users.role`，后期 RBAC 再启用 roles |
| `users.student_no` 和 `user_profiles.student_no` | 都可能保存学号 | V1 将学号只保存在 `users.student_no`，`user_profiles` 不再重复保存 |
| `tasks.related_cat_id` 和 `task_cats` | 都表达任务关联猫咪 | MVP 用 `related_cat_id`，多猫任务后期启用 `task_cats` |
| `tasks.required_items` 和 `task_required_supplies` | 都表达任务物资需求 | MVP 用文本字段，后期结构化再启用表 |

---

## 12.2 是否存在状态枚举冲突？

### 检查结果：无明显冲突。

注意区分以下状态：

### 任务状态：`tasks.status`

只表示任务整体流程：

```text
pending
in_progress
pending_review
completed
cancelled
```

### 参与状态：`task_participants.status`

只表示某个用户和任务的关系：

```text
joined
abandoned
completed
removed
```

### 打卡审核状态：`task_checkins.review_status`

只表示某条打卡记录的审核状态：

```text
pending
approved
rejected
no_review
```

### 点位状态：`map_points.status`

只表示地图点位是否有效：

```text
active
inactive
archived
deleted
```

### 点位可见性：`map_points.visibility`

只表示谁能看：

```text
public
admin_only
hidden
```

这几个字段不要混用。

## 12.3 是否存在字段冗余？

### 有少量有意冗余，属于合理冗余。

| 字段 | 冗余原因 |
|---|---|
| `tasks.area_id` | 可由 `map_points.area_id` 得到，但任务列表按区域筛选频繁，保留冗余 |
| `tasks.route_instruction` | 可由 `map_points.route_instruction` 得到，但任务可能有特殊路线说明，保留 |
| `tasks.participant_count` | 可由 `task_participants` 统计，但任务列表频繁展示，保留缓存 |
| `cats.primary_map_point_id` | 可由 `cat_map_points.is_primary` 得到，但猫咪列表常用，保留 |
| `cats.last_seen_at` | 可由观察记录统计得到，但猫咪列表常用，保留 |
| `task_checkin_photos.task_id` | 可由 `checkin_id` 关联得到，但按任务查照片更方便，保留 |

这些冗余字段需要在业务逻辑中维护一致性。

## 12.7 V1 排班与指派自检

### 检查结果：已调整为第一版必做。

# 12.10 本次鉴权微调自检

# 13. 最终结论

本库表设计可以支撑第一版 MVP，并且为后续扩展保留空间。

第一版优先落地：

```text
成员学号登录
验证码校验
地图点位
猫咪档案
任务接取
任务打卡
任务直接完成
任务动态
通知中心
个人中心
管理员发布和维护
排班与任务指派
```

后续再扩展：

```text
复杂权限
管理员完成审核
物资消耗
健康时间线高级分析
自定义校园地图
实时 WebSocket
```

排班与任务指派已经调整为第一版范围，不再作为后置功能。

最终核心关系可以概括为：

```text
map_points 负责“在哪里”
cats 负责“是哪只猫”
tasks 负责“要做什么”
task_participants 负责“谁参与”
task_checkins 负责“做完后反馈什么”
cat_observation_records 负责“猫咪长期发生了什么”
users 负责“是谁、用什么学号登录”
auth_captchas 负责“登录验证码校验”
admin_operation_logs 负责“管理员做过什么”
notifications 负责“谁需要被提醒”
```
