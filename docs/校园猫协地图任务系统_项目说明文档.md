# 校园猫协地图任务系统项目说明文档

## 1. 项目名称

项目名称暂定为：

- 校园猫协地图任务系统
- 猫协地图
- 校园猫咪任务地图
- 校园流浪猫协作地图

本文档统一使用 **校园猫协地图任务系统** 作为项目名称。

---

## 2. 项目背景

学校猫协成员在执行日常喂食、巡查、救助、送医、物资转移等任务时，经常会遇到位置描述不清楚、新成员不熟悉校园、任务信息分散在群聊中、猫咪资料难以沉淀等问题。

目前猫协任务通常依赖：

- 群聊通知
- 文字描述
- 人工带路
- 口头说明
- 零散照片

这些方式存在几个明显问题：

1. 新成员很难准确找到任务点。
2. 草丛、楼后、小树林、台阶、宿舍区夹角等非标准地点难以通过普通地图精确表达。
3. 猫咪常驻点、物资点、任务点分散在不同人的记忆中。
4. 任务接取、放弃、完成、审核缺少统一记录。
5. 猫咪观察记录和健康变化难以长期沉淀。
6. 管理员发布任务时缺少直观的地图标点工具。

因此，本项目希望做一个专门服务于本校猫协的移动端地图任务系统。

系统不是全国通用地图产品，而是一个面向校园猫协内部协作的工具。

---

## 3. 项目一句话描述

这是一个专门为校园猫协设计的移动端地图任务系统，用校园地图统一展示任务点、猫咪点和物资点，支持管理员在地图上标点发布任务、添加猫咪和物资点，成员可以查看位置、导航、接取多人任务、放弃任务、完成拍照打卡，并沉淀猫咪档案、观察记录、任务记录和个人出勤历史。

---

## 4. 项目核心目标

系统最终要解决的问题包括：

1. 让猫协成员快速找到任务地点。
2. 让常驻猫咪、物资点、日常任务、紧急任务集中展示在地图上。
3. 让管理员可以通过地图选点发布任务、添加猫咪点和物资点。
4. 让任务发布、接取、满员、放弃、完成打卡形成闭环，管理员审核作为后期扩展能力预留。
5. 让多人任务协作更清晰，所有成员都能看到谁接取了任务。
6. 让猫咪信息、观察记录、健康记录、相关任务记录沉淀下来。
7. 让成员可以查看自己的任务历史、打卡记录和参与记录。
8. 为后续通知、任务动态、校园内部路径、猫咪健康追踪等功能预留扩展空间。

---

## 5. 项目边界

### 5.1 项目服务范围

系统只服务于本校校园范围，不做全国或城市级通用地图产品。

地图服务主要面向：

- 校园内部
- 校园周边与猫协任务有关的局部区域
- 猫咪常驻区域
- 物资存放点
- 日常任务点
- 紧急救助任务点

### 5.2 第一版不追求的能力

第一版暂时不追求：

- 完全自研地图引擎
- 复杂 Web 管理后台
- 高并发架构
- 全国地图服务
- 精确到室内级别的导航
- 完整实时 WebSocket 系统
- 复杂数据大屏
- 复杂权限审批流
- 成员自主注册
- 手机号、邮箱、第三方登录
- 找回密码等复杂账号体系
- AI 自动识别猫咪身份
- 自动路线规划算法

第一版重点是完成一个可用、可演示、可迭代的移动端 MVP。

## 6. 技术选型

### 6.1 总体技术方案

推荐技术栈：

```text
前端：uni-app + Vue 3 + TypeScript + Pinia
后端：FastAPI + Pydantic + SQLAlchemy + Alembic
数据库：PostgreSQL + PostGIS
地图：高德地图
文件存储：本地存储起步，后期可切换 OSS / COS / MinIO
通知：第一版使用通知表 + 前端轮询
认证：学号 + 密码 + 字母验证码 + JWT
缓存：第一版不强制使用 Redis，后期根据实时推送、限流、缓存需求再接入
部署：Docker Compose
```

---

### 6.2 前端技术选型

前端推荐：

```text
uni-app
Vue 3
TypeScript
Pinia
uView Plus / 自写轻量组件
高德地图 SDK
```

选择原因：

1. 项目以移动端为主。
2. uni-app 可以较快开发 H5、小程序、App 等形态。
3. Vue 3 组件结构清晰，适合快速迭代。
4. TypeScript 可以让数据结构更清楚，方便后续维护。
5. Pinia 用于管理用户信息、任务状态、地图筛选条件等全局状态。
6. 前端页面数量不算多，适合使用轻量组件库或自写 UI。

---

### 6.3 后端技术选型

后端推荐：

```text
FastAPI
Pydantic
SQLAlchemy 2.0
Alembic
JWT 鉴权
密码哈希
字母验证码
```

选择原因：

1. 业务逻辑清晰可控。
2. 接口开发速度快。
3. Python 生态适合后续扩展 AI、图像处理、自动化任务等能力。
4. FastAPI 自动生成接口文档，便于前后端联调。
5. Pydantic 适合定义清晰的数据结构。
6. SQLAlchemy + Alembic 适合管理数据库模型和迁移。
7. JWT 适合移动端登录态维护。
8. 密码哈希与验证码机制可以满足第一版成员登录安全需求。

---

### 6.4 数据库技术选型

数据库推荐：

```text
PostgreSQL + PostGIS
```

选择原因：

1. 项目天然包含地图点位、经纬度、区域范围、附近任务等空间数据。
2. PostGIS 适合处理地理空间查询。
3. 后续可以支持附近点位、区域筛选、距离排序、校园边界判断等能力。
4. 比普通关系型数据库更适合地图任务系统。

---

### 6.5 地图技术选型

地图推荐：

```text
高德地图
```

第一版由高德地图负责：

1. 底图展示。
2. 地图缩放与拖动。
3. Marker 点位展示。
4. 用户当前位置定位。
5. 步行导航。
6. 管理员地图点击选点。
7. 限制地图显示区域。
8. 后续可叠加学校平面图或自定义图层。

项目自己的系统负责：

1. 猫咪点位数据。
2. 物资点位数据。
3. 任务点位数据。
4. 路线文字说明。
5. 现场照片。
6. 任务和猫咪业务逻辑。

---

### 6.6 缓存与 Redis 选型

第一版不强制使用 Redis。

原因：

1. 项目早期数据量有限。
2. 校园范围较小。
3. 点位数量不会特别大。
4. PostgreSQL 正常索引足以支撑第一版访问。
5. Redis 会增加缓存一致性、部署和维护成本。

可以预留缓存层，但不在第一版强制接入。

Redis 适合后期用于：

1. WebSocket 多实例广播。
2. Redis Pub/Sub。
3. 限流。
4. 短期验证码。
5. 热点地图点位缓存。
6. 在线用户状态。
7. 高并发通知推送。

---

### 6.7 通知技术方案

第一版推荐：

```text
通知表 + 前端轮询
```

也就是：

1. 后端把通知写入数据库。
2. 前端定时请求未读通知数。
3. 用户进入通知中心后查看历史通知。
4. 任务详情页定时刷新参与人和任务状态。

后期可以升级为：

```text
FastAPI WebSocket + PostgreSQL 通知表
```

再后期如需多实例部署，可以扩展为：

```text
FastAPI WebSocket + Redis Pub/Sub + PostgreSQL 通知表
```

原则：

- 数据库通知表负责可靠记录。
- 轮询或 WebSocket 负责提醒。
- Redis Pub/Sub 只负责实时广播，不作为通知存储。

---


### 6.8 用户认证技术方案

第一版认证方案收窄为：

```text
学号 + 密码 + 字母验证码 + JWT
```

认证模块只服务于猫协内部成员登录，不面向公开用户开放注册。

登录页面字段包括：

1. 学号。
2. 密码。
3. 字母验证码。
4. 协议勾选项。

账号来源：

1. 管理员提前创建成员账号。
2. 管理员可以设置初始密码。
3. 成员首次登录后可以修改密码。
4. 管理员可以重置成员密码或禁用账号。

第一版暂不支持：

1. 自主注册。
2. 手机号登录。
3. 邮箱登录。
4. 微信、QQ 等第三方登录。
5. 找回密码。

验证码第一版可以使用数据库表保存短期验证码记录，后期如果接入 Redis，可以把验证码迁移到 Redis 中。

系统主要有两类角色：

1. 普通成员 / 志愿者
2. 猫协管理员

后续可以扩展为更多角色，例如：

```text
学号 + 密码 + 字母验证码
```

登录页面对应字段：

1. 学号。
2. 密码。
3. 字母验证码。
4. 用户协议、隐私政策、校园猫协成员规范勾选项。

1. 系统不允许成员自主注册。
2. 管理员负责提前创建成员账号。
3. 学号是成员登录的唯一账号标识。
4. 密码由管理员设置初始密码，成员登录后可修改。
5. 管理员可以重置密码、禁用账号、恢复账号。
6. 普通成员登录后根据角色进入普通功能；管理员登录后额外显示管理员入口。

第一版暂不做手机号、邮箱、第三方账号登录。

# 10. 地图模块

## 10.1 模块定位

地图模块是系统首页，也是整个项目的核心页面。

它负责把猫协内部的空间信息集中展示出来，让成员可以直观看到校园中有哪些猫咪点、物资点和任务点。

---

## 10.2 地图展示内容

地图上主要展示三类点位：

1. 任务点位
2. 猫咪点位
3. 物资点位

任务点位可以继续区分：

- 日常任务
- 紧急任务
- 救助任务
- 喂食任务
- 巡查任务
- 物资任务

猫咪点位可以表示：

- 常驻猫咪点
- 高频出现区域
- 临时观察点

物资点位可以表示：

- 猫粮存放点
- 诱捕笼存放点
- 航空箱存放点
- 药品存放点
- 其他协会物资点

---

## 10.3 地图筛选功能

地图页需要支持筛选：

```text
全部
任务
猫咪
物资
紧急任务
日常任务
```

后期可以扩展为：

```text
只看待接取任务
只看进行中任务
只看附近任务
只看某个区域
只看健康异常猫咪
只看物资点
```

## 10.5 地图导航功能

导航功能第一版使用高德地图能力。

用户可以：

1. 查看自己的当前位置。
2. 点击点位。
3. 点击导航。
4. 导航到任务点、猫咪点或物资点附近。

对于校园内部非标准地点，需要配合：

1. 文字路线说明。
2. 现场照片。
3. 附近地标。
4. 入口说明。
5. 任务发布人备注。

示例：

```text
导航终点：体育馆西门
路线说明：到达体育馆西门后，沿右侧小路向后走约 50 米，看到草坪后往左侧树丛方向走，小橘常在长椅附近。
```

## 10.6 地图区域限制

系统只服务于本校校园范围，因此地图应默认定位到校园，并限制或弱化校外区域。

第一版可以实现：

1. 默认中心点为校园中心。
2. 默认缩放级别适合校园。
3. 限制地图可拖动范围。
4. 只加载校园范围内的业务点位。
5. 点位按照校园区域分类。

后期可以支持：

1. 叠加学校平面图。
2. 校园区域边界。
3. 自定义校园矢量图。
4. 校园内部路径网络。

# 11. 猫咪库模块

## 11.1 模块定位

猫咪库是校园猫咪信息数据库。

它不只是猫咪图鉴，而是猫协长期维护猫咪档案、健康状态、观察记录、任务记录的核心模块。

---

## 11.2 猫咪列表

猫咪列表中每只猫应展示：

1. 猫咪照片。
2. 猫咪名字。
3. 别名。
4. 花色。
5. 常驻区域。
6. 健康状态。
7. 绝育状态。
8. 最近出现时间。
9. 性格标签。

示例：

```text
小橘
常驻区域：体育馆后草坪
健康状态：健康
绝育状态：已绝育
最近出现：今天
性格：亲人、爱睡觉
```

---

## 11.3 猫咪搜索与筛选

猫咪库应支持：

1. 按名字搜索。
2. 按别名搜索。
3. 按花色筛选。
4. 按区域筛选。
5. 按健康状态筛选。
6. 按绝育状态筛选。
7. 按亲人程度筛选。
8. 按最近出现时间筛选。

---

## 11.4 猫咪详情

猫咪详情页应包含：

1. 猫咪名称。
2. 别名。
3. 照片。
4. 花色。
5. 性别。
6. 绝育状态。
7. 健康状态。
8. 亲人程度。
9. 性格标签。
10. 常驻区域。
11. 常驻点位。
12. 最近出现时间。
13. 性格描述。
14. 喂食注意事项。

5. 巡查任务。

2. 紧急任务。
3. 喂食任务。

5. 物资任务。

## 12.3 任务状态

任务状态建议包括：

1. 待接取。
2. 进行中。
3. 已完成。
4. 已取消。

后期如果恢复管理员审核流程，可以预留：

1. 待审核。

“已满员”不一定作为数据库状态存储，可以根据参与人数动态计算。

例如：

```text
当前参与人数 >= 最大参与人数
```

则前端显示为：

```text
人数已满
```

## 12.4 任务列表

任务列表应支持：

1. 查看全部任务。
2. 查看待接取任务。
3. 查看进行中任务。
4. 查看已完成任务。
5. 查看紧急任务。
6. 查看日常任务。
7. 按距离排序。
8. 按发布时间排序。
9. 按优先级排序。
10. 按区域筛选。
11. 按任务类型筛选。

任务卡片应展示：

1. 任务名称。
2. 任务类型。
3. 任务状态。
4. 优先级。
5. 地点。
6. 距离。
7. 发布时间。
8. 简短说明。
9. 当前参与人数。
10. 最大参与人数。
11. 操作按钮。

```text
北门草丛紧急救助任务
类型：紧急任务
状态：待接取
地点：北门草丛
距离：80m
参与人数：1/3
说明：发现一只疑似受伤猫咪，需要尽快观察并处理

按钮：立即接取 / 查看详情
```

16. 任务动态。
17. 完成记录。
18. 导航按钮。
19. 接取按钮。
20. 放弃按钮。
21. 完成打卡按钮。

## 12.6 多人任务接取机制

猫协任务通常需要多人协作，例如三个人一起出任务。

因此任务应支持最大参与人数，例如：

```text
最大参与人数：3
```

成员接取任务后，所有人都可以看到当前参与人。

示例：

```text
参与成员 2/3

[头像] 金炜然
[头像] 小林
[空位] 还差 1 人

金炜然、小林接了任务
```

满员后显示：

```text
参与成员 3/3

[头像] 金炜然
[头像] 小林
[头像] 阿杰

任务人数已满
```

按钮逻辑：

1. 未接取且未满员：显示「接取任务」。
2. 已接取：显示「放弃任务」「完成打卡」。
3. 未接取但已满员：显示「人数已满」。
4. 已完成：显示「查看完成记录」。
5. 已取消：显示「任务已取消」。

---

## 12.7 放弃任务

成员接取任务后，可以在任务未完成前放弃任务。

放弃任务后：

1. 释放任务名额。
2. 参与人列表移除或显示为已放弃。
3. 任务动态记录某成员放弃任务。
4. 通知任务发布者和已参与成员。
5. 保留放弃记录，便于后续查看。

---

## 12.8 完成任务与打卡

成员完成任务时，不应只点击一个“完成”按钮，而应提交任务反馈。

完成任务时可以填写：

1. 是否完成。
2. 是否见到猫。
3. 猫咪状态。
4. 现场照片。
5. 处理结果。
6. 备注说明。
7. 打卡位置。
8. 完成时间。

紧急任务建议强制要求：

1. 上传照片。
2. 填写处理结果。
3. 填写备注说明。

普通任务可以降低要求。

---

## 12.9 任务审核

第一版不做管理员完成审核。

V1 的完成流程为：

```text
成员提交完成反馈
        ↓
系统写入完成打卡记录
        ↓
任务直接进入已完成
        ↓
任务动态记录完成事件
        ↓
相关记录沉淀到个人历史和猫咪档案
```

后期如果需要提高任务质量控制，可以恢复审核流程：

```text
成员提交完成反馈
        ↓
任务进入待审核
        ↓
管理员查看照片、备注、处理结果
        ↓
管理员确认完成或驳回
```

---

## 12.10 任务动态

任务详情页应展示任务动态。

任务动态记录这个任务发生过什么。

示例：

```text
20:01 管理员发布了任务
20:03 金炜然接取了任务
20:05 小林接取了任务
20:08 阿杰接取了任务，任务已满员
20:30 金炜然提交了完成打卡
20:40 系统记录任务已完成
```

任务动态和通知不同：

- 任务动态属于任务详情的一部分。
- 通知属于发给具体用户的提醒。

# 15. 通知系统

## 15.1 模块定位

通知系统用于提醒成员和管理员关注任务变化。

它解决的问题是：

1. 新任务发布后成员能看到。
2. 有人接取任务后相关人员能知道。
3. 任务满员后相关人员能知道。
4. 有人放弃任务后相关人员能知道。
5. 任务完成后任务发布者和相关成员能知道。
6. 后期如果启用审核流程，成员可以收到审核结果。

---

## 15.2 第一版通知方案

第一版采用：

```text
通知表 + 前端轮询
```

前端轮询策略：

1. 地图页每 10-20 秒刷新任务点位。
2. 任务详情页每 5-10 秒刷新参与人和状态。
3. 通知未读数每 30-60 秒刷新。
4. 任务列表页每 10-30 秒刷新。

---

## 15.3 通知类型

通知类型包括：

1. 新任务发布。
2. 紧急任务发布。
3. 有人接取任务。
4. 任务满员。
5. 有人放弃任务。
6. 成员提交完成记录。
7. 任务完成。
8. 管理员取消任务。
9. 猫咪状态异常。
10. 物资点更新。

---

## 15.4 通知对象

不同事件通知不同人。

### 新紧急任务发布

通知对象：

```text
所有成员
```

### 有人接取任务

通知对象：

```text
任务发布者
已接取该任务的成员
```

### 任务满员

通知对象：

```text
任务发布者
已接取该任务的成员
```

### 有人放弃任务

通知对象：

```text
任务发布者
已接取该任务的成员
```

### 成员提交完成记录

通知对象：

```text
管理员
任务发布者
```

### 任务完成

通知对象：

```text
任务发布者
参与该任务的成员
```

### 管理员审核通过，后期扩展

通知对象：

```text
参与该任务的成员
```

## 15.5 通知中心

个人中心可以提供通知入口。

通知中心展示：

1. 未读通知。

3. 通知时间。

```

        ↓
任务达到人数上限后显示满员
        ↓
成员到达现场处理
        ↓
成员提交完成打卡
        ↓
任务直接进入已完成
        ↓
记录进入成员历史任务
        ↓
相关记录沉淀到猫咪档案
```

# 24. 最终项目定位

校园猫协地图任务系统的核心定位是：

```text
一个面向校园猫协的移动端地图协作工具。
```

它要解决的不是单纯展示猫咪，也不是普通任务列表，而是把：

```text
校园地图
猫咪档案
任务协作
多人接取
导航
物资点
观察记录
通知
个人历史
管理员维护
```

整合到一个系统中。

第一版的目标应当是：

```text
让成员能用学号登录系统，找得到任务点，知道谁接了任务，能接取、放弃、完成并打卡；让管理员能创建成员账号、在地图上发布任务、添加猫咪和物资点；让猫咪信息和任务记录开始沉淀。
```
