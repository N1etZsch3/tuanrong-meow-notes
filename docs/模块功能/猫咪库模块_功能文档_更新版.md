# 校园猫协地图任务系统｜猫咪库模块功能文档

> 版本：V1.1 管理员录入流程调整版  
> 模块名称：猫咪库模块  
> 适用阶段：MVP / 第一版  
> 前端建议：uni-app + Vue 3 + TypeScript + Pinia  
> 后端建议：FastAPI + Pydantic + SQLAlchemy  
> 本次调整重点：删除独立“亲人程度”字段，统一使用“性格标签”；新增管理员四页式录入流程；V1 引入前端本地草稿暂存机制；猫咪基础档案支持最多 5 张照片、猫咪故事、常驻区域文字描述。

---

# 1. 模块定位

猫咪库是校园猫咪档案中心，用于沉淀猫协长期维护的猫咪资料、照片、活动区域、健康状态、观察记录和相关任务记录。

它在底部主菜单中作为一级页面存在，面向普通成员提供浏览、搜索、筛选、查看详情、添加观察记录等能力；面向管理员提供猫咪档案创建、编辑、点位绑定、照片维护、健康与注意事项维护等能力。

猫咪库的核心目标：

1. 让成员快速识别校园猫咪。
2. 让成员知道猫咪常出现在哪个区域。
3. 让成员了解猫咪当前档案状态、绝育状态和健康状态。
4. 让成员可以补充观察记录，帮助长期追踪猫咪变化。
5. 让管理员可以高效建立猫咪档案，并逐步补充地图点位和健康注意事项。

# 2. 模块边界

## 2.1 猫咪库负责

| 能力 | 说明 |
|---|---|
| 猫咪档案 | 名称、花色、性别、绝育状态、档案状态、性格标签、猫咪故事 |
| 猫咪照片 | 主头像、档案照片、近照，每只猫最多 5 张有效照片 |
| 常驻区域 | V1 使用文字描述，如“学生宿舍区东侧草坪” |
| 地图点位绑定 | 可选；管理员可为猫咪新建或绑定地图猫咪点 |
| 健康状态 | 健康、需观察、异常、受伤、疑似生病、治疗中、已恢复、未知 |
| 注意事项 | 喂食注意、抓捕注意、医疗注意、备注 |
| 观察记录 | 成员记录看到猫咪时的时间、位置、状态、照片和备注 |
| 健康记录 | 管理员记录健康事件时间线 |
| 相关任务 | 展示与猫咪关联的喂食、观察、救助、送医等任务 |
| 收藏猫咪 | 个人中心可展示用户收藏的猫咪 |
| 管理员录入 | 分步录入、预览确认、草稿暂存 |

| 地图底图、地图拖拽、地图定位 | 地图模块 |
| 任务接取、放弃、完成打卡 | 任务模块 |
| 图片压缩、COS 上传、缩略图生成 | 文件上传模块 |
| 用户登录、角色鉴权 | 鉴权模块 |
| 我的任务、我的收藏猫咪入口 | 个人中心模块 |
| 物资库存管理 | 物资模块 / 管理员模块 |

# 3. 用户角色与权限

## 3.1 普通成员

普通成员可以：

1. 查看猫咪库首页统计。
2. 搜索猫咪名称、别名、花色、区域。
3. 按筛选项、可选值和排序方式筛选猫咪。
4. 查看猫咪列表卡片。
5. 查看猫咪详情。
6. 查看猫咪照片墙。
7. 查看猫咪活动点位。
8. 查看观察记录。
9. 添加观察记录。
10. 查看健康记录。
11. 查看相关任务。
12. 收藏或取消收藏猫咪。

## 3.2 管理员

管理员拥有普通成员能力，并额外可以：

1. 新增猫咪档案。
2. 编辑猫咪基础资料。
3. 维护猫咪照片。
4. 维护猫咪别名。
5. 维护猫咪常驻区域文字描述。
6. 新建或绑定猫咪地图点位。
7. 维护猫咪健康状态。
8. 维护喂食、抓捕、医疗注意事项。
9. 管理观察记录。
10. 添加健康记录。
11. 归档、恢复、更新猫咪档案状态。

# 5. 猫咪库列表页

## 5.1 页面目标

列表页用于快速浏览校园猫咪档案。它应优先回答：

1. 这只猫是谁？
2. 长什么样？
3. 常在哪？
4. 最近什么时候出现？
5. 当前是否需要关注？

## 5.2 页面布局

推荐结构：

```text
顶部标题区
统计概览卡片
搜索框
筛选列表区
猫咪卡片列表
底部主菜单
```

## 5.3 统计概览卡片

列表页顶部展示猫咪库整体状态。示例指标：

| 指标 | 说明 |
|---|---|
| 在档猫咪 | 当前系统中未删除的猫咪档案总数 |
| 正常在校 | 档案状态为正常在校的猫咪数量 |
| 待领养 | 档案状态为待领养的猫咪数量 |
| 待观察 | 当前健康状态为需观察的猫咪数量 |
| 已绝育 | 绝育状态为已绝育的猫咪数量 |
| 绝育率 | 已绝育数量 / 在档猫咪数量 |

统计卡片用于替代标题下方简单的“共多少只”，让用户进入页面时先看到猫咪库整体情况。

## 5.4 搜索能力

搜索框支持：

```text
猫咪名称
别名
花色
常驻区域文字描述
地图点位名称
```

示例关键词：

```text
小橘
橘子
狸花
宿舍区
北门
图书馆
```

## 5.5 筛选列表区

V1 不再铺开大量快捷筛选按钮，改为三段式筛选：

```text
筛选项     可选值     排序
```

示例：

```text
筛选项：健康状态
可选值：待观察
排序：最近出现
```

筛选区旁边保留：

```text
清除筛选
```

点击后恢复默认列表。

## 5.6 筛选项与可选值

| 筛选项 | 可选值 |
|---|---|
| 区域 | 全部区域、宿舍区、教学楼、体育馆、食堂、北门、图书馆、自定义区域 |
| 花色 | 橘猫、狸花、奶牛、黑猫、三花、白猫、玳瑁、自定义 |
| 健康状态 | 健康、需观察、异常、受伤、疑似生病、治疗中、已恢复、未知 |
| 绝育状态 | 未知、未绝育、已绝育、已预约、暂不适合 |
| 档案状态 | 正常在校、暂时失踪、待领养、已领养、已死亡、转移照护、归档 |
| 性格标签 | 亲人、怕人、胆小、警惕、温顺、活泼、贪吃、会哈气、自定义标签 |
| 最近出现时间 | 今天、三天内、一周内、一周前、未知 |

## 5.7 排序规则

| 排序 | 说明 |
|---|---|
| 最近出现 | `last_seen_at DESC NULLS LAST` |
| 最近更新 | `updated_at DESC` |
| 名称排序 | `name ASC` |
| 健康优先 | 异常、受伤、疑似生病、需观察优先 |
| 未绝育优先 | 未绝育、已预约、未知优先 |

## 5.8 猫咪卡片

猫咪列表卡片展示摘要信息，不展示完整字段说明。

卡片建议展示：

| 内容 | 说明 |
|---|---|
| 主头像 | `cats.avatar_url` 或主照片缩略图 |
| 名称 | `cats.name` |
| 花色 / 别名摘要 | 如“橘猫 · 别名橘子” |
| 常驻区域 | `resident_area_text` 优先 |
| 最近出现 | `last_seen_at` 格式化展示 |
| 状态标签 | 健康状态、绝育状态、档案状态、性格标签中最重要的 2-3 个 |
| 右箭头 | 点击进入详情 |

普通状态使用低饱和标签；异常、受伤、待观察、暂时失踪、未绝育等需要更明显的颜色。

# 6. 猫咪详情页

## 6.1 页面目标

详情页用于查看一只猫的完整档案，包括照片、故事、位置、健康与协作注意事项、观察记录、健康记录和相关任务。

## 6.2 页面结构

推荐结构：

```text
顶部照片轮播
猫咪名称与状态标签
基础档案
猫咪故事
位置与地图点位
健康与注意事项
观察记录
健康记录
相关任务
管理员操作入口
```

## 6.3 基础档案

展示字段：

```text
名称
花色
性别
绝育状态
档案状态
性格标签
常驻区域
最近出现时间
```

## 6.4 猫咪故事

`story` 用于展示 300 字以内的猫咪介绍，例如：

```text
小橘常在学生宿舍区东侧草坪活动，喜欢晒太阳，比较亲人，熟悉后可以轻轻靠近。
```

猫咪故事偏展示性，不承担医疗或任务说明。

## 6.5 位置区

位置区分两层：

1. 常驻区域文字描述：始终可展示。
2. 精确地图点位：管理员绑定后展示地图入口和导航入口。

如果没有绑定地图点位，成员端展示：

```text
暂无精确点位
常驻区域：学生宿舍区东侧草坪
```

管理员端额外提示：

```text
该猫咪暂未绑定地图点位，可在管理端补充。
```

## 6.6 健康与注意事项

展示字段：

```text
健康状态
喂食注意
抓捕注意
医疗注意
备注
```

第 3 步跳过时，默认健康状态为 `unknown`，注意事项为空。

# 7. 管理员录入猫咪流程

管理员新增猫咪采用：

```text
3 个录入步骤 + 1 个预览确认页
```

V1 引入草稿暂存机制，优先由前端本地状态和本地存储实现，不强制新增数据库草稿表。

---

## 7.1 第 1 步：基础信息

### 页面目标

完成第 1 步后，猫咪已经具备进入列表页和详情页的基础展示条件。

### 字段

| 字段 | 是否必填 | 规则 |
|---|---:|---|
| 猫咪图片 | 是 | 至少 1 张，最多 5 张 |
| 主头像 | 是 | 从已上传图片中选择 1 张，用于列表页展示 |
| 名称 | 是 | 1-20 字，建议不强制唯一 |
| 花色 | 是 | 预设值 + 自定义 |
| 性别 | 是 | 公、母、未知；默认未知 |
| 绝育状态 | 是 | 未知、未绝育、已绝育、已预约、暂不适合 |
| 档案状态 | 是 | 正常在校、暂时失踪、待领养、已领养、已死亡、转移照护、归档 |
| 性格标签 | 是 | 至少 1 个；每个 1-4 字；支持自定义 |
| 常驻区域 | 是 | 文字描述，最多 128 字 |
| 猫咪故事 | 否 | 300 字内 |

### 性格标签规则

1. 删除独立“亲人程度”字段。
2. 亲人、怕人、警惕、有攻击性等统一作为性格标签。
3. 每个标签最少 1 个字，最多 4 个字。
4. 同一只猫不能重复添加同名标签。
5. V1 建议每只猫最多 8 个标签。
6. 列表页最多展示 2-3 个关键标签。

---

## 7.2 第 2 步：详细猫咪点位，可跳过

### 页面目标

将猫咪与地图模块关联，管理员可以在地图上新建一个猫咪点位。

### 字段

| 字段 | 是否必填 | 说明 |
|---|---:|---|
| 是否绑定地图点位 | 否 | 可以跳过 |
| 地图选点 | 否 | 选择经纬度 |
| 点位名称 | 否 | 默认可生成“{猫咪名称}常驻点” |
| 位置名称 | 否 | 如“宿舍区东侧草坪” |
| 详细位置说明 | 否 | 更具体的位置描述 |
| 路线说明 | 否 | 到附近后如何找到 |
| 附近地标 | 否 | 如“三号宿舍楼后门” |
| 入口说明 | 否 | 如“从北门进入最近” |
| 点位照片 | 否 | 现场图、路线图、入口图 |

### 跳过后的表现

如果跳过第 2 步：

1. `cats.primary_map_point_id = null`。
2. 不创建 `map_points`。
3. 不创建 `cat_map_points`。
4. 管理员端详情显示“暂未绑定地图点位”。
5. 成员端仍展示第 1 步填写的常驻区域文字描述。

---

## 7.3 第 3 步：健康与注意事项，可跳过

### 页面目标

记录照护协作时需要注意的信息，服务后续喂食、救助、抓捕、送医等任务。

### 字段

| 字段 | 是否必填 | 说明 |
|---|---:|---|
| 健康状态 | 否 | 默认未知 |
| 喂食注意 | 否 | 如“不吃猫条”“容易抢食” |
| 抓捕注意 | 否 | 如“怕笼子”“靠近会跑” |
| 医疗注意 | 否 | 如“左眼曾发炎”“已绝育有耳标” |
| 备注 | 否 | 其他补充 |

### 跳过后的表现

如果跳过第 3 步：

1. `health_status = unknown`。
2. 注意事项字段为空。
3. 后续可在猫咪详情页补充。

---

## 7.4 第 4 步：预览与确认保存

### 页面目标

集中预览前三步内容，减少管理员误填。

### 页面结构

```text
基础信息预览
地图点位预览
健康与注意事项预览
确认保存按钮
```

### 保存逻辑

点击确认保存后：

1. 创建 `cats` 主档案。
2. 写入 `cat_photos`。
3. 设置主头像。
4. 如果填写点位，则创建 `map_points` 与 `cat_map_points`。
5. 如果填写健康与注意事项，则写入 `cats.health_status`、`feeding_notes`、`capture_notes`、`medical_notes`、`remark`。
6. 写入管理员操作日志。
7. 清空前端本地草稿。
8. 跳转猫咪详情页或猫咪管理列表页。

推荐保存成功后进入猫咪详情页，方便管理员立即检查展示效果。

# 8. 草稿暂存机制

## 8.1 设计定位

V1 支持管理员在录入过程中暂存表单。草稿机制优先由前端实现，不强制新增数据库表。

目的：

1. 防止管理员中途退出导致内容丢失。
2. 支持分步页面之间返回修改。
3. 支持上传图片后继续补充文本信息。
4. 降低后端草稿状态管理复杂度。

## 8.2 前端草稿保存内容

建议本地草稿结构：

```json
{
  "version": 1,
  "current_step": 2,
  "updated_at": "2026-06-27T12:00:00+08:00",
  "basic_info": {
    "name": "小橘",
    "coat_color": "橘猫",
    "sex": "unknown",
    "neuter_status": "neutered",
    "status": "active",
    "personality_tags": ["亲人", "贪吃"],
    "resident_area_text": "学生宿舍区东侧草坪",
    "story": "常在宿舍区晒太阳。"
  },
  "photos": [
    {
      "file_url": "https://cos.example.com/cat/x.jpg",
      "thumbnail_url": "https://cos.example.com/cat/x_thumb.jpg",
      "is_avatar": true,
      "sort_order": 0
    }
  ],
  "map_point": null,
  "health_info": {
    "health_status": "unknown",
    "feeding_notes": null,
    "capture_notes": null,
    "medical_notes": null,
    "remark": null
  }
}
```

## 8.3 触发时机

建议在以下时机自动保存草稿：

1. 表单字段变更后防抖保存。
2. 点击下一步时保存。
3. 点击上一步时保存。
4. 页面离开前保存。
5. 图片上传完成后保存。

## 8.4 草稿恢复

管理员再次进入新增猫咪页面时，如果存在未提交草稿，弹出选择：

```text
检测到未完成的猫咪档案
继续编辑 / 丢弃草稿
```

## 8.5 草稿清理

以下场景清理草稿：

1. 第 4 步确认保存成功。
2. 管理员手动丢弃草稿。
3. 草稿超过保留时间，例如 7 天。
4. 前端版本升级导致草稿结构不兼容。

## 8.6 图片临时状态

图片通常先通过统一文件上传模块上传，再写入本地草稿。

如果最终没有确认保存，可能产生未绑定业务对象的图片。V1 可以采用以下处理：

1. 文件模块记录临时文件状态，定时清理。
2. 或先允许少量孤立文件存在，管理员确认保存时再绑定到 `cat_photos`。
3. 前端丢弃草稿时可调用文件删除接口，若文件模块已提供。

```text
观察时间
观察地点
是否见到猫
现场照片

是否进食
```

# 11. 状态与标签规则

## 11.1 性别

| 值 | 说明 |
|---|---|
| unknown | 未知 |
| male | 公 |
| female | 母 |

## 11.2 绝育状态

| 值 | 说明 |
|---|---|
| unknown | 未知 |
| not_neutered | 未绝育 |
| neutered | 已绝育 |
| scheduled | 已预约 |
| not_suitable | 暂不适合 |

## 11.3 档案状态

| 值 | 说明 |
|---|---|
| active | 正常在校 |
| missing | 暂时失踪 |
| waiting_adoption | 待领养 |
| adopted | 已领养 |
| deceased | 已死亡 |
| transferred | 转移照护 |
| archived | 归档 |

## 11.4 健康状态

| 值 | 说明 |
|---|---|
| unknown | 未知 |
| healthy | 健康 |
| watching | 需观察 |
| abnormal | 异常 |
| injured | 受伤 |
| sick | 疑似生病 |
| in_treatment | 治疗中 |
| recovered | 已恢复 |

## 11.5 性格标签

性格标签用于替代原“亲人程度”。

规则：

```text
每个标签 1-4 字
支持管理员自定义
同一只猫标签去重
每只猫建议最多 8 个
列表页最多展示 2-3 个
```

示例：

```text
亲人、怕人、胆小、警惕、温顺、活泼、贪吃、会哈气、有攻击性、可摸、不可抱
```
