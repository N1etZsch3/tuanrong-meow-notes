# 校园猫协地图任务系统｜猫咪库模块库表设计文档

> 文档版本：V1.0  
> 模块名称：猫咪库模块  
> 适用阶段：MVP / 第一版数据库设计  
> 数据库：PostgreSQL + PostGIS  
> 后端建议：FastAPI + SQLAlchemy + Alembic  
> 关联模块：地图模块、任务模块、个人中心模块、管理员模块、文件上传模块  

---

# 1. 模块数据定位

猫咪库模块围绕 `cats` 猫咪档案主表展开，向外关联照片、别名、活动点位、观察记录、健康记录和相关任务。

核心数据链路：

```text
cats
  ├── cat_aliases
  ├── cat_photos
  ├── cat_map_points ── map_points
  ├── cat_observation_records ── users / map_points / tasks / task_checkins
  ├── cat_health_records ── cat_observation_records / tasks
  ├── cat_favorites，可选
  └── tasks.related_cat_id / task_cats，后期扩展
```

# 2. 设计原则

## 2.1 `cats` 保存当前档案摘要

`cats` 保存猫咪当前最新状态，例如：

1. 当前健康状态。
2. 当前绝育状态。
3. 当前亲人程度。
4. 最近出现时间。
5. 主要活动区域。
6. 主要常驻点位。

历史变化不堆在 `cats` 中，分别进入观察记录和健康记录。

## 2.2 观察记录与健康记录分开

| 表 | 作用 |
|---|---|
| `cat_observation_records` | 记录某一次看到或没看到猫，当时状态如何 |
| `cat_health_records` | 记录健康事件时间线，如受伤、生病、治疗、绝育、康复 |

例如「今天看到小橘左眼有分泌物」属于观察记录；「左眼异常开始跟进」或「送医处理」属于健康记录。

## 2.3 地图位置由 `map_points` 承担

猫咪库不直接维护地图展示规则。经纬度、路线说明、Marker 图标、点位可见性由地图模块的 `map_points` 管理。

猫咪库通过 `cat_map_points` 关联地图点位。

## 2.4 图片使用统一文件能力

猫咪头像、照片墙、观察照片、健康异常照片都通过统一图片上传能力获取原图和缩略图地址。

本模块表中只保存：

1. 原图 URL。
2. 缩略图 URL。
3. 图片类型。
4. 来源记录。
5. 排序和说明。

## 2.5 核心数据软删除或归档

猫咪档案不建议物理删除。

建议使用：

```text
cats.status = 'archived'
cats.deleted_at = now()
```

照片、点位关联等可以按业务需要软删除或保留历史。

# 4. `cats` 猫咪档案主表

## 4.1 表名

```text
cats
```

## 4.2 作用

保存猫咪基础档案和当前状态摘要。

## 4.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 猫咪 ID |
| `name` | varchar(64) | not null | 猫咪名称 |
| `avatar_url` | varchar(512) | nullable | 主头像 URL，列表页优先使用 |
| `coat_color` | varchar(64) | nullable | 花色，如橘猫、狸花、奶牛、黑猫 |
| `sex` | varchar(32) | not null, default 'unknown' | 性别 |
| `neuter_status` | varchar(32) | not null, default 'unknown' | 绝育状态 |
| `health_status` | varchar(32) | not null, default 'unknown' | 当前健康状态 |
| `friendliness_level` | varchar(32) | not null, default 'unknown' | 亲人程度 |
| `status` | varchar(32) | not null, default 'active' | 档案状态 |
| `primary_area_id` | uuid | FK -> campus_areas.id, nullable | 主要活动区域 |
| `primary_map_point_id` | uuid | FK -> map_points.id, nullable | 主要常驻点位 |
| `last_seen_at` | timestamptz | nullable | 最近出现时间 |
| `last_seen_map_point_id` | uuid | FK -> map_points.id, nullable | 最近出现点位 |
| `personality_tags` | text[] | nullable | 性格标签，MVP 用数组即可 |
| `personality_description` | text | nullable | 性格描述 |
| `feeding_notes` | text | nullable | 喂食注意事项 |
| `capture_notes` | text | nullable | 抓捕注意事项 |
| `medical_notes` | text | nullable | 医疗注意事项 |
| `remark` | text | nullable | 备注 |
| `created_by` | uuid | FK -> users.id, nullable | 创建人 |
| `updated_by` | uuid | FK -> users.id, nullable | 更新人 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间 |

## 4.4 枚举设计

### `sex`

| 值 | 说明 |
|---|---|
| `unknown` | 未知 |
| `male` | 公 |
| `female` | 母 |

### `neuter_status`

| 值 | 说明 |
|---|---|
| `unknown` | 未知 |
| `not_neutered` | 未绝育 |
| `neutered` | 已绝育 |
| `scheduled` | 已预约绝育 |
| `not_suitable` | 暂不适合绝育 |

### `health_status`

| 值 | 说明 |
|---|---|
| `healthy` | 健康 |
| `watching` | 需观察 |
| `abnormal` | 异常 |
| `injured` | 受伤 |
| `sick` | 疑似生病 |
| `in_treatment` | 治疗中 |
| `recovered` | 已恢复 |
| `unknown` | 未知 |

### `friendliness_level`

| 值 | 说明 |
|---|---|
| `friendly` | 亲人 |
| `normal` | 一般 |
| `shy` | 怕人 |
| `aggressive` | 有攻击性 |
| `unknown` | 未知 |

### `status`

| 值 | 说明 |
|---|---|
| `active` | 正常在校 |
| `missing` | 暂时失踪 |
| `adopted` | 已领养 |
| `deceased` | 已死亡 |
| `transferred` | 转移照护 |
| `archived` | 归档 |

## 4.5 索引

```sql
CREATE INDEX idx_cats_name
ON cats (name);

CREATE INDEX idx_cats_health_status
ON cats (health_status);

CREATE INDEX idx_cats_neuter_status
ON cats (neuter_status);

CREATE INDEX idx_cats_primary_area
ON cats (primary_area_id);

CREATE INDEX idx_cats_last_seen_at
ON cats (last_seen_at DESC);

CREATE INDEX idx_cats_status
ON cats (status);

CREATE INDEX idx_cats_active
ON cats (status, deleted_at)
WHERE deleted_at IS NULL;

CREATE INDEX idx_cats_personality_tags
ON cats USING GIN (personality_tags);
```

如果启用模糊搜索，建议补充：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX idx_cats_name_trgm
ON cats USING gin (name gin_trgm_ops);

CREATE INDEX idx_cats_coat_color_trgm
ON cats USING gin (coat_color gin_trgm_ops);
```

## 4.6 业务规则

1. `name` 必填，但不强制全局唯一，因为校园猫咪可能重名。
2. `avatar_url` 只保存主头像地址；完整照片进入 `cat_photos`。
3. `last_seen_at` 由观察记录更新，不建议前端直接修改。
4. `health_status` 可以由管理员手动修改，也可以由健康记录或异常观察触发更新。
5. `primary_map_point_id` 和 `cat_map_points.is_primary = true` 应保持一致。
6. 归档猫咪后普通列表默认不展示，但管理员可在管理端查询。

# 5. `cat_aliases` 猫咪别名表

## 5.1 表名

```text
cat_aliases
```

## 5.2 作用

保存猫咪多个称呼，支持按别名搜索。

## 5.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 主键 |
| `cat_id` | uuid | FK -> cats.id, not null | 猫咪 ID |
| `alias_name` | varchar(64) | not null | 别名 |
| `is_primary` | boolean | default false | 是否主别名 |
| `created_by` | uuid | FK -> users.id, nullable | 创建人 |
| `created_at` | timestamptz | not null | 创建时间 |

## 5.4 索引

```sql
CREATE INDEX idx_cat_aliases_cat
ON cat_aliases (cat_id);

CREATE INDEX idx_cat_aliases_name
ON cat_aliases (alias_name);

CREATE UNIQUE INDEX uq_cat_aliases_cat_alias
ON cat_aliases (cat_id, alias_name);
```

如需模糊搜索：

```sql
CREATE INDEX idx_cat_aliases_name_trgm
ON cat_aliases USING gin (alias_name gin_trgm_ops);
```

## 5.5 业务规则

1. 同一只猫下别名不能重复。
2. `is_primary = true` 的别名建议每只猫最多一个。
3. 删除别名不影响猫咪主名称。
4. 搜索猫咪时需要联查 `cat_aliases`。

# 6. `cat_photos` 猫咪照片表

## 6.1 表名

```text
cat_photos
```

## 6.2 作用

保存猫咪照片墙、识别照、近照、健康异常照、观察来源照片等。

## 6.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 照片 ID |
| `cat_id` | uuid | FK -> cats.id, not null | 猫咪 ID |
| `photo_type` | varchar(32) | not null | 照片类型 |
| `file_url` | varchar(512) | not null | 原图地址 |
| `thumbnail_url` | varchar(512) | nullable | 缩略图地址 |
| `caption` | varchar(255) | nullable | 图片说明 |
| `taken_at` | timestamptz | nullable | 拍摄时间 |
| `uploaded_by` | uuid | FK -> users.id, nullable | 上传人 |
| `observation_record_id` | uuid | FK -> cat_observation_records.id, nullable | 来源观察记录 |
| `health_record_id` | uuid | FK -> cat_health_records.id, nullable | 来源健康记录 |
| `sort_order` | int | default 0 | 排序 |
| `created_at` | timestamptz | not null | 创建时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间，建议补充 |

## 6.4 `photo_type` 枚举

| 值 | 说明 |
|---|---|
| `avatar` | 头像 |
| `profile` | 档案照 |
| `recent` | 近照 |
| `health` | 健康异常照 |
| `neuter` | 绝育相关照 |
| `observation` | 观察记录照 |
| `other` | 其他 |

## 6.5 索引

```sql
CREATE INDEX idx_cat_photos_cat
ON cat_photos (cat_id, sort_order, created_at DESC);

CREATE INDEX idx_cat_photos_observation
ON cat_photos (observation_record_id);

CREATE INDEX idx_cat_photos_health
ON cat_photos (health_record_id);

CREATE INDEX idx_cat_photos_type
ON cat_photos (cat_id, photo_type, created_at DESC);
```

可选唯一约束：

```sql
CREATE UNIQUE INDEX uq_cat_avatar_photo
ON cat_photos (cat_id)
WHERE photo_type = 'avatar' AND deleted_at IS NULL;
```

## 6.6 业务规则

1. 列表页优先使用 `cats.avatar_url` 或 `cat_photos.photo_type = 'avatar'` 的缩略图。
2. 观察记录上传的照片可以写入本表，并关联 `observation_record_id`。
3. 健康异常照片可以写入本表，并关联 `health_record_id`。
4. 删除照片建议先软删除，避免历史观察和健康记录断链。
5. 图片文件的物理删除由文件模块另行处理。

# 7. `cat_map_points` 猫咪点位关联表

## 7.1 表名

```text
cat_map_points
```

## 7.2 作用

保存猫咪与多个地图点位之间的长期关系。

一只猫可以同时拥有：

1. 常驻点。
2. 高频出现点。
3. 喂食点。
4. 临时观察点。
5. 最后出现点。

## 7.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 主键 |
| `cat_id` | uuid | FK -> cats.id, not null | 猫咪 ID |
| `map_point_id` | uuid | FK -> map_points.id, not null | 地图点位 ID |
| `relation_type` | varchar(32) | not null | 关系类型 |
| `is_primary` | boolean | default false | 是否主要常驻点 |
| `confidence_level` | varchar(32) | default 'confirmed' | 可信程度 |
| `first_seen_at` | timestamptz | nullable | 首次出现时间 |
| `last_seen_at` | timestamptz | nullable | 最近出现时间 |
| `description` | text | nullable | 点位说明 |
| `created_by` | uuid | FK -> users.id, nullable | 创建人 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间，建议补充 |

## 7.4 `relation_type` 枚举

| 值 | 说明 |
|---|---|
| `home_area` | 常驻点 |
| `frequent_area` | 高频出现点 |
| `feeding_area` | 喂食点 |
| `temporary_area` | 临时观察点 |
| `lost_area` | 最后出现点 |

## 7.5 `confidence_level` 枚举

| 值 | 说明 |
|---|---|
| `confirmed` | 已确认 |
| `likely` | 大概率 |
| `possible` | 可能 |

## 7.6 索引

```sql
CREATE INDEX idx_cat_map_points_cat
ON cat_map_points (cat_id);

CREATE INDEX idx_cat_map_points_point
ON cat_map_points (map_point_id);

CREATE INDEX idx_cat_map_points_type
ON cat_map_points (cat_id, relation_type);

CREATE INDEX idx_cat_map_points_last_seen
ON cat_map_points (cat_id, last_seen_at DESC);

CREATE UNIQUE INDEX uq_cat_map_points_cat_point
ON cat_map_points (cat_id, map_point_id)
WHERE deleted_at IS NULL;

CREATE UNIQUE INDEX uq_cat_map_points_primary
ON cat_map_points (cat_id)
WHERE is_primary = true AND deleted_at IS NULL;
```

## 7.7 业务规则

1. 创建常驻猫咪点时，应先写入 `map_points.point_type = 'cat'`，再写入本表。
2. 一只猫只能有一个 `is_primary = true` 的主要常驻点。
3. 设置主要常驻点时，同步更新 `cats.primary_map_point_id`。
4. 观察记录发生在某点位时，可以更新对应 `cat_map_points.last_seen_at`。
5. 如果猫咪长期不在某点出现，不建议删除关系，可以降低 `confidence_level` 或归档关联。

# 8. `cat_observation_records` 猫咪观察记录表

## 8.1 表名

```text
cat_observation_records
```

## 8.2 作用

记录成员每一次看到猫、没看到猫、观察异常、任务打卡生成的猫咪状态记录。

## 8.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 观察记录 ID |
| `cat_id` | uuid | FK -> cats.id, not null | 猫咪 ID |
| `observer_id` | uuid | FK -> users.id, not null | 观察人 |
| `source_type` | varchar(32) | not null | 记录来源 |
| `source_task_id` | uuid | FK -> tasks.id, nullable | 来源任务 |
| `source_checkin_id` | uuid | FK -> task_checkins.id, nullable | 来源打卡 |
| `observed_at` | timestamptz | not null | 观察时间 |
| `map_point_id` | uuid | FK -> map_points.id, nullable | 观察点位 |
| `area_id` | uuid | FK -> campus_areas.id, nullable | 观察区域 |
| `lng` | numeric(10,7) | nullable | 临时观察经度 |
| `lat` | numeric(10,7) | nullable | 临时观察纬度 |
| `geom` | geography(Point,4326) | nullable | 临时观察点 |
| `is_seen` | boolean | default true | 是否见到猫 |
| `cat_condition` | varchar(32) | nullable | 猫咪当时状态 |
| `health_status` | varchar(32) | nullable | 观察到的健康状态 |
| `behavior_status` | varchar(64) | nullable | 行为状态，可自由扩展 |
| `appetite_status` | varchar(32) | nullable | 进食情况 |
| `is_injured` | boolean | nullable | 是否受伤 |
| `injury_description` | text | nullable | 受伤描述 |
| `need_follow_up` | boolean | default false | 是否需要后续观察 |
| `suggest_task` | boolean | default false | 是否建议发布任务 |
| `description` | text | nullable | 行为描述 |
| `remark` | text | nullable | 备注 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间，建议补充 |

## 8.4 `source_type` 枚举

| 值 | 说明 |
|---|---|
| `manual` | 成员主动提交 |
| `task_checkin` | 任务完成打卡生成 |
| `admin` | 管理员补充 |
| `emergency` | 紧急上报 |
| `health` | 健康异常反馈 |

## 8.5 `cat_condition` 枚举

| 值 | 说明 |
|---|---|
| `normal` | 正常 |
| `weak` | 虚弱 |
| `scared` | 受惊 |
| `sleeping` | 睡觉 |
| `eating` | 进食 |
| `hiding` | 躲藏 |
| `active` | 活跃 |
| `unknown` | 未知 |

## 8.6 `appetite_status` 枚举

| 值 | 说明 |
|---|---|
| `normal` | 正常进食 |
| `low` | 食欲差 |
| `none` | 未进食 |
| `unknown` | 未知 |

## 8.7 索引

```sql
CREATE INDEX idx_cat_observations_cat_time
ON cat_observation_records (cat_id, observed_at DESC);

CREATE INDEX idx_cat_observations_observer_time
ON cat_observation_records (observer_id, observed_at DESC);

CREATE INDEX idx_cat_observations_health
ON cat_observation_records (health_status);

CREATE INDEX idx_cat_observations_follow_up
ON cat_observation_records (need_follow_up, observed_at DESC);

CREATE INDEX idx_cat_observations_suggest_task
ON cat_observation_records (suggest_task, observed_at DESC);

CREATE INDEX idx_cat_observations_point
ON cat_observation_records (map_point_id, observed_at DESC);

CREATE INDEX idx_cat_observations_geom
ON cat_observation_records USING GIST (geom);
```

## 8.8 业务规则

1. `is_seen = false` 时，健康、行为、食欲字段可以为空。
2. `is_seen = true` 且 `observed_at` 晚于 `cats.last_seen_at` 时，更新猫咪最近出现时间。
3. 如果 `health_status` 为异常、受伤、疑似生病，建议 `need_follow_up = true`。
4. 如果 `suggest_task = true`，可以生成通知给管理员，后续进入任务发布流程。
5. 任务打卡生成观察记录时，写入 `source_type = 'task_checkin'`、`source_task_id`、`source_checkin_id`。
6. 观察照片写入 `cat_photos`，并关联 `observation_record_id`。

# 9. `cat_health_records` 猫咪健康记录表

## 9.1 表名

```text
cat_health_records
```

## 9.2 作用

保存猫咪健康事件和医疗时间线。

## 9.3 字段设计

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | uuid | PK | 健康记录 ID |
| `cat_id` | uuid | FK -> cats.id, not null | 猫咪 ID |
| `record_type` | varchar(32) | not null | 健康记录类型 |
| `health_status_before` | varchar(32) | nullable | 变更前健康状态 |
| `health_status_after` | varchar(32) | nullable | 变更后健康状态 |
| `title` | varchar(128) | not null | 记录标题 |
| `description` | text | nullable | 详细说明 |
| `occurred_at` | timestamptz | not null | 发生时间 |
| `related_observation_id` | uuid | FK -> cat_observation_records.id, nullable | 来源观察记录 |
| `related_task_id` | uuid | FK -> tasks.id, nullable | 来源任务 |
| `hospital_name` | varchar(128) | nullable | 医院名称 |
| `treatment_result` | text | nullable | 处理结果 |
| `medicine` | text | nullable | 用药情况 |
| `next_follow_up_at` | timestamptz | nullable | 下次跟进时间 |
| `created_by` | uuid | FK -> users.id, nullable | 记录人 |
| `created_at` | timestamptz | not null | 创建时间 |
| `updated_at` | timestamptz | not null | 更新时间 |
| `deleted_at` | timestamptz | nullable | 软删除时间，建议补充 |

## 9.4 `record_type` 枚举

| 值 | 说明 |
|---|---|
| `injury` | 受伤 |
| `sickness` | 生病 |
| `treatment` | 治疗 |
| `medicine` | 用药 |
| `neuter` | 绝育 |
| `recovery` | 康复 |
| `death` | 死亡 |
| `abnormal` | 异常观察 |
| `checkup` | 检查 |
| `other` | 其他 |

## 9.5 索引

```sql
CREATE INDEX idx_cat_health_records_cat_time
ON cat_health_records (cat_id, occurred_at DESC);

CREATE INDEX idx_cat_health_records_type
ON cat_health_records (record_type);

CREATE INDEX idx_cat_health_records_follow_up
ON cat_health_records (next_follow_up_at)
WHERE next_follow_up_at IS NOT NULL;

CREATE INDEX idx_cat_health_records_observation
ON cat_health_records (related_observation_id);

CREATE INDEX idx_cat_health_records_task
ON cat_health_records (related_task_id);
```

## 9.6 业务规则

1. 健康记录用于时间线展示，不能只依赖 `cats.health_status`。
2. 创建健康记录时，可以同步更新 `cats.health_status`。
3. 绝育记录创建后，可以同步更新 `cats.neuter_status = 'neutered'`。
4. 康复记录创建后，可以同步更新 `cats.health_status = 'recovered'` 或 `healthy`。
5. 死亡记录创建后，可以同步更新 `cats.status = 'deceased'`。
6. 健康照片写入 `cat_photos`，并关联 `health_record_id`。

# 14. 常用查询设计

## 14.1 猫咪列表查询

主要表：

```text
cats
LEFT JOIN cat_aliases
LEFT JOIN campus_areas
LEFT JOIN cat_map_points
LEFT JOIN map_points
```

常用条件：

```sql
WHERE cats.deleted_at IS NULL
  AND cats.status IN ('active', 'missing')
```

常用排序：

```sql
ORDER BY cats.last_seen_at DESC NULLS LAST, cats.updated_at DESC
```

## 14.2 需关注猫咪

```sql
WHERE cats.health_status IN ('watching', 'abnormal', 'injured', 'sick', 'in_treatment')
   OR cats.last_seen_at < now() - interval '7 days'
   OR cats.neuter_status = 'not_neutered'
```

## 14.3 猫咪详情查询

```text
cats
cat_aliases
cat_photos
cat_map_points + map_points
cat_observation_records
cat_health_records
tasks
```

建议详情页接口可以拆分加载：

1. 基础档案首屏先返回。
2. 观察记录分页加载。
3. 健康记录分页加载。
4. 相关任务分页加载。
5. 照片墙分页加载。

## 14.4 我的观察记录

```sql
SELECT *
FROM cat_observation_records r
JOIN cats c ON c.id = r.cat_id
LEFT JOIN map_points p ON p.id = r.map_point_id
WHERE r.observer_id = :current_user_id
ORDER BY r.observed_at DESC;
```

## 14.5 我的收藏猫咪

```sql
SELECT c.*
FROM cat_favorites f
JOIN cats c ON c.id = f.cat_id
WHERE f.user_id = :current_user_id
  AND f.deleted_at IS NULL
  AND c.deleted_at IS NULL
ORDER BY f.created_at DESC;
```

# 15. 数据一致性规则

## 15.1 最近出现时间

新增观察记录时：

```text
if is_seen = true and observed_at > cats.last_seen_at:
    update cats.last_seen_at
    update cats.last_seen_map_point_id
```

如果观察点位存在于 `cat_map_points`：

```text
update cat_map_points.last_seen_at
```

## 15.2 主常驻点

设置主要常驻点时：

```text
1. 将同 cat_id 的其他 cat_map_points.is_primary 改为 false
2. 将目标 cat_map_points.is_primary 改为 true
3. 更新 cats.primary_map_point_id
4. 更新 cats.primary_area_id，可选
```

## 15.3 健康状态

新增健康记录时：

```text
1. 写入 cat_health_records
2. 根据 health_status_after 更新 cats.health_status
3. 如 record_type = neuter，更新 cats.neuter_status
4. 如 record_type = death，更新 cats.status
```

## 15.4 照片头像

设置头像时：

```text
1. 写入或更新 cat_photos.photo_type = 'avatar'
2. 更新 cats.avatar_url
3. 列表页优先读取 cats.avatar_url
```

);

    created_at timestamptz NOT NULL DEFAULT now()

CREATE TABLE cat_photos (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),

    thumbnail_url varchar(512),
    caption varchar(255),
    taken_at timestamptz,
    uploaded_by uuid NULL REFERENCES users(id),
    observation_record_id uuid NULL,
    health_record_id uuid NULL,

CREATE TABLE cat_map_points (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    cat_id uuid NOT NULL REFERENCES cats(id),
    map_point_id uuid NOT NULL REFERENCES map_points(id),

    is_primary boolean NOT NULL DEFAULT false,
    confidence_level varchar(32) NOT NULL DEFAULT 'confirmed',
    first_seen_at timestamptz,
    last_seen_at timestamptz,
    description text,

CREATE TABLE cat_observation_records (

    health_status_after varchar(32),

CREATE TABLE cat_favorites (
