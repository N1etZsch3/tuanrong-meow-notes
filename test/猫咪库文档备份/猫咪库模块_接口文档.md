# 校园猫协地图任务系统｜猫咪库模块接口文档

> 文档版本：V1.0  
> 模块名称：猫咪库模块  
> Base URL：`/api/v1`  
> 适用阶段：MVP / 第一版接口设计  
> 适用端：移动端微信小程序 / H5  
> 字段命名：统一使用 `snake_case`  
> 响应结构：统一使用 `{ code, message, data, trace_id }`  
> 关联模块：地图模块、任务模块、个人中心模块、管理员模块、文件上传模块  

---

# 1. 文档定位

本文档用于定义「校园猫协地图任务系统」中猫咪库模块的 HTTP API。

猫咪库模块负责围绕校园猫咪档案提供：

1. 猫咪列表浏览、搜索、筛选、排序。
2. 猫咪档案详情查询。
3. 猫咪照片墙查询。
4. 猫咪常驻点位、出现点位、地图跳转信息查询。
5. 猫咪观察记录提交与查询。
6. 猫咪健康记录查询与管理员维护。
7. 猫咪相关任务查询。
8. 猫咪收藏与取消收藏。
9. 管理员新增、编辑、归档猫咪档案。
10. 管理员维护别名、照片、点位关系、健康记录。

本接口文档基于猫咪库功能文档和猫咪库库表设计文档编写，遵循项目统一接口规范。

# 2. 模块边界

## 2.1 猫咪库接口负责

| 能力 | 说明 |
|---|---|
| 猫咪列表 | 返回猫咪摘要卡片、统计数据、筛选结果 |
| 猫咪详情 | 返回基础档案、别名、照片、点位、注意事项、记录摘要 |
| 搜索筛选 | 支持按名称、别名、花色、区域、健康、绝育、亲人程度、档案状态筛选 |
| 观察记录 | 支持成员提交观察记录，支持详情页查询观察历史 |
| 健康记录 | 支持查询健康时间线，管理员维护健康事件 |
| 照片墙 | 查询猫咪照片，管理员维护照片元信息 |
| 地图联动 | 返回猫咪关联点位，提供地图页跳转参数 |
| 任务联动 | 查询与猫咪相关的任务摘要 |
| 收藏猫咪 | 支撑个人中心「收藏猫咪」入口 |
| 管理维护 | 管理员新增、编辑、归档猫咪档案 |

## 2.2 猫咪库接口不负责

| 能力 | 所属模块 | 说明 |
|---|---|---|
| 文件压缩、上传、缩略图生成 | 文件上传模块 | 猫咪库只保存文件模块返回的 URL / 文件 ID |
| 地图 Marker 渲染和导航路线 | 地图模块 | 猫咪库只返回点位关联和跳转参数 |
| 任务接取、放弃、完成打卡 | 任务模块 | 猫咪库只展示相关任务摘要和跳转入口 |
| 用户资料与个人统计 | 个人中心模块 | 个人中心可反查猫咪收藏、观察记录 |
| 登录、改密、Token 校验 | 鉴权模块 | 猫咪库接口只遵循统一鉴权规则 |

# 3. 通用约定

## 3.1 Base URL

```text
/api/v1
```

示例：

```http
GET /api/v1/cats
Authorization: Bearer <access_token>
```

## 3.2 鉴权规则

猫咪库接口均属于内部成员系统接口，除特殊说明外均需要登录。

```http
Authorization: Bearer <access_token>
```

当用户处于首次登录强制改密状态时，业务接口返回：

```json
{
  "code": 40301,
  "message": "请先修改初始密码",
  "data": null,
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

当用户尚未完成资料初始化时，业务接口可以返回：

```json
{
  "code": 63006,
  "message": "请先完成个人资料初始化",
  "data": null,
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 3.3 Content-Type

普通 JSON 接口：

```http
Content-Type: application/json
```

猫咪库本身不直接接收二进制文件上传。图片应先通过文件上传模块上传，拿到 `file_url`、`thumbnail_url` 或文件资源 ID 后，再调用猫咪库接口保存图片引用。

## 3.4 统一响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "trace_id": "018f3c2e9b7e4f1a9a7e6c8d9"
}
```

## 3.5 分页规范

列表接口统一使用：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| `page` | int | 1 | 当前页，从 1 开始 |
| `page_size` | int | 20 | 每页数量，最大 100 |

分页响应结构：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0,
  "has_more": false
}
```

## 3.6 时间格式

所有时间统一使用 ISO 8601 / RFC3339 字符串。

```json
{
  "last_seen_at": "2026-06-27T09:20:00+08:00"
}
```

## 3.7 经纬度格式

经纬度字段统一使用：

| 字段 | 类型 | 说明 |
|---|---|---|
| `lng` | number | 经度 |
| `lat` | number | 纬度 |

# 6. 通用 DTO

## 6.1 CatStatsDTO

用于猫咪库顶部统计卡片。

```json
{
  "total_cats": 36,
  "active_count": 28,
```

| `suggest_task` | boolean | 否 | 是否建议发布任务 |
| `description` | text | 否 | 观察说明 |
| `remark` | text | 否 | 备注 |
| `photos` | array | 否 | 图片引用数组，不直接传二进制 |

| 61004 | 健康状态不合法 |

| 61021 | 猫咪已归档，不能执行该操作 |

# 10. 数据更新联动规则

## 10.1 观察记录联动

提交观察记录后，后端根据情况更新：

| 场景 | 联动 |
|---|---|
| `is_seen = true` 且 `observed_at` 晚于当前最近出现时间 | 更新 `cats.last_seen_at` |
| 传入 `map_point_id` 且为最新观察 | 更新 `cats.last_seen_map_point_id` |
| 传入 `health_status` 为异常类状态 | 可更新 `cats.health_status`，或仅设置 `need_follow_up` |
| 传入观察照片 | 写入 `cat_photos` 并关联 `observation_record_id` |
| `suggest_task = true` | 可生成管理员通知，后期进入任务创建流程 |

## 10.2 健康记录联动

| 场景 | 联动 |
|---|---|
| `update_cat_health_status = true` | 更新 `cats.health_status = health_status_after` |
| `record_type = neuter` | 可更新 `cats.neuter_status = neutered` |
| `record_type = death` | 可更新 `cats.status = deceased` |
| 创建健康照片 | 写入 `cat_photos` 并关联 `health_record_id` |

## 10.3 头像联动

| 场景 | 联动 |
|---|---|
| 添加照片时 `set_as_avatar = true` | 更新 `cats.avatar_url` |
| 设置照片为头像 | 更新 `cats.avatar_url`，可更新 `photo_type = avatar` |
| 删除当前头像 | 清空头像或选择下一张头像 |

## 10.4 常驻点联动

| 场景 | 联动 |
|---|---|
| 新增主要常驻点 | 更新 `cats.primary_map_point_id` |
| 取消主要常驻点 | 清空或重设 `cats.primary_map_point_id` |
| 观察记录在某点位出现 | 更新 `cat_map_points.last_seen_at` |

# 11. 前端调用建议

## 11.1 猫咪库列表页

建议进入页面时并发调用：

```text
GET /api/v1/cats/overview
GET /api/v1/cats?page=1&page_size=20
```

页面交互：

1. 搜索框输入关键词后调用 `GET /cats?keyword=...`。
2. 多级筛选菜单选择筛选项和值后调用 `GET /cats?filter_key=...&filter_value=...`。
3. 排序切换后调用 `GET /cats?sort_by=...&sort_order=...`。
4. 清除筛选时保留默认排序，清空 `filter_key`、`filter_value`、快捷筛选参数。
5. 下拉刷新重新请求第一页。
6. 上拉加载下一页。

## 11.2 猫咪详情页

建议首屏调用：

```text
GET /api/v1/cats/{cat_id}?include=photos,map_points,observations,health_records,tasks
```

如果用户进入具体 Tab，再调用独立分页接口：

```text
GET /api/v1/cats/{cat_id}/photos
GET /api/v1/cats/{cat_id}/observations
GET /api/v1/cats/{cat_id}/health-records
GET /api/v1/cats/{cat_id}/tasks
```

## 11.3 添加观察记录页

推荐流程：

```text
1. 进入页面时读取猫咪详情或点位上下文
2. 用户填写观察记录
3. 图片先调用文件上传模块
4. 拿到 file_url / thumbnail_url
5. 调用 POST /cats/{cat_id}/observations
6. 成功后刷新猫咪详情和个人中心观察记录
```

## 11.4 管理员新增猫咪

推荐流程：

```text
1. 图片先上传到文件模块
2. 如果需要地图点位，先调用地图模块创建 map_points
3. 调用 POST /admin/cats 创建猫咪档案
4. 后端写入 cats、cat_aliases、cat_photos、cat_map_points
5. 返回 cat_id
6. 前端跳转猫咪详情或管理员编辑页
```

# 12. 后端实现建议

## 12.1 服务分层

建议后端模块结构：

```text
app/modules/cats/
  ├── models.py
  ├── schemas.py
  ├── service.py
  ├── repository.py
  ├── enums.py
  └── permissions.py
```

## 12.2 查询优化

1. 猫咪列表使用分页，不一次性返回全部数据。
2. 列表页避免返回完整观察记录和健康记录。
3. 详情页只返回少量预览记录，完整列表走分页接口。
4. 搜索名称和别名建议使用 `pg_trgm` 模糊索引。
5. 猫咪列表需要联查区域、别名、头像、收藏状态，建议封装查询视图或 repository 聚合方法。

## 12.3 权限判断

| 接口类型 | 权限 |
|---|---|
| 成员端查询 | 登录用户 |
| 添加观察记录 | 登录用户 |

## 12.4 数据一致性

建议把以下操作封装为事务：

1. 创建猫咪 + 别名 + 照片 + 点位关系。

3. 新增健康记录 + 写入照片 + 更新猫咪健康状态。
4. 设置主要常驻点 + 更新 `cats.primary_map_point_id`。
5. 设置头像 + 更新 `cats.avatar_url`。

# 14. 自检

## 14.1 是否覆盖列表页？

已覆盖：

1. 统计卡片：`GET /cats/overview`。
2. 搜索：`GET /cats?keyword=...`。
3. 多级筛选：`filter_key + filter_value`。
4. 排序：`sort_by + sort_order`。
5. 列表卡片：`CatListItemDTO`。
6. 收藏状态：`is_favorited`。

## 14.2 是否覆盖详情页？

已覆盖：

1. 基础档案：`GET /cats/{cat_id}`。
2. 别名：`aliases`。
3. 照片预览：`photos_preview`。
4. 活动点位：`map_points_preview`。
5. 观察记录：`latest_observations` 和独立分页接口。
6. 健康记录：`latest_health_records` 和独立分页接口。
7. 相关任务：`latest_tasks` 和独立分页接口。
8. 权限按钮：`permissions`。

## 14.3 是否覆盖观察记录？

已覆盖：

1. 查询观察记录。
2. 新增观察记录。
3. 观察照片引用。
4. 最近出现时间联动。
5. 异常状态联动。
6. 管理员修正和删除。

## 14.4 是否覆盖管理员维护？

已覆盖：

1. 新增猫咪。
2. 编辑猫咪。
3. 修改档案状态。
4. 维护别名。
5. 维护照片。
6. 维护点位关系。
7. 维护健康记录。
8. 管理观察记录。

## 14.5 是否避免和其他模块职责重复？

已避免：

1. 图片文件上传仍归文件上传模块。
2. 地图导航仍归地图模块。
